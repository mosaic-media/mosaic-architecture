# Purpose

This document establishes the governing standards for all documentation produced within the Mosaic Architecture repository.

It defines:

- documentation philosophy
- document responsibilities
- document lifecycle
- writing standards
- organisational conventions
- governance expectations

The purpose of MDG-001 is not to document Mosaic itself.

Instead, it defines **how Mosaic documentation is authored, organised, reviewed and maintained**.

It therefore acts as the authoritative guide for every future architectural document.

---

# Authority

MDG-001 is the governing authority for documentation standards within the Mosaic Architecture repository.

All documentation should conform to the principles defined within this guide unless an explicit exception has been approved by the project maintainers.

Where documentation conflicts with this guide, the guidance contained within MDG-001 shall take precedence.

---

# Scope

This guide applies to every documentation type maintained within the Mosaic Architecture repository.

This includes, but is not limited to:

- Mosaic Architecture Canon (MAC)
- Mosaic Design Proposal (MDP)
- Mosaic Architecture Decision (MAD)
- Mosaic Engineering Guide (MEG)
- Mosaic Integration Protocol (MIP)
- Mosaic Operations & Playbook (MOP)
- Mosaic Design Language (MDL)
- Mosaic Design System (MDS)

Future documentation types are expected to adopt these standards unless a documented exception is approved.

---

# Intended Audience

This document is intended for:

- architecture authors
- software engineers
- maintainers
- contributors
- reviewers
- technical writers

Readers are expected to use this guide whenever creating, reviewing or modifying documentation within the repository.

---

# Normative Language

The following terms are used throughout this document.

| Term | Meaning |
|------|---------|
| **Must** | Mandatory requirement. |
| **Should** | Strong recommendation. Exceptions should be justified. |
| **May** | Optional behaviour. |
| **Must Not** | Explicitly prohibited behaviour. |

These terms are used consistently throughout Mosaic documentation to distinguish mandatory architectural requirements from implementation guidance and optional practices.

---

# Versioning

Documentation follows semantic progression rather than implementation maturity.

| Version | Meaning |
|---------:|---------|
| 0.1 | Initial draft. |
| 0.2 | Editorial review completed. |
| 0.3 | Structural review completed. |
| 0.4 | Cross-reference review completed. |
| 0.5 | Technical review completed. |
| 0.9 | Candidate for approval. |
| 1.0 | Canonical release. |

Version increments should represent meaningful improvements to the document rather than simple passage of time.

---

# Review Policy

Every document should undergo review before progressing between versions.

Reviews should consider:

- clarity
- consistency
- completeness
- architectural alignment
- terminology
- references
- duplication

Documentation quality is considered an architectural concern rather than an editorial exercise.

---

# Relationship to Other Documents

MDG-001 governs the structure and quality of Mosaic documentation.

It does not replace any existing specification.

Instead, it defines the standards those specifications are expected to follow.

Subsequent chapters establish:

- documentation hierarchy
- document responsibilities
- versioning strategy
- writing standards
- governance
- repository organisation

Together these chapters define the Mosaic documentation system.
