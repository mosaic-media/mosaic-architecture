# Roadmap

Derived from the real state of `platform`, not from a plan written ahead of it.

---

## Where the build actually is

The Platform boots against real PostgreSQL, serves its client API as two Connect services — `AuthService` to mint a session and the two-lane `SessionService` to spend it ([ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md), [ADR 0061](adr/0061-one-client-transport.md)) — plus two plain-HTTP origins on the same port, `/artwork` ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)) and `/playback/{ticket}` ([ADR 0045](adr/0045-playback-consumer-and-media-origin.md)). It runs an outbox worker with retry and dead-lettering, resolves secrets, reports component health, shuts down gracefully with a final outbox drain, holds a content-agnostic object graph, exposes a content command and query API over that graph, and records its own behaviour as structured logs and spans it can serve back ([ADR 0053](adr/0053-telemetry-is-ambient-in-context.md)–[ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md)). That API's surface has been extracted into a standalone SDK module — [`github.com/mosaic-media/sdk`](https://github.com/mosaic-media/sdk), published at `v0.20.0`, holding the content models, the `ContentService` methods, the `Capability` interface and its `Manifest`, the seven provider roles, the dependency-free `Telemetry` interface and an opaque `Caller` — which the Platform, the reference capability and all five modules depend on as an external module. Every slice passes `go build`, `go vet` and `go test -race` against a real database.

**A slice recorded as done here means the Platform capability landed. It does not always mean a user can reach it.** Permissions management, configuration versioning and user administration are all recorded below as done, and are all correct — and none of them has a client surface. [Unreachable capability](unreachable-capability.md) is the register of that gap, kept separately because nothing in the build or the test suite will ever report it. Read the two together before concluding Mosaic can do something.

The repositories are public and licensed ([ADR 0022](adr/0022-licensing.md)): the Platform under AGPL-3.0 with a module-linking exception, the SDK under Apache-2.0, optional modules under their authors' choice (the first is MIT), and this documentation under CC-BY-4.0.

Two further slices — uniform store resolution and its PostgreSQL follow-up — were built and then reverted under [ADR 0012](adr/0012-capabilities-do-not-own-stores.md), which found they solved a case the architecture had already ruled out.

The content model, its published surface, a capability that uses only that surface, and the surface's extraction into a standalone SDK module have all landed — **the thesis test passes and the critical path is complete.** Since then three modules — the **Stremio** addon source, **remote playback** (the first *consumer* module) and **TMDB** (the first *core* module, [ADR 0062](adr/0062-two-module-tiers.md)) — have been composed into the binary and invoked through a capability registry, so the composition-and-invocation half of the extension story works too ([ADR 0019](adr/0019-module-capability-and-invocation.md), [ADR 0020](adr/0020-optional-module-composition.md)). The Shell — a Server-Driven-UI client in the [`web`](https://github.com/mosaic-media/web) workspace ([ADR 0023](adr/0023-server-driven-ui-and-the-shell.md), [ADR 0024](adr/0024-primitives-and-definitions.md), [ADR 0042](adr/0042-frontend-workspace.md)) — is built too, and the Platform emits the screens it renders and pushes them over the live session. A user can search, add a film and watch it. What follows (media formats, module distribution, a second content capability) builds on a foundation that is proven rather than assumed.

The build spans **ten repositories**: `architecture`, `platform`, `sdk`, `contracts`, `web`, and the five modules (`module-stremio-addons`, `module-aiostreams`, `module-remote-playback`, `module-tmdb`, `module-cinemeta`). The three former web repositories were consolidated into `web` under [ADR 0042](adr/0042-frontend-workspace.md) and archived.

## The critical path

Everything below is one thread. Nothing else should start until it lands, because it is the test of Mosaic's central thesis: **that a developer who is not you can extend Mosaic through the SDK.**

### 1 — The content-agnostic object model — **done**

The node tree and relation graph, designed in [ADR 0013](adr/0013-object-graph.md), with authority and media linking settled in [ADR 0014](adr/0014-storage-authority-and-transaction-scope.md).

`nodes`, `parts`, `relations` and `source_bindings` shipped with domain types, store contracts, PostgreSQL implementations, the four stores added to `Tx`, and adapter-agnostic contract tests against real PostgreSQL. Identifiers are UUIDv7 in native `uuid` columns; the existing infrastructure tables keep their `text`/UUIDv4 ids and were not migrated. ADR 0013's four deliberate non-uniformities each have a contract test, since each is cheap to normalise away by accident.

This was the real blocker, and it had been mistaken for something else. The reference capability was recorded as blocked on an empty `contracts/platform/v1` and a closed `Tx`; both were symptoms of building the extension mechanism before the thing it extends. A capability now has somewhere to put an anime.

**Deliberately not in the slice, and still unbuilt:** export formats, the filesystem projection, streaming, the job queue, `LISTEN/NOTIFY`, and IPTV listings. None blocks the reference capability.

### 2 — Content commands and queries — **done**

The stores existed and were proven, but nothing above them did. No application service touched the graph, so the only way in was `Tx`, which the composition root holds — and a capability reaching for `Tx` directly would bypass the policy engine entirely.

This slice put the command order over the content model. Nine application services now exist, each validating, authenticating, authorising and — for writes — committing state and an outbox event in one transaction:

- **Reads:** `SearchContent` (title substring, media type, kind), `FindContentByExternalID` (provider id against the GIN index), `GetContentNode` (one node, optionally its direct children). Before this, `FindBySource` answered only *"have I bound this exact provider reference?"*; "do I already have this anime?" by title or external id was unanswerable, and the `nodes_external_ids_gin` and `nodes (media_type, title)` indexes had no query reaching them.
- **Writes:** `AddContentWork`, `AddContentChild`, `AttachContentPart` build the containment tree; `RelateContent`, `BindContentSource`, `ResolveContentBinding` write the association graph and resolve identity.

Two shapes were decided in the building. A child inherits its work id and media type from its parent, so a season cannot declare a different media type than its series. A binding is created confirmed or pending-review, never rejected — rejection is a resolution of an existing binding, not a state to create one in.

It was scheduled ahead of the reference capability because it is needed under every resolution of the boundary problem below, so none of it could be wasted. It did **not** resolve either blocker: every content command still authorises on a user session, and the contracts still reference `domain` types under `internal/`.

### 3 — Reference capability path — **done**

Attempted twice and correctly reported as blocked rather than forced, then blocked a third time on two things neither attempt had reached — both decided (ADR 0016, ADR 0017) and built. The capability now exists and passes: **the thesis holds.** A package that imports only `contracts/platform/v1`, owns no schema and touches no Platform internals sources an anime over HTTP, searches by provider id to avoid duplicating, creates the work with a source binding, builds its season/episode/part tree, and — honouring ADR 0013 — creates the source manga as its own Work joined by an adaptation edge rather than folding it into one tree. It runs against real PostgreSQL through the postgres harness: a fake provider serves canned metadata over `httptest`, and the test asserts the work is findable by external id, the tree is where it belongs, the edge and both bindings landed, every command emitted its event, and a second import is idempotent.

`capabilities/reference/` lives outside `internal/` and a boundary test parses its imports to keep it to `v1` and the standard library — the stop point made executable. It owns its provider integration outright (an HTTP metadata source on `net/http`), since the Platform offers no HTTP contract and does not need to (ADR 0007).

Two findings from building it. The published surface was **sufficient** — no private import was needed, which is the proof the contracts are ready. And one gap: `ContentService` has no *read* for relations (`ListFrom`/`ListTo`), so the test reads the adaptation edge through the store; the capability's own flow does not need it, so it is a candidate addition to `v1`, not a blocker.

The remainder of this section records how the block was cleared.

#### Done: the published surface ([ADR 0016](adr/0016-published-contract-surface.md))

The block was that every contract signature references a `domain` type under `internal/`, which Go forbids an external module from importing. The fix corrected the roadmap's own framing: a capability does **not** call the store contracts (`NodeStore`, `Tx`, `StorageAdapter`) — those are Platform↔engine plumbing. It calls the application-service API.

`contracts/platform/v1` now holds that surface: the content models (moved out of `internal/platform/domain`, not duplicated), the nine content command, query and result types, the `ContentService` interface `app.Service` implements, and an opaque `Caller`. The store contracts and the identity and configuration models stayed internal. The enforcement is a standing test — `test/sdkprobe` is a separate Go module that imports only `v1` and exercises the whole service; `test/sdkboundary` builds it, and because an external module cannot import `internal/`, a public signature that leaked an internal type fails that build. Verified by watching it fail on a deliberately added internal import.

#### Decided: how a capability acts ([ADR 0017](adr/0017-how-a-capability-acts.md))

A capability does not originate authority. The Platform invokes it within a context carrying a principal, and it forwards that context to every service it calls. For the reference capability the principal is the invoking user, so nothing in the policy engine changes and every created node traces to the person who caused it. This is realised as the `Caller`: a session reference the capability forwards, only as authoritative as the session behind it. A system principal for background work, and module-granular authority, are named future decisions rather than machinery built now.

#### How the block was cleared

Its purpose changed under [ADR 0012](adr/0012-capabilities-do-not-own-stores.md). It was to prove a capability could own a store and join a transaction. Since capabilities own no schema, it proves instead what a capability actually does, shaped like the anime module the platform exists to support: source metadata from an external provider, search existing content, create nodes and relations, and cause events. All four are done.

Sourcing metadata needs no Platform HTTP contract, which is worth stating because it looks like a gap and is not one. Capabilities compile into the binary with trust established before the build ([ADR 0007](adr/0007-static-go-module-composition.md)), so a capability imports `net/http` itself — as the reference capability's metadata source does. A Platform-provided client is worth having eventually for rate limiting and secret handling; it is not a prerequisite.

#### Not carried here after all: the `media_types` registry

The original plan ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) had this slice carry the registry that catches a media type which was never real (a missing separator or a misspelling, which normalisation cannot recover). It did not, and ADR 0015 is amended to say why: the reference capability is an anime importer that uses only known media types, so it never introduces a novel one, and the registry depends on the capability manifest shape, which is still undecided. The registry now waits on that manifest mechanism and on a capability that genuinely introduces a new type. Until then normalisation is the whole of the enforcement.

**Exit criteria — met.** The surface of [ADR 0016](adr/0016-published-contract-surface.md) is published into `contracts/platform/v1`, and one capability does all four using only those packages, acting as its invoking user ([ADR 0017](adr/0017-how-a-capability-acts.md)), owning no schema and touching no Platform code.

### 4 — SDK extraction — **done**

Whether the contracts proven across the completed slices can leave the Platform repository as a standalone SDK a third party can build against. They can, and they have: the surface is now its own module, [`github.com/mosaic-media/sdk`](https://github.com/mosaic-media/sdk), published at `v0.1.0`.

`contracts/platform/v1` moved out of the Platform repository into the SDK module unchanged, and the Platform now depends on it as an ordinary tagged dependency — no `replace` directive, so a fresh clone resolves it through the module proxy. Several separate modules build against the published SDK: the Platform, the reference capability, `test/sdkprobe`, and the Stremio module in its own repository. The surface holds no private Platform internals, because Go itself now forbids it — an external module cannot import the Platform's `internal/`, so a leak would fail to compile rather than needing a test to catch it.

**Exit criteria — met.** The surface left the repository as a standalone module, and the Platform and the reference capability build against it as external consumers.

The SDK is pre-1.0 deliberately (now at `v0.13.0`, thirteen additive bumps driven by what modules turned out to need) and may still change. The relation-read gap noted in step 3, and any surface a second capability turns out to need, are the kind of change a `v0.x` bump absorbs before the surface is declared stable.

### The stop point — cleared

> **The Platform is ready for SDK work when the reference capability uses only published contract packages and no private Platform internals.**
>
> **If the reference capability requires a private import, the contracts are not ready to publish.**

The reference capability landed against `contracts/platform/v1` alone, enforced by a boundary test that parses its imports. It needed no private import, so the line held rather than being waved through — which is what makes the surface ready to extract.

**Step 3 was the thesis test, and it passed; step 4 extracted the surface it proved.** A capability can be built entirely against the published contract surface, and that surface is now a standalone module a third party depends on rather than a package inside this repository. The extension model is real, end to end.

### Since the thesis: the platform runs

Beyond the critical path, the Platform is now a usable process rather than only a tested library:

- **A client API is served over HTTP** — the composition root constructs `app.Service`, and the API listens on `:8081` (the Supervisor handoff surface stays on `:8080`). It began as a hand-written GraphQL handler; that was replaced by the Connect transport and then deleted outright ([ADR 0061](adr/0061-one-client-transport.md), recorded below).
- **Real password auth** — an Argon2id hasher (`internal/adapters/crypto`), so sign-in actually verifies.
- **Permissions can be assigned through the Platform** — `PermissionStore` gained `CreateRole`/`GrantRole`, with commands behind them. `PermissionStore` was read-only before; authority could only be seeded by raw SQL. Those commands have no client surface, which is the [unreachable capability](unreachable-capability.md) register's first row.
- **The first admin bootstraps itself** — `bootstrap.EnsureAdmin`, env-gated and idempotent, so a human can start the binary and use it ([ADR 0018](adr/0018-first-admin-bootstrap.md), a deliberate bridge until Supervisor onboarding owns it). Proven end to end over HTTP against real PostgreSQL: sign in with a password, import a work and a season, query them back.

### Acceptance baseline

The standing gate every slice passes before the next begins:

- `go build ./...`, `go vet ./...` and `go test -race ./...` pass
- adapter contract tests pass against real PostgreSQL
- migration tests run from an empty database
- import boundary checks pass (`test/sdkboundary` builds `test/sdkprobe` as an external module; `capabilities/reference` is parsed for its imports)
- the boundary conformance suite passes — it reflects over `*app.Service` and fails when a caller-bearing method is missing from its table or does not answer `Unauthenticated` then `PermissionDenied` ([ADR 0066](adr/0066-authorization-is-carried-in-the-type.md))
- the end-to-end test signs in over `AuthService`, then subscribes and navigates over `SessionService`, asserting the pushed content region
- health probes answer against a running process

## What is next

The critical path is complete and the platform runs. Four threads are open, and two of them are half built.

**The active thread is remote playback** (immediately below) — **built through the continue-watching payoff (6a); what remains is materialise-on-commitment (6b), cache-first rendering (4) and the affordance gate (7).** A film or series added from a Stremio addon plays in a browser, chosen out of the whole candidate set and remuxed per stream where the browser cannot decode what the source had; it remembers where you got to, and now *surfaces* it — a continue-watching rail on home with resume-progress artwork, a watched check on finished episode rows, and a "Next episode" control beside the player. What it does not yet do is play a virtual item straight from its `ContentRef` and materialise it only on sustained watching (6b).

Behind it, **telemetry and observability** — **built through the expert-mode UI**; metrics, the audit store and OTLP export remain. It was the limiting factor on development speed across seven repositories, and it is no longer: a click now carries one trace id to the committed row.

Behind that, **more than one user** (ADRs 0066–0069, partly built) — Mosaic has never had a second account, which is the difference between a household media server and a personal one.

A fourth thread was opened after those: **two module tiers and how a module is distributed** (ADRs 0062–0065), which supersedes the per-install build pipeline the Supervisor records assumed. Its first piece is now built — two **core** metadata modules under ADR 0062's guarantee clause, replacing the Stremio addon that was standing in as Mosaic's guaranteed metadata source: `module-tmdb` for richer data behind an API key, and `module-cinemeta` as the zero-configuration floor the clause turned out to need ([ADR 0072](adr/0072-the-guaranteed-metadata-provider-needs-no-credential.md)) — while the delivery half is now largely built — the out-of-process extension boundary works end to end (wire, harness, host, handle, lifecycle, egress proxy), and what remains is the CI-built signed Platform binary (Supervisor-shaped, for core modules) and the Platform's own signed-artefact install path for extension modules ([ADR 0079](adr/0079-the-platform-manages-extension-modules.md), buildable rather than Supervisor-blocked). The rest (exports, jobs, the remaining SDK gaps) runs around all four.

### The active thread: remote playback, the first consumer capability — **built through slice 6a (the continue-watching payoff); slice 4, slice 6b (materialise-on-commitment) and slice 7 unbuilt**

The library was **inert**: `Import` snapshotted a `RemoteLocation` Part whose `Ref` is a direct URL or a magnet ([ADR 0028](adr/0028-virtual-and-materialized-content.md)), nothing resolved it, nothing served bytes, and the `Play` action resolved to a toast. That is fixed — `module-remote-playback` (`v0.2.0`) resolves a Ref, the Platform mints a ticket and relays the bytes from its own origin, and a `Player` node renders in the Shell. **The durable layer is now built too** (slices 5 and 6a): position is remembered and keyed to the viewer and the node, and it is surfaced — resume on the detail, a continue-watching rail on home, watched marks on episode rows, and a next-episode control. What remains of the thread is materialise-on-commitment (6b), cache-first rendering (4) and the affordance gate (7).

**Stremio is the reference, and what it shows is the split, not the protocol.** Mosaic already has the addon protocol, complete ([ADR 0037](adr/0037-completing-the-stremio-source-surface.md)). Stremio is *three* parts: addons return stream descriptors; a local **server** turns a descriptor into a seekable byte-range HTTP origin (torrent engine, remux, subtitle conversion, cache); a player consumes that origin, reporting a position that syncs independently of whichever addon served the bytes. Mosaic had the first and neither of the others, and reading the module contract as "addons" is what kept the gap invisible. **It now has two and a half:** the origin exists and remuxes, but it serves only what a debrid service already resolved to a URL — there is no torrent engine — and the player reports no position.

Four decisions were written ahead of the build, as [ADR 0027](adr/0027-modules-as-typed-capability-providers.md) and [ADR 0028](adr/0028-virtual-and-materialized-content.md) had been. **Three of them are built, each in part; [ADR 0046](adr/0046-playback-state-is-platform-owned.md) is untouched**, which is why nothing is remembered. The unbuilt parts are named in the sequence below.

- **[ADR 0045](adr/0045-playback-consumer-and-media-origin.md) — the playback consumer contract and the Platform-hosted media origin. — built for `Direct`; `Served` is not implemented by anything.** `RolePlayback`/`PlaybackProvider` is the first **consumer** role in a vocabulary that had been source-only. A resolution is discriminated — `Direct` (a URL, from an addon's plain `url` stream or a debrid service) or `Served` (an `io.ReadSeekCloser` the module produces, for a torrent engine or a subtitle transcode). The module never speaks HTTP: the Platform mints a signed, short-lived, session-bound ticket, hosts `GET /playback/{ticket}` and `http.ServeContent`s the reader, so ranges and seeking come free and the viewer's IP and any debrid token stay server-side ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)'s argument, applied where it matters more). Streams and subtitles from other modules are resolved **by the Platform** and passed in as data, keeping [ADR 0017](adr/0017-how-a-capability-acts.md)'s cross-module authority seam closed.
- **[ADR 0046](adr/0046-playback-state-is-platform-owned.md) — playback state is Platform-owned. — not built.** Position, duration and finished, keyed by **(user, node)** — not by part, because a user resumes an episode, not a particular release of it. A fifth Platform store beside the four content stores; written through `ContentService` as the invoking user; read as an in-progress query that the continue-watching rail renders server-side. The playback module cannot own it ([ADR 0012](adr/0012-capabilities-do-not-own-stores.md)), and should not: a local-file player and an export module read the same surface.
- **[ADR 0047](adr/0047-player-as-client-primitive.md) — the player is a client primitive over a server-issued ticket. — built for one direction.** The SDUI thesis gains its first stated limit, and a narrow one: **the server owns everything about a playback session except the decoding pipeline and the transport controls.** `playPart` resolves to a `Player` node carrying the ticket URL and a resume offset, and the client renders it. **The other direction does not exist**: the client declares no capabilities on `Attach` and reports no progress back, so the node carries no subtitle tracks and no next-up reference and the resume offset is always zero. Those are the same gap seen from two ends — slice 3's hard-coded browser preference and slice 5's missing state. Same move as `AppShell` ([ADR 0031](adr/0031-server-owned-app-shell.md)) — a structural primitive owning a mechanism the server names but does not micro-manage.
- **[ADR 0048](adr/0048-stream-selection-against-a-client-profile.md) — stream selection against a client profile. — built, against a server-assumed profile rather than a declared one.** The decision that makes playback *work*, and the one place Mosaic should **not** copy Jellyfin or [remux](https://github.com/lostb1t/remux). Their decision engines read a device profile and fall back to transcoding, which is correct when a library is one file per item. A Stremio aggregator returns *forty* candidates per item — different releases, containers, codecs, qualities — so the library is a menu, and transcoding an incompatible MKV does expensive work to reach a result already sitting in the list. **The Platform ranks candidates against the declared profile and picks the best playable one**; the user can override through the source picker; no playable candidate is an honest state with counts and reasons. The ranking is built and reports what it chose out of how many; the picker and the honest empty state are not. Whole-file transcoding stays deferred — what the build added instead was the finer axis [ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md) names, encoding one stream and copying the rest.

**The browser is a deliberately limited-mode client**, accepted knowingly. MSE takes only fMP4 and WebM, so **Matroska cannot pass through it whatever codec is inside** — a 1080p h264 MKV is as unplayable as an HEVC one — and Chrome decodes no AC3/EAC3. Selection cannot manufacture a release no scraper offers, so much of what it finds will be h264/AAC at 720p/1080p while 2160p HEVC stays out of reach in a browser. What closed most of that gap was not the renderer but the **per-stream decision** (slice 2): an MKV is rewritten into fMP4 with its video copied, and only the audio it cannot decode is re-encoded.

**The web player is a bare `<video>` element, not Shaka** — a revision of what this thread first planned, recorded as [ADR 0070](adr/0070-the-web-player-is-the-browser.md), which supersedes that bullet of ADR 0047 and the profile-source clause of ADR 0048. Shaka adds no codec support of its own — the browser decodes either way — and both paths the Platform serves today are progressive MP4, so it would forward to the same element behind a dependency. The trigger to revisit is a served format, not a date: it goes in when the origin emits HLS. Until then the browser's own `canPlayType` is what a profile would be measured from. **A desktop client with libmpv is the real answer to a fat profile** — it is what Stremio and Seanime both ship, and it is a later thread.

**Build sequence — pick the lowest unbuilt slice; each passes `go build`/`go vet`/`go test -race` before the next:**

1. **The skeleton — one file, playing, end to end. — done.** The thinnest real vertical, the method the first optional module used: `RolePlayback`/`PlaybackProvider` with **only** `Resolve` returning **only** `Direct`; `ResolvePlayback` reading the Part and minting a ticket; `internal/transport/playback` relaying with range pass-through; `module-remote-playback` resolving an `http(s)` Ref; a `Player` node ([ADR 0047](adr/0047-player-as-client-primitive.md)) rendered by the web runtime's `Player` primitive. No state, no selection, no `Served`, no lifecycle. It existed to hold the two contracts with the most risk — the player and the ticket — before anything was built on them, and both held: the ticket is minted in the transport and the upstream URL never leaves the server, `playPart` resolves to a bare node that rides its own region so the screen underneath survives the player closing.
2. **The ACL, the probe and the per-stream decision — done** ([ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md), [ADR 0051](adr/0051-modules-as-anti-corruption-layers.md)). Slice 1 was run against a live AIOStreams instance and failed in a way that rewrote this slice. The resolved URL carried its container in a **query parameter** (`…MIRCrew.mkv&name=…&media_id=…`), so the extension heuristic saw none and would have relayed ten gigabytes of Matroska to a browser. Two fixes, and neither is more parsing.

   **The module translates at the boundary** — a dialect table keyed on addon manifest id, an explicit list of sources actually tested, a generic fallback for the rest, and the SDK's typed fields (`Part.Container`, `VideoCodec`, `AudioCodec`) filled instead of left empty. [ADR 0043](adr/0043-repository-naming-convention.md) always called a module an anti-corruption layer; this is it doing the job, and a community member adding a dialect is a module bump touching nothing else.

   **ffprobe decides what a release actually is**, run against the resolved URL — the approach [remux](https://github.com/lostb1t/remux) uses (`playback/probe.rs`), and strictly better than reading a filename. It range-requests the header rather than downloading the file, so asking is cheap even for a thirty-gigabyte release. The parse ranks the list; the probe settles the winner. **The result is not persisted on the Part**, as the plan called for: it is sealed into the ticket so the origin does not re-probe on every range request a seeking player makes, and it is re-derived on the next play. Persisting it is worth doing with the resolution cache, since both are the same durable/perishable question.

   **And the decision is per stream, not per file.** Copy the video, encode only the audio — which is why the origin plays an h264+E-AC3 release in a browser that decodes neither E-AC3 nor Matroska. "Remux or transcode" was too coarse an axis and the code written for it said so; it was corrected here. Building it added one thing the record did not predict: **HDR is tone-mapped rather than copied**, because passing HDR10 metadata to an SDR browser decoder produces the purple-and-green picture that made the bug obvious.

   **What did not land with it: HLS.** The plan called for anything encoded to be emitted as HLS, which is what makes an encoded stream seekable and would retire ADR 0045's non-seekable caveat. The origin emits **fragmented MP4 off a pipe** instead — it has no index and no length, so it answers `Accept-Ranges: none` and a player treats it as unbounded. **A remuxed stream therefore cannot be seeked or resumed**, which is stated rather than worked around: it is the constraint slice 5's resume mechanism will hit, and segmenting is what clears it.

3. **Selection, capability classes and the resolution cache — selection, the declaration, the classes and the cache are built; the source picker and the no-candidate state are not** ([ADR 0048](adr/0048-stream-selection-against-a-client-profile.md), [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md)).

   **Built.** Container, video codec and audio codec are carried on the attach-part command and filled by the module's dialect table; **import writes the whole candidate set as Parts** instead of one, sampled across resolutions rather than off the head of the list, and a re-import tops the set up; `ResolvePlayback` filters and ranks against a `PlaybackPreference` and reports what it chose out of how many, so a bad pick is distinguishable from a short list. `FirstPlayablePart` walks a work to its entry point.

   **The client now declares what it can play, and the hard-coded browser preference is gone.** `AttachRequest` gained an optional `ClientProfile` (sdui `v0.10.0`), the web Shell fills it from the browser's own `canPlayType` rather than from a table — which is what makes a Safari that decodes E-AC3 and a Chrome that does not resolve to different releases without either being told about the other — and the Platform stores it on the live session, so one read serves both the ranking and the per-stream plan. Ranking them against different profiles would re-encode the very release selection chose to avoid re-encoding. **HDR joined `PlaybackPreference`** at the same time: a client that cannot render it penalises HDR candidates, because the fix is a tone-map and a tone-map is a full video re-encode — the purple-and-green failure of slice 2, moved one step earlier where it costs nothing.

   **The capability class and the resolution cache are built, and verified against real PostgreSQL.** The class is a sorted digest of the declared profile, computed once on `Attach`; sorting is the load-bearing part, because two clients listing the same codecs in a different order must land in one class or the cache fragments with no symptom other than never seeming warm. `playback_resolutions` is keyed by **(part, capability class)** and by nothing else — not by user, not by device — and sits off `Tx` deliberately: it has no outbox event to be atomic with, and ADR 0049 requires its write not to block the stream. The integration test proves the hit by *removing the module*: a second Service with no playback capability registered, the configuration that otherwise fails with "no playback module is installed", resolves the same part successfully — so the answer can only have come from the cache. A different class misses, and an unclassed client bypasses the cache rather than sharing one bucket.

   **Verified live, and the verification found a silent failure.** Against the local stack with a real AIOStreams instance: the Platform logs `client profile declared class=fc8cfc2f0ea05c11 video_codecs=5 audio_codecs=5 hdr=false max_height=1080`, two separate sessions derive the same class, a cold play reports `cached=false candidates=40 direct_play=true` and writes a `playback_resolutions` row, and the next play of the same release reports `cached=true` and plays from it. Invoke went from **3.76 s cold to 2.42 s warm** — the aggregator call, removed.

   It did not work on the first attempt, and the way it failed is the useful part: the Shell was pinned to `@mosaic-media/sdui@^0.9.0`, whose bindings predate the field. Connect-ES serialises only what its schema knows, so the client sent the profile, the wire dropped it, Attach returned `200`, and `GetProfile()` was nil — no error at any layer, and the host typechecked clean because the host's `node_modules` held a different version from the container's. `web`'s CLAUDE.md had also claimed the lockfile links that package to the sibling checkout; it does not, and that wrong belief is what made a stale version invisible. Both are corrected.

   **The probe is now durable too, and the play is effectively instant** ([ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md)). ffprobe used to run against the resolved URL on every play; once the resolution cache removed the aggregator call it *was* the remaining latency. It is now written onto the Part and reused: measured live, the same play went **3.75 s cold to 11 ms warm**, logging `cached=true probe_reused=true`.

   **ADR 0050's storage plan was not sufficient, and the record's own example is why.** It said probe results belong in the Part's empty technical columns. Those columns cannot express a track list — and the release that motivated the record has four audio tracks whose first is Hindi. Persisting only `AudioCodec` would have let the second play skip the probe and then choose a *different* audio track from the first, a regression the columns could not even represent. So the whole track list is stored as a versioned document in `Part.Attributes` (merged, never replacing another writer's keys) and the columns carry the summary — with `AudioCodec` set from the track the plan *would* choose rather than the first one, so the column and the decision agree. An unrecognised document version decodes as "not probed", costing one ffprobe run rather than risking a wrong plan.

   The probe is also authoritative over the module's parse, which is ADR 0050's demotion of release-name parsing made real: the live run corrected a Part from `mp4` to `matroska`, and **HDR is cleared unconditionally** — a name-derived HDR guess that the bytes contradict would otherwise tone-map an SDR file forever.

   **One honest limit.** Recording a probe writes to the content graph, so it authorises `content.bind`, which a read-only viewer does not have. That viewer still plays; they simply never warm the cache, and every one of their plays re-probes. It is the correct refusal and the wrong outcome, and the missing piece is the **system principal** ([ADR 0017](adr/0017-how-a-capability-acts.md)) — work that belongs to the install rather than to whoever pressed play.

   **Departures worth recording.** A cache hit reports an empty `ModuleID`, which is accurate — no module was asked — and has the side effect that a cached play survives an uninstalled or unreachable playback module. The cache write happens on the request goroutine rather than in the background, which still satisfies the record: nothing is streaming yet when it runs, the client has not been handed a ticket, and a write failure is logged and swallowed rather than failing the play. And a client that declares no display ceiling still gets one on an *encode* (1080p), because an uncapped panel is not an argument for an uncapped encoder.

   **Not built.** The **origin's invalidate-on-read retry**: a dead cached link currently fails the play instead of being re-resolved transparently, which is the half of ADR 0049 that makes the cache safe rather than merely fast — it needs the ticket to carry the part and the class so the origin can re-ask. There is no **source picker** and no **no-candidate state**, so a user cannot override the ranking and an item with nothing playable presents as a failure rather than as an answer with counts and reasons. The **refresh cron** remains blocked on the jobs runner, the scheduler and the system principal, as ADR 0049 said it would be.

   The shape that makes it fast is the **durable/perishable split**: candidates persist forever, resolved URLs are cached lazily per capability class and corrected when they fail. Invalidate-on-read, with the origin re-resolving transparently on a failed fetch, is the mechanism; a TTL is only a hint, because a debrid link also dies whenever its torrent leaves the provider's cache.
4. **Cache-first rendering — not built** ([ADR 0052](adr/0052-cache-first-rendering-and-source-health.md)). Found by restarting the Platform under a live client: the home screen rendered *"Nothing to show yet — try adding an addon in Settings"* on an install with a configured addon and a library full of content. The client was blameless — it reconnected unprompted and the server rebuilt and pushed — but the emit-side discards catalog errors (`if err == nil`), so when every cold call failed the page had no rows and fell through to the empty state. A transient upstream failure presented as a configuration mistake, which is worse than an error because it sends a user to fix something that is not broken.

   The fix: source-backed screens render from a durable snapshot when one exists, revalidate in the background, and push the live result as a `RegionUpdate` — the first real use of the op-set ADR 0041 defined and nothing has exercised. The snapshot holds *items, never the rendered tree*: artwork and playback URLs are signed with process-scoped keys, so a cached tree would come back after a restart full of URLs signed by a key that no longer exists, producing a page that looks right and is broken. A source that stays unreachable gets a **persistent** shell notification rather than a toast, because a transient message for a lasting condition is announced once and then invisible.

5. **Playback state — built; the emit-side payoff is slice 6** ([ADR 0046](adr/0046-playback-state-is-platform-owned.md)). The store, the migration, the commands and queries, coalescing on the session, and Resume on the detail screen. `playback_states` is the **fifth content store and the first per-user one** — everything in the object graph until now was install-global — keyed by **(user, node)** and never by Part, so watching half from one source and finishing from another is one position rather than two half-watched copies. The release is recorded *beside* the position so a resume returns to the same encode.

   **Finished is derived, then sticky**, as the record has it, plus a floor the record does not anticipate: a player reports a duration of a few seconds while metadata loads, and at that value almost any position is 95% of the way through, so without a minimum an item marks itself finished the instant it starts. A finished item resumes at the beginning rather than at its stored position, which is thirty seconds from the end.

   **Progress coalesces on the session** beside the existing input debounce, with the opposite end condition: a dropped keystroke is harmless because another follows, and a dropped *final* position is the one that mattered — so a final report bypasses the timer, and a reaped or shut-down session flushes what it was holding. `reportProgress` is also the first **silent** action: Invoke's default is to confirm and re-render, which would have made a playing film raise a "Done" toast four times a minute while re-rendering the screen underneath itself, tearing down the player that was reporting.

   **Two things the live run found that no test could.** A new action does not reach an account that already exists — presets are *snapshotted* into role rows at creation — so progress recording failed with `no role grants "playback.write"` on an install whose admin predated the action. The bootstrap now reconciles the owner account's role on every boot, matching on "this account holds exactly one role" rather than on the role's *name*: matching by name found nothing on the very install that motivated it, whose bootstrap role is called `Administrator` from a build that named it differently. Separately, a nil store returning an empty answer is indistinguishable from "you have not watched this", so a forgotten wiring would have presented as a resume feature that silently never worked; it is `Unavailable` now.

   **Verified live:** watched to 41:15, closed the player, and the detail screen came back offering **Resume 41:25** beside **Start over** — which resumed at 41:35, the stored position plus the seconds that played before the report. **It runs into slice 2's carried gap:** a remuxed stream cannot seek, so resume is exact only on a directly relayed one until segmenting lands.
6. **The payoff — built as 6a (the emit-side over materialised content); 6b, materialise-on-commitment, deferred.** Continue-watching on home, watched marks on `EpisodeRow`, and next-episode continuity: search → add → play → resume now works end to end for an in-library item. Three findings reshaped it from the "mostly emit-side" it was planned as:

   - **Artwork had to move onto the node** ([ADR 0071](adr/0071-content-artwork-is-stored-on-the-node.md), partly superseding [ADR 0034](adr/0034-rich-metadata-preview.md)). The rail renders many items on every home load, and re-deriving each card's art from the provider is a round-trip per card — worse, a materialised node cannot be turned back into a provider-bearing ref at all (the module id and the provider's native type are stored nowhere), so it was not merely costly but impossible. The Stremio module now stores the poster/backdrop/logo it already fetches, and the rail reads them from one node read. **This — not the relation read — was the real load-bearing gap.**
   - **Watched marks bridge the preview to the tree.** The detail lists a series' episodes from the provider's live preview (season/episode numbers, no node id); playback state is keyed by node. The marks come from walking the materialised tree (`GetContentNode`+`WithChildren`, the number carried as `NaturalOrder`) to map each preview episode to its node, then one batched `ListPlaybackStates`. A virtual series has no nodes and shows no marks.
   - **Next-episode is the containment tree, not the relation read.** "Next" is the next sibling in the season or the first episode of a later season — a Node-tree walk. The player region now carries a server-emitted **Next episode** `Button` beside the player, pushed as a second region update (`APPEND`): generalising a dispatched action from one pushed surface to a *sequence* was the small transport change it needed — the two-lane transport ([ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md)) exists exactly so the server drives the region unprompted, and one action driving two region updates is that capability, not a workaround. No client change: the Shell renders whatever nodes the region holds.

   **Verified live** against Cinemeta: adding *Breaking Bad* stored real artwork on the work node; the continue-watching rail rendered its poster with a resume-progress bar and the episode label, above Trending; the finished episode carried a watched check on the detail while the in-progress and unseen ones did not. **Not demonstrated live:** next-episode *playback* — the control emits and dispatches, but a metadata-only fixture has no playable Part to advance to, so end-to-end auto-advance awaits a streaming source. The rail's playback-state rows were seeded, because producing them naturally needs a real stream and a debrid the Cinemeta fixture lacks.

   **6b — materialise-on-commitment — is deferred.** Playing straight from a `ContentRef` and writing to the library only past a watch threshold still needs a play-from-virtual-ref path (playback requires a materialised Part today) and a decision on a real collision: [ADR 0028](adr/0028-virtual-and-materialized-content.md) admits "only two crossings" while [ADR 0046](adr/0046-playback-state-is-platform-owned.md) keys progress by node — and pre-commitment progress has no node to key on. That is its own record and its own slice.
7. **The affordance gate — not built** ([ADR 0036](adr/0036-capability-gated-affordances.md)). Moved to the end, and the ADR's own sequencing revised with it: it argued the gate is buildable before any consumer exists, which was right while playback was indefinite. Building it *first* now would mean writing the predicate against a role interface that does not exist and shipping an empty state about to be exited. "Consumer absent" is only testable once "consumer present" exists to contrast against.

**Verification fixture: AIOStreams + TorBox, and it earned its keep.** The owner runs it, and it was a better fixture than a synthetic one: it is what caught the container-in-a-query-parameter bug that rewrote slice 2, the HDR copy that produced a purple picture, and a candidate list whose head was all one resolution. AIOStreams resolves the debrid side itself and returns streams carrying a `url`, which [`Stream.Ref()`](https://github.com/mosaic-media/module-stremio-addons) prefers over a magnet — so **the playback module needs no debrid API integration at all**, and its config-path manifest URL is the shape `normaliseAddonURL` already handles.

**Deferred and named, not omitted:**

- **Segmented output (HLS)** — the seekability gap slice 2 left open, and the closest thing to a prerequisite the remaining slices have: without it, resume works only on a directly relayed stream.
- **Full transcoding.** Deferred rather than rejected — it is the right answer for local files and for what per-stream encoding cannot cover, and it costs session lifecycle and hardware acceleration on top of the ffmpeg dependency the origin now has. It is also the **only** route to true adaptive bitrate: seamless mid-stream switching needs one encode segmented into an aligned ladder, and a menu of unrelated releases cannot supply that at any level of effort ([ADR 0048](adr/0048-stream-selection-against-a-client-profile.md) records why). Bandwidth-aware selection and stall-triggered reselection are what Mosaic gets until then.
- **A desktop client with libmpv.** The fat profile that makes most of the above moot for the users who install it.
- **The torrent engine** (the other half of `stremio-server`): sequential piece selection, cache and eviction — the `Served` half of ADR 0045's resolution. Until it lands, a magnet Part no debrid service resolves is honestly unplayable rather than silently broken. It is what forces **module lifecycle** (`Start`/`Stop`), a **Platform-granted scratch directory**, and — for prefetch and eviction, which have no user — the **system principal** [ADR 0017](adr/0017-how-a-capability-acts.md) reserved.
- **The URL refresh job.** Keeping in-progress items (and the next episode of an in-progress series) warm in the resolution cache. Deliberately split from the cache's own read path, which is self-sufficient: the job needs a scheduler, the jobs runner (the `jobs` tables exist with no service) and — since background work has no session — the **system principal** [ADR 0017](adr/0017-how-a-capability-acts.md) reserved. Those are one slice, and invalidate-on-read must not wait on them.
- **Per-user provider credentials.** Module settings are per-module, not per-user ([ADR 0021](adr/0021-module-settings.md)), so one debrid account serves the whole install. Harmless while the Platform relays and no credential reaches a client, but it is a real question [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md) names and does not answer.
- **Multi-user visibility.** Playback state is the first content state that differs between two users of one install. Whether a household shares a continue-watching rail is a real question ADR 0046 opens and does not answer.

**The gap this thread was expected to make load-bearing — the relation read (`ListFrom`/`ListTo`) — turned out not to be.** Next-episode and series rollup are the **containment tree** (series → season → episode as Nodes), reachable through `GetContentNode`+`WithChildren` and `Node.ParentID`, which already existed; the relation graph carries only associations (adaptation, sequel, collection) and no episode order, so slice 6 never needed it. It stays genuinely open, but as tidiness, off this thread's critical path. What *did* prove load-bearing was turning a materialised node back into renderable art — met by storing artwork on the node ([ADR 0071](adr/0071-content-artwork-is-stored-on-the-node.md)). (The naming gap is closed: [ADR 0043](adr/0043-repository-naming-convention.md) now names the consumer case explicitly — a consumer module names its *capability*, because it has no upstream, and `module-remote-playback` is the worked example.)

### The next thread: telemetry and observability — **built through the expert-mode UI; metrics, audit and export remain**

The build spans seven repositories and more than one process, and a bug crosses several of both. A user action passes through the Shell, the session transport, a command handler, a module in its own repository, a third-party HTTP call, PostgreSQL and the outbox worker — seven hops, four repositories — and **nothing joined them**. Diagnosing a failure meant reading timestamps in three places and guessing. That cost was the limiting factor on development speed, which is what made this a thread rather than a chore.

What existed was better than it looked, and is what these records built on rather than replaced. `diagnostics.Logger` already had a **fail-closed** redaction model — a field is redacted unless explicitly classed safe. `domain.Event` already carried `CorrelationID` and `CausationID`, with a comment saying they were *"Empty until request-scoped propagation exists."* The `event_outbox` already committed atomically with the state change that produced it. The seven error categories were a ready-made span status and metric dimension. Config already declared a reload class per field.

What was missing was that almost none of it was used: the good logger was called from three non-test places, the boot narration was `fmt.Printf` and the session transport's lines were `log.Printf`. That is now inverted — telemetry is ambient in `context.Context` and reached with one line at a call site, two `Printf`s survive in non-test code, spans cover the nine seams, and a click in the browser and the row it commits share one trace id.

Eight decisions, written ahead of the build:

- **[ADR 0053](adr/0053-telemetry-is-ambient-in-context.md) — telemetry is ambient in `context.Context`. — built.** No constructor takes a logger; every method already takes a `ctx`, and that is the carrier. One line at the call site, full correlation in the output — the friction that pushed people to `fmt.Printf` is gone, and so are all but two of the `Printf`s.
- **[ADR 0054](adr/0054-the-correlation-id-is-the-trace-id.md) — the correlation ID is the W3C trace ID. — built.** No second identifier. The Shell mints it where the user actually clicks, a Connect interceptor carries `traceparent` on every call, `Event.CorrelationID` finally has a source, and the outbox *links* rather than parents so a later publish does not appear as a request that hung for minutes. Implementation corrected one thing: the span tree roots at the caller's span rather than at a phantom the server invented.
- **[ADR 0055](adr/0055-instrument-at-the-seams.md) — instrument at the seams, not the call sites. — built.** Nine of them. The uniform command handler order is the decisive one: the rule that made the codebase uniform is what makes it observable, and a new handler is instrumented because it is a handler. Seam nine — `netguard.ModuleClient`, handed to every module instead of `nil` — closed the SSRF gap alongside the tracing.
- **[ADR 0056](adr/0056-redaction-classes-are-the-pii-boundary.md) — redaction classes are the PII boundary; no scrubbing pipeline. — built, except its enforcement.** Redaction at construction, so a sensitive value is never buffered or written at all, and `Identifier` (salted hash) so occurrences correlate without storing who. **The vet-style check is not built** — the rule is currently developer discipline, which is the arrangement [ADR 0066](adr/0066-authorization-is-carried-in-the-type.md) demonstrated does not hold. The `licenseheader` tool is the model to copy.
- **[ADR 0057](adr/0057-audit-is-a-store-not-a-log-stream.md) — audit is a store. — not built.** `audit_log` on `Tx`, so the record commits if and only if the change did; append-only by database grant rather than by policy; read through the existing ABAC engine. The deliberate reversal: an audit write failure fails the command. There is no `audit_log` migration; audit retention is configured and floored in code with nothing writing the table it governs.
- **[ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md) — dual-sink storage and an in-product expert mode. — built for logs and spans.** File always (it is what records PostgreSQL failing); PostgreSQL additionally (migrations `0014`, `0015`), to make the UI queryable. An administrator sees logs and a **trace waterfall** inside Mosaic — no container, no external stack — reached only with `telemetry.read`, and the affordance is hidden from anyone without it. Retention is per-signal and read from the Active configuration, with audit floored in code. **Metrics and audit are absent from it**, because neither signal is produced yet; retention is declared but nothing deletes, since deletion needs the jobs runner below.
- **[ADR 0059](adr/0059-modules-observe-through-the-sdk.md) — the Platform owns the observability plane; modules hook into it through the SDK. — built (SDK `v0.13.0`, Stremio module `v0.17.0`).** A dependency-free `v1.Telemetry`, reached ambiently as `TelemetryFrom(ctx)` so one pattern covers `Import` and all seven provider roles. The Platform stamps attribution a module cannot forge and enforces per-module quota; a module configures no exporter, sink or retention. **The SDK does not take an OpenTelemetry dependency** — it has none today, that is load-bearing, and exporting OTel would publish an implementation choice as a contract.
- **[ADR 0060](adr/0060-the-supervisor-observes-independently.md) — the Supervisor observes itself, file-only.** Neither of ADR 0058's sinks covers a Platform that *never starts* — a failed migration, a bad DSN, a crash loop — because both live inside it. The Supervisor records the lifecycle a process cannot report about itself, keeps no database and no exporter, serves a minimal status page when the Platform is down, and merges into expert mode when it is up. **The dependency never inverts:** the Platform stays observable standalone.

**The shape of the answer, and what it rejects.** The instinctive stack — Collector, Prometheus, Loki, Tempo, Grafana — is right for an operations team and wrong for a self-hosted media server, where five more containers means nobody bothers. The split that resolves it: **instrumentation is architecture and is always on; export is configuration and defaults to nothing.** What that meant in the build is narrower than "instrument against the OTel API": the *formats* are the standard's — W3C `traceparent` on every edge, OTel's span and resource semantics — while the implementation is the Platform's own and takes no OpenTelemetry dependency. That is what keeps an OTLP exporter additive rather than a rewrite, and it is the same argument [ADR 0059](adr/0059-modules-observe-through-the-sdk.md) made for the SDK. The full stack is to ship as a `docker-compose.obs.yml` *developer* profile, with the exporter; neither exists yet.

**Order, and where it has reached.** Facade and process identity → correlation across processes → dual-sink storage → spans at the seams, including the SDK surface modules reach → the expert-mode UI → **metrics and the audit store → optional OTLP export**. Storage landed third on purpose: "queryable" cannot be retrofitted onto a file sink. The SDK surface rode with the seams because the module seam is where it is seeded, attributed and quota-bounded. **The first five are built, and they were what fixed the multi-repository debugging problem**; the two in bold remain, and they are the compliance and interop story rather than the diagnostic one. Nothing is instrumented against the OpenTelemetry SDK — the wire formats and semantics are OTel's so an exporter is additive, but the implementation is the Platform's own.

**What this thread waits on, and what waits on it.** Retention deletion and partition management are recurring no-user work, so both [ADR 0057](adr/0057-audit-is-a-store-not-a-log-stream.md) and [ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md) want the **jobs runner, a scheduler and the system principal** — the same trio [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md)'s refresh cron and the torrent engine's eviction already want. Four callers now; it should be built once. Until it exists, partitions are created eagerly and retention is manual — stated, not silent. Nothing else in this thread is blocked on it.

**The Supervisor is decided but not buildable.** [ADR 0060](adr/0060-the-supervisor-observes-independently.md) says how it will observe; [ADR 0004](adr/0004-supervisor-as-host-manager.md)–[ADR 0007](adr/0007-static-go-module-composition.md) remain largely unbuilt and there is no Supervisor on disk, so nothing else in that record can be built yet. **The one piece that could land ahead of it did:** the Platform reads `MOSAIC_BOOT_ID` and adopts an inbound boot id rather than always minting one, so a Supervisor has something to hand over to when it arrives, and a boot has a name in the logs meanwhile. [ADR 0060](adr/0060-the-supervisor-observes-independently.md) also leaves one question open on purpose: where the shared record format lives, given two binaries must agree on it — a packaging decision that belongs with the Supervisor's own scaffold, and one the module-tier thread below now also touches.

**No longer undecided: the module process boundary.** This thread deliberately left the module seam a **wrapper rather than an inline edit**, so that moving modules out of process would replace it rather than duplicate beside it, and declined to write that record. It has since been written — [ADR 0064](adr/0064-extension-module-boundary.md), in the thread below — and it supersedes [ADR 0007](adr/0007-static-go-module-composition.md) for the extension tier exactly as anticipated. It also turns seam 9 from a courtesy into a control: in process the Platform sees a module's outbound calls only when the module uses the client it was handed, whereas through the boundary's forward proxy it sees every host, attributed per module, whether the module cooperates or not. [ADR 0059](adr/0059-modules-observe-through-the-sdk.md)'s surface is unaffected, which is the payoff for declaring an interface instead of exporting OTel — the module calls the same methods over a different transport. **SIEM export** remains a further audit exporter, left until a deployment needs one rather than guessed at now.

### The thread behind it: more than one user — **ADRs 0066–0069; the authorization model is built, the second account is not**

Mosaic has never had a second user. `bootstrap.EnsureAdmin` seeds the first and no transport has ever exposed `CreateLocalUser` — that is in the [unreachable capability](unreachable-capability.md) register, and it is the gap that makes everything below matter. A media server on a household VPS is a multi-user product or it is a personal toy, and every unbuilt piece here sits between those two.

**What forced the thread was a performance bug.** Phase 4 span data showed one search producing thirty-nine queries and ten full authenticate-plus-authorize cycles, because a per-result helper had no way to say "already inside the boundary" and so re-entered a public entry point once per result. [ADR 0066](adr/0066-authorization-is-carried-in-the-type.md) is the fix and is **built**: an `authorized` value only the boundary can construct, every handler entering through one function, plus the conformance suite described in the acceptance baseline above. Pulling that thread exposed the rest.

**Then the model itself turned out to have a hole. — [ADR 0069](adr/0069-privilege-cannot-escalate.md), built.** `CreateRole` accepted any permission set and `GrantRole` bound any role to any user, neither checking what the *caller* held — so `role.create` was really "hold every permission", two steps away: mint a role carrying everything, grant it to yourself. Every careful distinction elsewhere was decorative while that stood. Delegation now intersects with what the granter holds, and the grantable set is computed rather than assumed. It also corrected two earlier records: ADR 0057 and ADR 0058 had reasoned that `audit.read` and `telemetry.read` should be withheld from the bootstrapped account, which does not survive an account that can grant itself anything.

**Four accounts on one box is the whole specification**, and it is more demanding than it looks. An administrator; two adults differing only in whether they want anime; a child limited to PG12. [ADR 0067](adr/0067-three-authorization-mechanisms.md) finds that only the first is an action question, and that the other two are *different from each other* — one adult's exclusion is a preference, the child's limit is an access control, and building either as the other is a defect in opposite directions. Three mechanisms: roles for actions (**built**), a content scope for visibility (**not built**), preferences for taste (a general per-user preference store with its own policy actions is **built** — it went in for the expert-mode toggle — but nothing reads a taste preference, and it must stay out of the engine when something does).

**The credential side is decided and unbuilt.** [ADR 0068](adr/0068-one-principal-many-credentials.md): password, passkeys, per-key-attenuated API keys and legacy Basic-auth adapters all resolve to one Principal, stamped in one place. No JWT — not for fashion but because a claims-carrying token makes a tightened age limit take effect only when the token expires. Attenuation intersects live rather than at issue time, for the same reason.

**Two things block the child account, and neither is an authorization problem.** Ratings arrive from third-party addons in unvalidated JSONB whose correctness [ADR 0013](adr/0013-object-graph.md) assigns to the writing capability, so a child's safety would rest on data the Platform neither owns nor checks — that needs a Platform-owned field and an unrated-means-invisible default, which will hide a substantial part of a real library. Separately, the discovery plane returns *virtual* results with no stored attributes at all, so scoping the library does not scope search. Both are stated in [ADR 0067](adr/0067-three-authorization-mechanisms.md); neither is solved there.

**Order.** Content scope and the scoped `NodeQuery` first — it is what makes a second account mean anything, and it forces the request-scoped principal everything else needs. Then the admin surface, since users and roles have services, policy actions and passing tests but no way to reach them. Passkeys and API keys after, in either order. The legacy adapter last, and deliberately so: a Jellyfin client cannot ask to be scoped, so it is safe only once no code path can produce an unscoped read.

### The fourth thread: two module tiers, and how a module reaches a user — **two core metadata modules are built; the delivery half of ADRs 0062–0065 is not**

Every module today is a Go library statically linked into one binary ([ADR 0007](adr/0007-static-go-module-composition.md)), and [ADR 0002](adr/0002-module-storage-and-delivery-model.md) went further: essential and community modules "differ only in delivery… Architecture: Identical." That symmetry was a deliberate anti-second-class-ecosystem stance and it has been proven twice — both modules are their own repositories importing only the SDK. **It does not survive contact with what the two kinds of module actually need**, and four records say what replaces it. Only the *classification* half is built (below); the delivery half is written down so the Supervisor is not scaffolded blind.

**Built: `module-tmdb`, the first core module under the guarantee clause.** Mosaic's guaranteed metadata source was a Stremio addon — Cinemeta, bundled *inside* `module-stremio-addons` and prepended to the user's addon list. [ADR 0035](adr/0035-metadata-as-required-capability.md) recorded that as unresolved in its own Status line ("whether the default belongs to the Platform or to the module is a question this record answers one way and the code answers the other"), and [ADR 0062](adr/0062-two-module-tiers.md) answered it: a metadata provider Mosaic *guarantees* cannot live inside a module a deployment might not install. `module-tmdb` (`v0.1.0`) is the provider that can — core by tier, ordinary by shape, its own repository importing only `sdk` and `contracts`, registered through the same capability registry as the other two. It fills `RoleMetadata`, `RoleSearch`, `RoleCatalog` and `RoleSettingsUI`, and deliberately **no stream role**: a TMDB import is a Work and its season/episode tree with **no Parts**.

It also closes two of [ADR 0034](adr/0034-rich-metadata-preview.md)'s recorded gaps that the addon protocol structurally cannot carry — **clearlogos** (a per-language `images` collection rather than one string most addons leave empty) and **cast with character names and headshots**. And it made [ADR 0035](adr/0035-metadata-as-required-capability.md)'s composition-root check real: the serving composition now fails before the serve loop when no registered module fills `RoleMetadata` *and* `RoleSearch`, which that record specified and nothing had implemented.

**Five things it did not do, named rather than left to be discovered:**

- **It does not make first boot zero-configuration, and that was the point of the guarantee clause.** TMDB has no anonymous access. The composition check passes because a provider is *registered*; a user still sees nothing until they paste an API key. **ADR 0035's requirement is met in letter and not in spirit, and no TMDB-based module can meet it in spirit.** *Closed by `module-cinemeta` below, which needs no credential; the alternatives considered first — a Mosaic-held key, or onboarding that collects one before the binary is servable — are recorded in [ADR 0072](adr/0072-the-guaranteed-metadata-provider-needs-no-credential.md).*
- **The bundled Cinemeta default was still in `module-stremio-addons`.** It was deliberately left because with TMDB needing a key, removing it would have made a fresh install strictly worse: no addons, no key, no metadata at all. **That objection stopped holding once `module-cinemeta` landed, and the removal is now taken** (`v0.20.0`) — leaving it would have shown every title twice in search, since the Platform unions providers without cross-provider dedup. One thing survives the deletion: the addon **directory** (`addon_catalog`), which the bundled default quietly provided and which is the entire reason the settings screen can offer installable addons rather than demanding a pasted URL. It is a discovery source, read only by the browse section and never merged into what the provider roles source from.
- **Search results do not dedup against IMDb-keyed sources.** A ref carries one external identity and a TMDB search result's is `tmdb/<id>`; TMDB's search endpoint returns no IMDb id and fetching one would cost a round trip per result. So a TMDB result for a title already added via Stremio reads as *new*. The reverse works — an import binds the Work under both `tmdb` and `imdb`.
- **Collections and "similar" were fetched and discarded.** TMDB carries `belongs_to_collection`, which is exactly the franchise data ADR 0034 wanted, and `ContentMetadata` had no field for it. **Closed in `v0.2.0` / SDK `v0.17.0`** — see below.
- **A settings change echoes the API key through the client.** `configureModule` replaces the settings document wholesale ([ADR 0021](adr/0021-module-settings.md), no partial update), so a control that sets a language must carry the key or erase it. The key is never *rendered* — masked to its last four characters — but it rides inside the screen's action payloads, where the Platform's redaction classes ([ADR 0056](adr/0056-redaction-classes-are-the-pii-boundary.md)) cannot see it. The fix is an SDK change: a merge semantic on `configureModule`, or a write-only settings field.

**Built: `module-cinemeta`, the zero-configuration floor — [ADR 0072](adr/0072-the-guaranteed-metadata-provider-needs-no-credential.md).** Building `module-tmdb` showed that ADR 0062's guarantee clause asks for the wrong property: "no install step that can fail" and "no configuration that can be omitted" are different things, and a fresh install needs both. `module-cinemeta` is a **direct client of Cinemeta** — not a second Stremio addon client: it fetches no manifest, negotiates no resources, holds no addon list and applies no ordering policy, because a fixed known upstream needs none of them. It has **no settings at all**, no `RoleSettingsUI`, and a service address that is a constant rather than a field any deployment can reach; what it gives up in generality it gains in being unable to be misconfigured. It fills `RoleMetadata`, `RoleSearch` and `RoleCatalog` (Cinemeta's Popular and Featured collections for both types — the four of its declared catalogs answerable without an argument the browse surface has no way to supply), and like TMDB no stream role: an import is a Work and its season/episode tree with **no Parts**.

It binds content under **`imdb` rather than a scheme of its own**, because Cinemeta's ids *are* IMDb ids — so its search results dedup natively against Stremio-sourced content, which is the gap TMDB has and cannot cheaply close. Two core modules now fill the metadata/search class, which is the arity [ADR 0063](adr/0063-platform-binary-built-by-ci.md)'s role-class table already declares ("one or more") rather than a change to it: Cinemeta guarantees a floor with no configuration, TMDB gives richer data to a user willing to hold a key.

**Verified against the live service by hand, which caught what the hermetic fake hid.** Cinemeta answers `200` for an id it does not know, in *two* shapes — an unknown series returns an empty document, an unknown **film** returns a record echoing the id and type with no name — so testing emptiness the obvious way passes and the Platform materialises a library Work titled `tt99999999`. The name is the test, and both shapes are now pinned. This is the Stremio module's User-Agent lesson in a different place: a source checked against a fake is not a source that was checked.

**What it does not do, and one of them is a seam rather than a gap.** Cinemeta has no clearart, banners, collections, "similar", or character names and headshots on its cast — [ADR 0034](adr/0034-rich-metadata-preview.md)'s recorded gaps, structural to the source, and exactly what TMDB is for. And **the guarantee now rests on a free public service Mosaic does not operate**: it is honest about installation, not about uptime. The seam: **neither core metadata module attaches a Part, and the Platform cannot bridge that** — `ImportContent` routes solely to `ref.Provider`, so a stream provider that did not source the metadata never gets asked. A library built through either plays nothing until a source that *does* index content materialises it. That is the **provider-precedence seam** kept open below, now load-bearing rather than hypothetical, and ADR 0072 deliberately does not close it.

**`module-tmdb` `v0.2.0` — the endpoints the first release left on the table.** Ten endpoints became sixteen, and the module's job was to find what the contract could not express: SDK `v0.17.0` grew `Keywords`, `Certification`, `Similar`, `Collection` and `Trailers` for data TMDB was already returning. **Nothing in the Platform changed**, which is the part worth recording — a module found five gaps, the contract grew additively, the composition root did not move.

Four of the additions are free: `keywords`, `recommendations`, `videos` and certifications fold into the existing detail request through `append_to_response`, so a film detail is still **one** round trip. A fifth was already in a response being made and simply not decoded — `tvdb_id`, so a series now binds under `tvdb` as well as `tmdb` and `imdb`. Three earn a request each: `/collection` for the franchise (returned in popularity order, re-sorted chronologically, because a rail wants release order); **`/find`**, which resolves an incoming IMDb id to TMDB's own and without which the richer provider could not describe a single work in the IMDb-keyed library ADR 0072 makes the floor; and `/configuration` for the CDN layout TMDB publishes rather than promises.

**User-defined catalogs needed no SDK change at all.** `CatalogProvider.Catalogs` already returned a list rather than a constant, so a `/discover` query a user writes ("French thrillers, rated above seven") becomes a browsable catalog beside the built-in eight — a settings question the whole time. The query is raw discover parameters deliberately: a filter builder would model every parameter TMDB has and go stale the moment it added one. It is **sanitised before use** — `api_key`, `page`, `language` and `include_adult` are stripped — because it is free text from a settings screen appended to a request carrying the credential, and without that a query reading `api_key=…` would silently replace it.

**Two limits close, one is new.** Collections and similar now render — but **`ContentService` has no relation read**, so a franchise is re-derived per render rather than persisted: `RelateContent` could write a `RelationCollectionMember` edge and nothing could read it back. Until `ListFrom`/`ListTo` exist, writing one would be writing into a hole. New: the `images` response carries every poster and backdrop variant and `v1.Artwork` holds one string per slot — ADR 0071 anticipates a candidate set, but changing what a stored artwork value *means* is an ADR rather than a field. Deliberately not added, each with a reason: the person endpoints, alternative titles, and `/changes` (which wants the jobs runner).

**`module-tmdb` `v0.3.1` — watch providers, and the field that looks like a source and is not.** Where a title can be streamed, rented or bought, folded into the existing detail request so a film detail is still one round trip; SDK `v0.18.0` added `ContentMetadata.Watch` for it. It was on the previous slice's *deliberately not added* list — availability elsewhere read as off-mission for a library that resolves its own sources — and that call was too narrow: "where can I actually watch this" is exactly what a detail screen is asked for a title Mosaic's own sources cannot resolve.

Three properties carry the weight, and the first is a standing hazard for the emit side. **An offer is not a source**: nothing in it becomes a `Part`, nothing is bound as a source binding, and nothing is playable through the Platform, so a client must render it informationally — pointed at the per-region link — and never as a play control. That is a promise the Platform cannot keep, and it is the one thing about this field that can be got badly wrong. **It is region-exact or absent**, the same rule the certification follows: TMDB reports over a hundred regions, and with none configured the module returns nothing rather than picking one; a region with a link and no offers is still an answer, because "none known here" is a different fact from "no data". **It carries its own attribution** — the data is JustWatch's and TMDB's terms require crediting them wherever it is shown, so the credit travels in the value rather than in a screen's memory.

*A note on the release: `v0.3.0` was pushed with a file that was not gofmt-clean — the gate caught it and a shell-chaining mistake let the push proceed anyway. `v0.3.1` is the same code formatted, and is the version required.*

**`module-tmdb` `v0.4.0` + SDK `v0.19.0` — the library can be grouped by streaming service, and deliberately is not.** Two halves landed. The module writes availability into the node's `Attributes` at import under a `tmdbWatch` key; `SearchContentQuery.AttributesContain` filters on it by **containment** — `@>` over the existing `nodes_attributes_gin` index, the same operator and index `FindByExternalID` has always used on the neighbouring document.

Containment rather than a typed filter is the design, not a shortcut. Attributes are module-owned and the Platform never interprets them ([ADR 0013](adr/0013-object-graph.md)), so "does this document contain this shape" is the only question that respects the ownership; a typed filter would make the Platform learn what a module put there. It is carried in the **contract suite** rather than the Postgres tests, because the SDK now promises any `StorageAdapter` answers it.

**No client path, on purpose, and it is [owed](unreachable-capability.md).** Nothing refreshes stored availability — the jobs runner, scheduler and system principal are all unbuilt — and it churns monthly. A group that says "on Netflix" for something that left in March is *actively wrong*, which is worse than an absent group: a user can see a missing feature and cannot see a lying one. So the storage and the query shipped and the surface did not, with a `checkedAt` stamp on every record so the eventual refresh knows what to re-fetch first. Adjacent and not blocked: a provider **catalog** — "what's on Netflix" as a discover-backed rail — asks TMDB live, needs none of this, and works today through a custom catalog.

**[ADR 0073](adr/0073-stream-resolution-is-decoupled-from-metadata-provenance.md) — stream resolution is decoupled from metadata provenance. Built.** This is the one that makes the core metadata modules usable as the way content is actually added, and it closes the seam ADR 0072 left open.

`ImportContent` resolved exactly one capability and handed it the whole materialisation. That was right when one module both described content and indexed it, and wrong the moment metadata became its own tier: TMDB and Cinemeta fill **no stream role**, so a title imported from either was a Work and a season/episode tree with **no Parts** — permanently unplayable — while `module-stremio-addons` sat registered alongside, able to resolve that exact title, and was never asked because the ref said `tmdb`. The effect was backwards: **a playable library cost you your metadata**, since the only way to get Parts was to import through Stremio and accept Cinemeta-grade descriptions however much richer a provider was configured.

Import is now two steps. The capability the ref names builds the tree; the Platform then asks **every** registered stream provider to resolve what plays, and attaches what comes back. A provider is addressed by the work's **shared external identity** plus **season and episode numbers read off the tree** — never a native id, because Stremio's episode id is `tt0903747:1:2` and a Platform that built that string would have a provider's dialect in its kernel ([ADR 0051](adr/0051-modules-as-anti-corruption-layers.md)). `StreamRequest` grew `Season`/`Episode` for it (SDK `v0.20.0`), and `module-stremio-addons` `v0.21.0` composes its own id from them.

Two properties were worth the design effort. **Best-effort:** every failure logs and continues, because an import that produced a tree has succeeded whether or not anything could be found to play. **Idempotent:** only items with no Parts are enriched, so a re-import attaches no second copy, an item that got nothing because no stream source was installed yet is filled in later, and a module that attached its own Parts keeps them. That rule needs the parts read handle, and a `Service` composed without one declines rather than enriching unsafely — found by a panic in the existing Stremio integration test rather than by reasoning.

**Verified end to end against real PostgreSQL with both real modules**: a TMDB ref imports a series whose episodes carry Stremio `RemoteLocation` Parts, and a second import changes nothing. Neither module can satisfy that alone.

**Left open, deliberately:** cross-provider Part dedup. One provider's set is taken per item and the rest are not merged, because nothing stops two sources returning the same release and there is no dedup to do it with — the same open seam as provider precedence, now reachable from a second direction. Also unaddressed: subtitles, which have no consumer yet, so `SubtitlesRequest` gained no coordinates.

**Composition-root wiring is landed.** Both modules are published (`module-tmdb` `v0.4.0`, `module-cinemeta` `v0.1.0`) and required at their tags with no `replace`, as every module is; `module-stremio-addons` goes to `v0.20.0` for the removal above. All four capabilities register identically — nothing in the composition root distinguishes a core module from the extension one, which is the ADR 0062 design showing through — and `RequireRoles` runs before the serve loop.

**Verified live, not only green.** The container gate passes including the PostgreSQL contract tests; the dev stack boots with all four capabilities registered at their published versions; the home screen's hero rotates through Cinemeta's *Popular Films*, *Popular Series* and *Featured Films* catalogs; a search for "the matrix 1999" returns its results with artwork; and the Stremio settings screen renders with no bundled-default card and Cinemeta offered as an ordinary installable addon.

**[ADR 0074](adr/0074-artwork-is-a-candidate-set.md) + [ADR 0075](adr/0075-the-artwork-provider-role.md) — artwork becomes a candidate set, and `module-fanart-tv` is the first extension module. Contracts, Platform and module are built and green; the artwork has not been seen on a screen.**

Three records had already named the same limit from three directions and none had closed it: `v1.Artwork` held **one string per slot**. [ADR 0030](adr/0030-artwork-proxy-and-cache.md) deferred candidate selection to "slice 2", [ADR 0071](adr/0071-content-artwork-is-stored-on-the-node.md) said a later slice "may fetch several candidates and let a user select among them", and the `module-tmdb` entry above recorded that its `images` response was fetched and discarded. What forced it was a source whose *entire value* is the set — fanart.tv returns forty posters per title, each with a language and a vote count, and keeping one throws away the reason to install it.

**The shape is what makes it additive.** The four flat slots stay and keep their type; their **meaning** narrows from "the artwork" to "the artwork that was *chosen*", with a new `Candidates` list holding what it was chosen from. Every existing reader — the continue-watching rail, the hero, the search card — keeps working untouched, and there is still exactly one answer to "what is this node's poster". The alternative, a list plus a selection index, would have made every surface resolve the selection itself. **And there is no migration**, exactly as migration `0019_node_artwork.sql` predicted in writing when it chose `jsonb` over three text columns: "leaves room for a candidate set and a user selection later without a second migration." A test pins that a pre-candidate document still decodes to the selection it always was.

**The eighth role, and the mistake it exists to prevent.** fanart.tv fills none of [ADR 0027](adr/0027-modules-as-typed-capability-providers.md)'s seven roles — no titles, no overviews, no search, no catalogs, so it can answer one question and none of the others. `RoleArtwork` is neither a source role nor a consumer role: it **enriches content that already exists**, the first of its kind and the shape a ratings or subtitles-database module would also take. The near miss was `RoleMetadata`, and taking it would have been a fault with a specific failure — [ADR 0035](adr/0035-metadata-as-required-capability.md) makes a registered `RoleMetadata` part of the composition-root check that Mosaic can identify content, so a module that cannot name a film would satisfy it, and the result would be a deployment that boots and finds nothing. No test would catch that, so `boundary_test.go` in the module asserts the manifest never grows the role.

**Invocation reused ADR 0073's pass wholesale.** `enrich_artwork.go` is `enrich_streams.go` with one difference: it does **not** stop at the first provider that answers. Artwork candidates union across providers rather than competing, because they are additive and attributable — which means that for artwork specifically the **provider-precedence seam dissolves** rather than staying open. That is not a general answer; streams still need one.

**What actually makes the images better**, before any picker exists, is two mappings and one rule. `lang: "00"` is fanart.tv's marker for **textless** and is mapped to an empty language, which the Platform's selection rule then *prefers* for backdrops and landscapes — a backdrop with a burned-in title sitting under a hero's own clearlogo is the collision this removes, and it is invisible to a rule that only counts votes. **HD variants outrank their SD twins**, because likes accumulate per image and an older SD logo frequently outscores the HD one that replaced it. And posters deliberately do *not* prefer textless: a poster's typography is part of the artwork. Seven tests cover the rule, since every way it can be wrong is silent — a worse image is still an image.

**It also closes [ADR 0034](adr/0034-rich-metadata-preview.md)'s oldest recorded gap.** Clearart and banners were written down as "not built — no data in the source", waiting on "a TMDB/Fanart-class provider". The slot vocabulary now carries clearart, banner, disc and characterart, and **season artwork needed nothing new** — a season is a `ContainerSeason` node and `Artwork` is stored per node, so the shape already fit and was simply never filled.

**One SDK addition was forced that nothing had needed before: a way to *update* a node.** `ContentService` was creates and attaches only, which is precisely the gap ADR 0071 recorded — "there is no command that updates a stored work's fields, so a re-import does not refresh its artwork today". The enrichment pass has a node and no way to put art on it, so `SetContentArtwork` landed (SDK `v0.21.0`); it is also what a user-picks-a-poster feature needs. It replaces rather than merges, and **a caller that re-resolves the slots will overwrite a user's choice** — nothing marks a choice as the user's because nothing sets one yet, and the selection feature will need that marker.

**`module-fanart-tv` is the first genuinely optional module in the build.** Every module before it is core under ADR 0062's coupling or guarantee clause, `module-remote-playback` included; nothing breaks without this one, artwork just stays as good as the metadata source made it. The extension tier's *mechanism* ([ADR 0064](adr/0064-extension-module-boundary.md)) is still unbuilt, so it composes statically like the others and the classification is a delivery decision waiting for a process boundary.

**Not done, and the gap is the visible half.** The contracts, the Platform pass and the module are written and their gates pass — SDK green, the platform's full container gate green including the PostgreSQL contract tests, all 18 module tests green and hermetic. **Publishing and wiring have since landed; the visible half has not.** SDK `v0.21.0` is tagged, `module-fanart-tv` is published at `v0.1.1` and required at that tag, and it is registered in the composition root (`cmd/mosaic-platform/main.go`) alongside the others. What remains undone is the part that was always the gap: **no dev-stack run has put a fanart clearlogo on a hero.** **The picker does not exist** — selection resolves by rule, a user overriding it has no client path and is [owed](unreachable-capability.md). Two limits are structural rather than pending: **fanart.tv keys television by TVDB id**, and Cinemeta binds only `imdb`, so a series imported through Cinemeta gets no artwork enrichment while the same series through TMDB does; and **the response-key → slot table has not been checked against fanart.tv's live documentation**, so coverage may under-deliver (an unrecognised key is ignored rather than guessed at, which is the right way round).

**Not built: everything about delivery.** The six records below still describe an unbuilt system.

- **[ADR 0062](adr/0062-two-module-tiers.md) — two tiers.** A **core module** for one of exactly two reasons — *coupling* (it must share the address space, as `postgres` must to join a `UnitOfWork`) or *guarantee* (Mosaic is not Mosaic without it, as playback and metadata are) — compiled into the binary, first-party, curated. Everything else is an **extension module** in its own process. The asymmetry is real and is stated rather than smoothed over: a third party can never be a core module, because both clauses exclude it. The tier is a delivery decision, not a contract decision — both tiers implement the same SDK interfaces and a module does not know which tier it is in.
- **[ADR 0063](adr/0063-platform-binary-built-by-ci.md) — Mosaic's CI builds and signs the Platform binary; the Supervisor selects rather than compiles.** This deletes [ADR 0006](adr/0006-supervisor-orchestrates-isolated-builds.md)'s per-install build pipeline, and the reason is what a self-hosted user would have experienced: a Go toolchain in the image, network to the module proxy before Mosaic serves anything, an arm64 NAS compiling the dependency closure behind a progress bar, and a third-party compile error as the first thing a new user sees. Every install would have produced a binary nobody had ever built. ADR 0062 is what makes the deletion possible — third-party modules leave the binary entirely, so the only thing being composed is a small curated set that is the same for everyone.
- **[ADR 0064](adr/0064-extension-module-boundary.md) — the boundary an extension module is crossed at, without changing what its author writes.** The SDK's Go interfaces stay the contract; the wire is an implementation of them on both sides, and nothing above the `CapabilityRegistry` knows it exists. The SDK splits rather than grows a dependency: `sdk` stays dependency-free, and a **nested module** at `sdk/host` carries the serving harness, so a module author's code is unchanged but for a `main.go` of roughly `host.Serve(stremio.New())`. All extension egress goes through a Platform-operated forward proxy: hosts attributed per module, content unseen, and — where the OS allows — no direct network at all, so the proxy is the only path rather than the easy one. **The process boundary is still not a general sandbox**, and the record says so.
- **[ADR 0065](adr/0065-module-distribution-and-trust.md) — signed artefacts and user-added repositories.** Trust moves from source to artefact: under ADR 0007 a module was source pinned by the Go checksum database, and it is now a binary someone downloads and executes. Mosaic's own repository is trusted by default, a user may add third-party repositories with explicit informed consent, and the provenance of what was installed stays visible afterwards — Jellyfin's plugin repositories are the worked example of the middle path between a curated closed set and an open one where the first malicious module is indistinguishable from the official ones.

- **[ADR 0079](adr/0079-the-platform-manages-extension-modules.md) — the Platform manages extension modules end to end; the Supervisor manages the binary.** This reverses a specific point of ADRs 0064 and 0065: they split an extension module's life across two processes, with the Supervisor installing and verifying it and the Platform running it. That re-introduced the coupling the tier split exists to break — a failure-prone third-party download becoming a durable-layer concern — split verify-then-spawn across a boundary, and grew the component that must not grow (ADR 0004). So the line is drawn by *tier*: core modules are the binary, which the Supervisor owns; extension modules are the dynamic ecosystem — discover, download, verify, spawn, health, restart, kill — which the Platform owns throughout. ADR 0065's signing and trust *mechanism* is unchanged; only the actor moves. The practical upshot is that the stremio cutover no longer waits on a Supervisor that does not exist — it waits on the Platform's own install-and-verify path, which is buildable now.

- **[ADR 0077](adr/0077-go-plugin-as-the-extension-harness.md) — `hashicorp/go-plugin` is the harness, and the wire stays protobuf.** ADR 0064 decided gRPC over a Unix socket but not whether Mosaic writes that harness or adopts one. A **msgpack** alternative (`tinylib/msgp`) was considered seriously, because it generates marshallers *from* the hand-written Go structs and would have collapsed the dual-source-of-truth cost ADR 0064 named as ongoing. It was rejected for two reasons: msgp generates methods **on the types**, so it would either force a dependency into `sdk` — the exact mechanism [ADR 0059](adr/0059-modules-observe-through-the-sdk.md) refused for OpenTelemetry — or reinstate the dual source of truth as hand-written DTOs in `sdk/host`; and **a codec is not a protocol**, so it would leave Mosaic owning framing, correlation, multiplexing, cancellation and backpressure across a bidirectional call graph. go-plugin runs gRPC for precisely that reason. `HandshakeConfig.ProtocolVersion` becomes the SDK major version, the non-executing manifest check precedes the handshake rather than replacing it, and MPL-2.0 enters the dependency graph in `sdk/host` only.
- **[ADR 0078](adr/0078-core-modules-keep-their-repositories.md) — core modules keep their repositories; CI carries the version bump.** Moving them into the Platform repository was proposed and rejected on three costs: MIT modules would need relicensing against the Platform's AGPL, the SDK-only boundary would stop being enforced by Go itself and degrade to a test, and ADR 0062's tier mobility would become a repository extraction. Instead a module tag dispatches to `platform`, which warms the Go module proxy, bumps the `require`, runs the gate and opens a pull request that auto-merges on green. The Platform edit is **automated rather than eliminated** — Go modules offer no way around it — and a core module tag produces a green `main` rather than a Platform release.

**What it supersedes, and what that costs.** [ADR 0007](adr/0007-static-go-module-composition.md) narrows to the core tier, [ADR 0006](adr/0006-supervisor-orchestrates-isolated-builds.md)'s build sequence moves to CI, and [ADR 0002](adr/0002-module-storage-and-delivery-model.md) §1 and §3 are amended. The existing modules are unaffected as *source* — both already import only the SDK — but `module-stremio-addons` becomes an extension module and therefore stops being a Go dependency of the Platform, which is a composition-root change rather than a module change.

**Built: the `sdui` → `contracts` rename, and the first piece of ADR 0078.** [ADR 0044](adr/0044-contracts-protobuf-workspace.md) decided the rename when it made these contracts protobuf and then read "Accepted (built)" for three days while it had not happened — `platform` still required `github.com/mosaic-media/sdui`. [ADR 0064](adr/0064-extension-module-boundary.md) forces it, because the module wire lives in that workspace and its bindings end up behind `sdk/host`, which a third party compiles against; that import path has to be final before publication. `contracts` `v0.12.0` is the renamed module, and the four modules importing it followed (`module-tmdb` `v0.6.0`, `module-fanart-tv` `v0.2.0`, `module-stremio-addons` `v0.22.0`, `module-aiostreams` `v0.2.0`) — both halves in one step, because a module on the old path and a Platform on the new one would carry the same SDUI types under two module paths and stop being assignable across the seam. The contract itself did not move: `mosaic.sdui.v1`, `contracts/sdui` and the npm name `@mosaic-media/sdui` are all unchanged. On the delivery side, `module-tmdb` now dispatches to `platform` on release and `platform` answers with an auto-merging pull request ([ADR 0078](adr/0078-core-modules-keep-their-repositories.md)).

**Built: the wire itself — `mosaic/module/v1`, the fourth package in the contracts workspace ([ADR 0064](adr/0064-extension-module-boundary.md), [ADR 0077](adr/0077-go-plugin-as-the-extension-harness.md)).** Three services in two directions: `CapabilityService` served by the module, `ContentService` and `TelemetryService` served by the Platform and called back over go-plugin's broker. The file's own header states that **it is not the contract** — the SDK's hand-written Go is, and this is an implementation of it on both sides of a process boundary. Two modelling calls are worth knowing: the open vocabularies ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) stay strings, because making `media_type` an enum would close a vocabulary the architecture leaves open and turn every new media type into a wire change; and error categories are carried explicitly rather than mapped onto gRPC status codes, because that mapping is lossy in both directions and would turn a `Conflict` into an `Unknown` at the boundary. Every enum was checked against the Go constants rather than inferred, and four first drafts were wrong. It generates **gRPC** stubs rather than Connect — go-plugin hands you a `*grpc.Server` to register on, which a Connect `http.Handler` cannot satisfy — which is why `buf.gen.module.yaml` is a second template. Published at `contracts` `v0.13.0`. **Nothing consumes it yet:** `sdk/host`, the Platform's extension host, the `Caller` handle table and a probe module are all unbuilt, so the wire is a schema that compiles and has never carried a call.

**Built: `sdk/host`, the harness — and the wire has now carried a call ([ADR 0064](adr/0064-extension-module-boundary.md), [ADR 0077](adr/0077-go-plugin-as-the-extension-harness.md)).** A nested Go module at `github.com/mosaic-media/sdk/host` (`host/v0.1.0`), carrying **both** sides of the boundary: a module's `main.go` becomes `host.Serve(mymodule.New())`, and the Platform gets a proxy implementing `v1.Capability` that its registry cannot distinguish from a compiled-in module. Nested so the parent `sdk`'s `go.mod` stays a module line and a Go version — the property [ADR 0059](adr/0059-modules-observe-through-the-sdk.md) established, now asserted by a test in `host` because it decays silently. Proven over a real gRPC connection with a live broker: manifest, import, a callback carrying the `Caller` handle, settings crossing uninterpreted, a module error arriving with no category and a Platform error arriving with one, a role round trip, and an unfilled role refused. **What that does not cover is the handshake and the Unix socket**, which need a spawned process — `test/extprobe` and the Platform's extension host are the next slices.

Three implementation findings worth the roadmap rather than only the code. The callback stream uses a **fixed broker id** rather than one negotiated per invocation, which would have put a broker-id field on every request: it is unnecessary because the connection is not what scopes authority — the `Caller` handle is, and a retained connection is useless without a live handle. The Platform-side proxy **implements every role interface unconditionally**, so `cap.(v1.SearchProvider)` always succeeds against it and role discovery must read the manifest instead of type-asserting. And **error categories are injected, not extracted**: `sdk/host` compiles against the SDK, where the Platform's categories are not published, so it cannot read one off an error — the Platform passes a `CategoryFunc` in. That last one is faithful rather than a gap, since a module cannot classify an error in process either.

**Built: the boundary works end to end — [ADR 0064](adr/0064-extension-module-boundary.md)'s build-order step 1 is complete.** `internal/adapters/extension` launches a module process and the Platform holds it as an ordinary `v1.Capability`; `test/extprobe` is the module. Nine tests spawn a real child over a real Unix socket, which is what the in-package `sdk/host` tests could not reach: handshake, manifest, `Import` calling back into `ContentService` within the invocation and carrying the `Caller` handle, a role round trip, a role the module does not serve refused, and a manifest mismatch refusing the connection. `Launch` runs both of ADR 0064's checks in that record's own order — go-plugin's handshake refuses a binary built against a different SDK major, then the running binary's manifest is checked against what the manifest file claimed. The package sits under `internal/adapters/` rather than `internal/modules/`: it implements no Platform contract, it adapts an external process into a published one. It is the first thing that is host *of* a tier rather than a member of one, and the package-tier model has no word for that yet.

**The boundary found a real defect in the capability registry, and it would have been a silent one.** Eleven of the registry's thirteen role-resolution sites discovered roles by **type assertion** — `c.(v1.StreamProvider)` — with no manifest check. That was correct by accident for as long as every capability was a local struct that declared what it implemented. It stops being correct against a proxy, which implements *every* provider interface unconditionally because the registry must not be able to tell it from a local struct and Go type assertions cannot be conditional at runtime. A metadata-only extension module would therefore have been enumerated as a stream, artwork, catalog and playback provider, and — worse — **`RequireRoles` would have been satisfied by a module declaring no roles at all**, producing exactly the deployment [ADR 0035](adr/0035-metadata-as-required-capability.md) exists to prevent: one that boots claiming it can identify content and cannot. Every resolution now requires the role to be both declared and implemented. The registry's own `declares()` comment already said a role not declared is not resolvable by role; only two of the thirteen sites did that.

**Built: `Caller` is an invocation-scoped handle, so Stage 1 of the boundary is complete.** A module no longer receives a session reference at all — it receives a handle minted for one invocation and revoked when that invocation returns, and the `ContentService` it calls back into exchanges the handle for the `Caller` it was minted for. [ADR 0064](adr/0064-extension-module-boundary.md) rejected a short-TTL token and the reason survives contact with the code: **a TTL is a window in which a retained value still works, and retention is the entire risk serializing `Caller` introduces.** A handle that stops resolving the instant the invocation returns has no window, nothing to tune and no clock-skew story. Every method mints, not only `Import` — a read role never calls back, but handing it a real session reference "because it cannot use it" would make the guarantee depend on the module's behaviour rather than the Platform's. An unknown handle is `PermissionDenied` rather than `NotFound`, because the module is attempting to act as somebody. Resolving is **not** authorisation: it says which session, and the application service then authenticates and authorises exactly as it would for any other request ([ADR 0017](adr/0017-how-a-capability-acts.md)). Proven over a spawned process, including that a *failed* invocation still revokes — a revoke on the success path only would leave a usable handle after exactly the calls most worth abusing.

**Built and measured: the callback cost, which answers the question [ADR 0064](adr/0064-extension-module-boundary.md) deliberately left open.** That record said chattiness "should be measured against a real Stremio import before the protocol is fixed, and allowed to send the service shape back for coarser, batched verbs". Measured: **a callback costs ~700µs**, so a 24-episode season's 25 calls spend ~17.5ms crossing the boundary. **The batched verbs are not needed**, and `mosaic/module/v1`'s callback shape can be treated as settled rather than provisional. The figure is a floor rather than a best case — it was taken in a container on macOS over a virtiofs mount, slower than the Linux deployment [ADR 0004](adr/0004-supervisor-as-host-manager.md) targets. What is measured is deliberately *cost per call* and not the cost of a real import: a tree import costs (calls per import) × (cost per call), those two have different owners, and a number that fused them would describe neither. A 5ms ceiling stays in the test as a regression guard, on the stated understanding that breaching it answers the batched-verb question the other way rather than licensing a looser ceiling.

**Built: `module-stremio-addons` can run as its own process** (`v0.23.0`) — a `cmd/` whose whole body is `host.Serve(stremio.New(nil))`. It is additive: nothing imports it, `stremio.New` is still what the composition root calls, and the Platform still composes the module statically. The module's boundary test now **walks** the module rather than reading its root directory, which it had to: a check that looked only at the root would have passed while never reading the one new file that imports the harness. Two things it immediately caught are worth keeping — a module importing its own package from `cmd/`, which is the one legitimate self-import, and `sdk/host` resolving under the SDK prefix, which is correct because the harness is published beside the contract precisely so a module needs no dependency the SDK did not already sanction.

**Built: the real Stremio module runs out of process — [ADR 0064](adr/0064-extension-module-boundary.md)'s step 2, as a proof rather than a cutover.** The actual `module-stremio-addons` binary, with its four source roles and a real import that calls back into `ContentService` several times, is spawned over the boundary against a fake Stremio addon and works. `extprobe` (step 1) established the wire with a trivial module; this shows the *real* one's compiled binary serves correctly under `host.Serve` — the integration risk a trivial probe cannot surface, since a global, an init or a stray stdout write would corrupt the handshake and the module's four-role code was never written to expect a process it did not choose.

**The distinction between the proof and the production cutover is the load-bearing part.** The Platform still composes this module statically, and that is not a limitation of the proof — it is the correct default, because the cutover is blocked three ways and each is a real hazard rather than unfinished plumbing. Dropping the static `require` and spawning the installed binary would (1) route the module's user-supplied addon URLs around `netguard`'s dial guard, since an `*http.Client` cannot cross the process boundary and the egress forward proxy that replaces it is unbuilt — an SSRF regression for the one module whose URLs come from user input; (2) make a module crash a Platform death rather than a degraded capability, until process lifecycle exists; and (3) presuppose an install-and-verify path that hands the Platform a trusted binary. The first two are now built (the egress proxy and process lifecycle); the third is the Platform's own signed-artefact install path ([ADR 0079](adr/0079-the-platform-manages-extension-modules.md)) rather than a Supervisor that does not exist. So the boundary is *proven* against the hardest real module while the switch that would ship it waits for that install path. The cutover has its own slice; it is not this one.

**Built: module process lifecycle — [ADR 0064](adr/0064-extension-module-boundary.md)'s step 3, and its open crash-loop policy is now closed.** `Supervised` launches a module, actively probes whether it is still *answering* (a bounded gRPC ping each interval, not a bare exit check, so a hung-but-alive process is caught too), restarts it with exponential backoff when it is not, and disables one that will not stay up. Disabling is **never a Platform exit** — the capability reads `Unavailable` until an operator acts, which is exactly how [ADR 0035](adr/0035-metadata-as-required-capability.md) and [ADR 0036](adr/0036-capability-gated-affordances.md) expect an absent capability to degrade. The record's three named questions are answered with conservative defaults and the reasoning beside each: backoff from 1s to a 30s ceiling; a launch that stays up past a healthy threshold forgives its past crashes, so a module dying once a week is never mistaken for one dying once a second; disable after five crashes-before-healthy; and the admin told through telemetry — down-and-back is `Warn`, disabled is `Error`, both carrying the module id. Resource limits (memory, CPU per module) stay deferred, as that record leaves them: this observes health, it does not cap consumption.

The load-bearing design point is the **stable proxy**: `Supervised` is what the registry holds, once, and a restart swaps the process and the gRPC connection underneath it without the registry knowing. `Manifest` is answered from a copy cached at first launch, so a module's roles do not flicker while it is down — a role-resolution pass landing in the restart window would otherwise drop them. Without this, a restart would orphan the registry's reference and every call would hit a dead connection.

**Built: the role-class table and core-module selection — most of [ADR 0063](adr/0063-platform-binary-built-by-ci.md)'s Platform half, minus the Generation activation that needs the Supervisor.** The role-class table is now the single in-code copy of that record's table (`app.RoleClasses`): storage (exactly one, destructive), metadata/search (one or more, required), playback (one or more, not required). It lives in Platform code and not in module manifests, as ADR 0063 insists — arity is a property of the class, and a manifest that could assert "I am the only one of me" would be one that could lie. The composition root's hand-kept `RequireRoles(RoleMetadata, RoleSearch)` is replaced by `RequireComposedRoleClasses`, which reads the required roles from the table, so adding a required class is a table edit and a boot failure names the short class rather than only the short role. Two departures are recorded rather than smoothed: **storage** is in the table for completeness but has no capability role — its exactly-one arity is enforced structurally by the `ContractSet` holding a single `StorageAdapter`, which is stronger than a runtime check because you cannot wire two; and **playback** is marked not-required, refining ADR 0063's "one or more", because that arity is about the maximum (local and remote are complementary) while a deployment with no consumer is discovery-only ([ADR 0036](adr/0036-capability-gated-affordances.md)), a degraded state rather than a failed boot.

**Selection decides construction, and an unselected module never boots.** `registerCapabilities` is now a descriptor list the selection filters; a module the selection does not name has its `New` never called, so nothing it holds is opened and the registry never sees it — "code in the binary that never boots," exactly as ADR 0063 puts it. (One honesty about Go recorded in the code: a compiled-in module's *package* init runs regardless, unavoidable for static linking; what selection skips is construction and registration, where every resource and effect lives.) The default is every module, so a fresh install is unchanged and still gets the zero-configuration metadata floor ([ADR 0072](adr/0072-the-guaranteed-metadata-provider-needs-no-credential.md)). Selection is Generation-class configuration; the Generation-activation half is Supervisor-shaped and unbuilt, so it is read from `MOSAIC_MODULES` — the same environment bridge the DSN and log level take. Demonstrated against the real binary: the default registers all six; `MOSAIC_MODULES=cinemeta` constructs only cinemeta and logs the other five as not selected; a typo names itself as a config error; and `MOSAIC_MODULES=""` refuses to serve because nothing fills metadata/search. **Mutability (destructive vs new-Generation) is carried as data** for onboarding to read — ADR 0063 requires the storage choice be presented as one to get right — but nothing consumes it yet, and no onboarding surface reaches it.

**Built: the egress forward proxy and its deny list — [ADR 0064](adr/0064-extension-module-boundary.md)'s layer 2.** Every outbound call an out-of-process module makes goes through a per-module CONNECT proxy the Platform runs, which applies the same `netguard` deny list the in-process client's dialer had — loopback, RFC1918, link-local (including `169.254.169.254`) refused by default — and attributes every host to the module. For HTTPS it is a tunnel: the Platform sees the host, never the content. An operator override (off by default) re-opens the LAN for the genuine local-service case.

**A test forced out the finding that makes this more than setting `HTTP_PROXY`, and it is the target that matters most.** Go's `ProxyFromEnvironment` hardcodes a bypass for `localhost` and loopback, so a module using an ordinary client would reach `127.0.0.1` — the Platform's own PostgreSQL — *around* the proxy and its deny list. An env-var-only proxy leaves the highest-value SSRF target open. So the mechanism is two parts: the proxy, and `sdk/host` (`host/v0.2.0`) forcing the module's default transport through it with no loopback exception. Proven end to end with a real spawned module: `test/extprobe` fetches a loopback URL with an ordinary `http.Get`, and it is refused by default and allowed only under the override — the deny covering exactly the case a naive proxy misses. (The probe's first cut counted a proxy `403` as success, since a denied plain-HTTP fetch is a response rather than a Go error; the test caught it.) **Convention now, enforcement later:** this makes the proxy the *easy* path and closes the accidental and user-URL SSRF path for a cooperating module — which every first-party module is — but a module that ignored the proxy and dialled directly still could, because it still has a network. Layer 3 (OS-level network denial) is what makes the proxy the only path, and it is deployment-dependent. Rate-limiting, which ADR 0064 says the proxy "can" do, is deferred rather than invented without a policy.

**Built: the layer-3 egress-containment posture is reported, honestly (slice 5.2, [ADR 0080](adr/0080-deployment-topologies.md)).** Layer 3 — denying a module process direct egress so the proxy is the *only* path — is an OS mechanism (a network namespace, a dedicated-uid firewall owner rule, seccomp on `connect`), and a non-root Platform cannot apply it to itself; some platforms have no low-cost form of it at all. So the buildable thing is the one ADR 0080 asks for: rather than claim the guarantee uniformly, the Platform **determines and reports which posture the deployment is in**. `DetermineEgressContainment(goos, declared)` resolves it purely — enforced only on a platform that has a mechanism (Linux and other unixes) *and* where the deployment attests it via `MOSAIC_MODULE_EGRESS=enforced`; macOS and Windows are always attributed-only, and a claim there is downgraded rather than believed; the default, nothing declared, is never enforced, so the claim is never made where a deployment did not attest it. It is logged at boot ("enforced" vs "attributed but not enforced", with the reason) and carried in the diagnostic support bundle. What this does **not** do is implement the OS enforcement — that stays deployment-provided and privileged (the container's netns, the systemd unit's firewall rule); what it prevents is the Platform overpromising a containment it may not have, which is the exact failure [ADR 0080](adr/0080-deployment-topologies.md) exists to stop.

**Built: extension modules are verified before they run — [ADR 0065](adr/0065-module-distribution-and-trust.md)'s gate, Platform-side per [ADR 0079](adr/0079-the-platform-manages-extension-modules.md).** Before the Platform spawns a downloaded module and hands it a `Caller`, it runs five checks that refuse on the first failure: the manifest is signed by a trusted publisher key (ed25519, standard library); it parses and names a module; it was built against this Platform's SDK major; it ships a binary for this OS/arch; and the binary on disk hashes to the digest the now-authentic manifest declared, compared in constant time. The manifest is *read*, never executed — the alternative is running an unverified binary to ask it what it is, which is the thing the whole scheme exists to prevent. On success a module is pinned three times over: the **signature** says the publisher stands behind the manifest, the **digest** says these bytes are the ones it stands behind, and the **handshake** (already built) says the running binary agrees with what was signed. The verifying publisher is returned as provenance, which is where ADR 0065's after-install provenance begins. `tools/modulesign` is the publisher side — genkey, digest, sign — and the digest it prints comes from the same exported function the Platform hashes with, so the one format that must match on both sides has a single definition rather than two that drift into a silent "signature does not verify". A round-trip test signs with the tool and verifies with the Platform. **What is not built:** the download-from-a-repository-index that produces the manifest+signature+binary trio on disk (the repository/trust surface, [ADR 0065](adr/0065-module-distribution-and-trust.md) mechanism), and real key custody — the official signing key is infrastructure Mosaic must operate, not a constant to invent. So the *gate* exists; what feeds it does not yet.

**Built: the Platform installs extension modules from a signed repository — [ADR 0065](adr/0065-module-distribution-and-trust.md)'s repository model, Platform-side per [ADR 0079](adr/0079-the-platform-manages-extension-modules.md).** A repository is a signed index over HTTPS and the key that signs it. **Trust is per repository**, not per module and not global: adding a repository *is* the trust decision — the moment a user takes on the risk that its modules run with the Platform's authority — so consent attaches to the registry, not to each install. Mosaic's own repository is trusted by default; a third-party one is added with explicit consent. A design choice recorded rather than assumed: **one key vouches for everything a repository distributes** — it signs the index and is what a module from it is verified against — which collapses "the repository is trusted" and "this module is authentic" onto the one decision the user actually made, rather than a separate publisher key per module. The install flow refuses at the first failing check — the index's own signature (this catalogue came unaltered, authenticating every inline manifest at once), the module being present, a binary for this platform, and the downloaded binary hashing to the signed digest, which catches a tampered mirror. A binary that fails is removed rather than left on disk. **Provenance stays with the installed module** — which repository, verified against whose key — because a consent dialog clicked months ago is not context when an admin is looking at a broken import. The one network reach is behind a `Fetcher` interface whose production implementation routes through `netguard`'s dial guard, since a repository URL resolving into the host's own network is exactly that guard's hole. `tools/modulesign` gains `sign-index` and `build-index` — the latter assembles the index from module manifests, computing each binary's URL from a template, so a repository is *producible* and not only consumable. **A repository is hosting-agnostic by construction — signed static files over HTTPS — so the official one is intended to be hosted on GitHub, the way other open-source plugin ecosystems are: the index on GitHub Pages, the cross-compiled binaries as Release assets, the index's binary URLs pointed at the release download URLs.** The load-bearing point is that GitHub is an *untrusted* host here — the signature and per-binary digests protect a download, not the host — so a tampered mirror is refused whoever serves it, which is the Jellyfin model [ADR 0065](adr/0065-module-distribution-and-trust.md) cited and needs no new decision. A round-trip test builds an index, signs it, serves it and installs from it with the binary URL in GitHub-releases shape. **Not built, named rather than pretended:** the admin surface that collects consent to add a *third-party* repository (a client path). The composition-root wiring to a live official repository, the official key, and the `mosaic-media` registry repo have since landed — see below — but a running Platform still installs no real module, because none is catalogued yet.

**Built: the CI release matrix — the artefact every deployment topology installs ([ADR 0063](adr/0063-platform-binary-built-by-ci.md), [ADR 0080](adr/0080-deployment-topologies.md)).** A version tag on `platform` runs the full container gate (so the release is pre-proven against real PostgreSQL, not only the push gate), then cross-compiles the one binary to `linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64` and `windows/amd64` — each with a SHA-256 — publishes a GitHub release carrying them plus a combined `SHA256SUMS`, and builds a multi-arch container image. `CGO_ENABLED=0`, so every target cross-compiles from one runner rather than a per-platform C toolchain — the loop-not-a-build-farm property the arm64-NAS install depends on. Verified before writing the workflow: all five targets build clean (~15MB pure-Go binaries) and the optional TMDB token's `-X` injection resolves. **The image is a packaging artefact, not a second build** — it copies the same binaries the matrix produced, so a container and a native deployment run identical bytes — and it bundles `ffmpeg` because the Platform shells out to `ffprobe` at runtime ([ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md)); a container without it would relay undecodable audio silently, and a bare-metal user installs it themselves. It runs as a non-root user. Verified locally end to end: the image builds from a cross-compiled binary, carries `ffprobe`, runs unprivileged, and the real binary boots. **Deferred and named:** signing the binary and checksums (waits on key custody), and the Supervisor that downloads, verifies and activates a release — a user runs the binary directly meanwhile.

**Two things the rename exposed, both gaps rather than breakages.** A textual rewrite of the module path **corrupted the generated `.pb.go` files** — a `.pb.go` carries its `FileDescriptorProto` as a length-prefixed byte string, so substituting `sdui` for `contracts` inside it grew a string by five bytes without updating the prefix, and the package panicked at init with `slice bounds out of range [-5:]`. The fix is that generated protobuf is regenerated, never rewritten. What made it worth recording is why nothing caught it locally: **`contracts`' drift guard did not cover the protobuf bindings.** `scripts/check-generated.sh` regenerated and diffed the quicktype and genui output, but `gen/**/*.pb.go`, the Connect and gRPC stubs, and `gen-ts` were generated by hand and committed with no equivalent check, so a corrupt or stale binding passed every gate in that repository and failed at init in the consuming one. **Now closed (`contracts` `v0.15.0`):** the guard regenerates the protobuf bindings too — buf and the pinned Go plugins in the image, `protoc-gen-es` from `npm ci` — and diffs `gen/` and `gen-ts/`. On its first run it caught two real stale bindings, which is the whole point of it: `gen-ts` still encoded the *old* `sdui` module path in its embedded `FileDescriptorProto` (the rename regenerated the Go but never the TypeScript), and a `Field` proto comment was wrong twice over (it claimed a type tag that does not exist and a flattening the SDK does not do). Both were the silent kind — no build failed, no test went red. Separately, **only `module-tmdb` had any CI at all** — the other five module repositories had no `verify.yml` and no `release.yml`, so ADR 0078's dispatch covered one module rather than the set. **Now closed:** all five (`module-cinemeta`, `module-remote-playback`, `module-fanart-tv`, `module-stremio-addons`, `module-aiostreams`) carry the same two workflows — a `verify.yml` mirroring their container gate on push/PR, and a `release.yml` that validates the tag, proves it resolvable through the public proxy, and dispatches ADR 0078's Platform bump. The dispatch is inert until each repo's `PLATFORM_DISPATCH_TOKEN` secret is set, which is an operational step, not a code one.

**Built: the producer chain is complete and the Platform trusts the official repository at boot — the last of [ADR 0065](adr/0065-module-distribution-and-trust.md)'s repository model that was named-not-built, Platform-side per [ADR 0079](adr/0079-the-platform-manages-extension-modules.md).** Three pieces close the gap between the install machinery above and a real repository. (1) The **`mosaic-media/registry`** repository exists: the catalogue is `registry.yaml`, a CI step (`assemble.sh`) downloads each catalogued module's own `manifest.json` from its release and runs `build-index` over them, and `publish.yml` signs the index with the `REGISTRY_SIGNING_KEY` secret and deploys it to GitHub Pages. The registry aggregates and signs; it computes no URLs, because a module's binary URLs live in its own manifest — the module knows its repository name, which need not match its module id, so the URL is its to declare rather than a registry template keyed on id. The workflow refuses to catalogue a module that publishes no binary rather than emit a broken index, and the empty catalogue is a clean no-op until one does. (2) The Platform **embeds the official ed25519 public key** and its composition root trusts `mosaic-official` (`https://mosaic-media.github.io/registry`) at boot — the live-repository wiring the entry above named as unbuilt. (3) A module's release **produces the out-of-process artefacts**: `module-stremio-addons`' `release.yml` gains a `binaries` job that cross-compiles the five targets (CGO off, so one runner covers all), asks the binary its identity through `host`'s `--mosaic-manifest`, and joins that with the SDK major and the binaries' digests into a `manifest.json` — the artefact the registry catalogues — uploading both to the GitHub release. The first real release is now cut and live: `module-stremio-addons@v0.24.0` published the five cross-compiled binaries and a `manifest.json` (the tag overrode the build's `(devel)` version and the Windows entry is correctly `.exe`-suffixed, via an `{ext}` placeholder in the URL template — the asset carries the suffix Go's build gives the binary, and a matrix that includes `windows/amd64` would otherwise catalogue a URL that 404s). The registry catalogues it (`registry.yaml` names the *repository*, since a repo name does not follow from a module id — `module-stremio-addons` publishes a module whose id is `stremio`), and the publish workflow's non-empty path ran for the first time: it downloaded that release's manifest, built an index over it, signed it with the `REGISTRY_SIGNING_KEY`, and deployed it to Pages. The published index at `https://mosaic-media.github.io/registry/index.json` verifies against the ed25519 key the Platform embeds, and each catalogued binary's digest matches the bytes the release serves — the whole producer→registry chain is proven **live**, not only in a round-trip test. **What remains is the Platform-side switch (slice 2.5), and its shape is now decided — [ADR 0081](adr/0081-extension-installation-is-user-initiated-and-persistent.md).** The open question the producer chain raised — what triggers the install, and with what boot semantics — is answered: an extension is **installed by a user from a settings surface, nothing is installed by default, and the installation is durable state the Platform re-adopts across restarts** until the user uninstalls it. Not an environment flag (that stays the bridge for *core* selection) and not a boot-time default. That makes the switch a small feature rather than a one-line flip, and it sequences into its own slices: (2.5b) a durable installed-extensions store the Platform reads at boot and re-adopts — re-verify then spawn — from; (2.5c) install and uninstall as runtime actions, which makes the capability registry a structure that can add and remove *while serving* rather than only at composition; (2.5d) the SDUI settings surface that lists the repository's offer and the installed set and carries the install/uninstall actions on the `Invoke` lane, the shape `configureModule` already has. Dropping the static `require` on `module-stremio-addons` lands with 2.5b/c: Stremio becomes an available-not-installed extension, and a fresh Platform stays functional because the guaranteed metadata floor is core (Cinemeta), not this module.

**Built: the durable installed-extensions set and boot re-adoption — [ADR 0081](adr/0081-extension-installation-is-user-initiated-and-persistent.md)'s first half (slice 2.5b).** The Platform now holds a durable record of which extension modules a user installed — a Platform-owned store keyed on module id, carrying the source repository, the pinned version and the verifying-key provenance — and re-adopts that set at boot. Default-empty: a fresh Platform composes only its core modules and reaches the adoption loop with nothing to do. Re-adoption prefers the on-disk cache: `Installer.Adopt` re-verifies the cached binary against the manifest that was authenticated at install — confirming the exact *pinned* bytes without re-fetching an index that may by then list a newer version — and re-installs from the repository only when the cache is gone (install now writes the authenticated manifest beside the binary for this). A restart therefore reconstructs exactly the set the user last installed, offline and without re-consent. The adopted module spawns under the existing supervisor and registers as a `Supervised` capability the registry cannot distinguish from a compiled-in one; a module that fails to adopt is a **degraded capability**, logged and skipped, never a boot failure, because extensions fill no required role class. Proven against real PostgreSQL (the store's round trip — empty default, install, reinstall-replaces, idempotent uninstall) and a real spawned subprocess (adoption from cache touches no network; a wiped cache reinstalls). **Not yet:** the runtime install/uninstall that writes this set (slice 2.5c, which makes the capability registry mutable while serving) and the settings surface that drives it (2.5d). The static `require` on `module-stremio-addons` still stands; it goes when an install can put Stremio in the set.

**Built: runtime install/uninstall and a registry mutable while serving — [ADR 0081](adr/0081-extension-installation-is-user-initiated-and-persistent.md)'s second half (slice 2.5c).** An extension can now be installed and uninstalled *while the Platform serves*, not only composed at boot. Two pieces make that safe. The **capability registry became concurrency-safe**: it was a bare map, correct only while every registration happened before the first request, and it now takes a `RWMutex` (a resolution during serving takes the read lock, an install or uninstall the write lock) and gained an `Unregister` that makes a module unresolvable across every seam — `Lookup`, the role resolvers, the enumerations, and `RequireRoles` — so nothing routes to a process being torn down. Proven under `-race` with concurrent register/unregister/read. And a composition-root **`ExtensionManager`** ties the three parts that do not know each other — the installer (download/verify), the durable store (what a user installed), the registry (what the Platform resolves) — into one runtime lifecycle: adopt the set at boot, install (verify and spawn first, then persist, rolling the process back if the persist fails, so a durable record only ever names a module that ran), uninstall (unregister, stop, drop the record; idempotent), and stop every process on shutdown. Proven against a genuinely spawned module: an install makes it resolve through the registry indistinguishably from a compiled-in module, an uninstall makes it unresolvable and drops the record, and a fresh Manager re-adopts the set across a simulated restart from the on-disk cache. The boot loop that slice 2.5b put in the composition root is now the Manager's `AdoptInstalled`. **What remains (2.5d):** the authorized action entry points a user's install/uninstall click lands on, and the SDUI settings surface that lists the repository's offer and the installed set — the Manager's `Install`/`Uninstall` are the mechanism those actions call. The static `require` on `module-stremio-addons` goes with that slice, when a user can put Stremio in the set.

**Built: authorized install/uninstall actions reachable through the client — [ADR 0081](adr/0081-extension-installation-is-user-initiated-and-persistent.md)'s action layer (slice 2.5d, part 1).** A user can now install and uninstall extensions through the Platform's own client transport, authorized like every other administrator action. An `ExtensionManager` **port** in the app package fronts the composition-root manager, so the Service depends only on an interface, not the extension adapter; `Service.InstallExtension`/`UninstallExtension`/`ListInstalledExtensions` follow the command boundary — validate, authenticate, authorize the new `extension.manage` action (added to the administrator preset), then delegate. Install and uninstall deliberately do **not** open a `UnitOfWork`: they are side-effectful process operations whose persistence and rollback live with the manager that also spawns the process, so they are audited through its telemetry rather than the outbox. The reflection-enforced boundary conformance suite gains all three methods, and the session transport routes `installExtension`/`uninstallExtension` on the `Invoke` lane, the shape `configureModule` has. The composition root builds the manager before the Service, injects it, and closes the construction cycle — the Service drives the manager, the manager calls back into the Service — with a `SetContent` after both exist. **What remains (2.5d, part 2):** the SDUI settings *screen* that lists the repository's offer and the installed set so the actions have a surface to be reached from, and dropping Stremio from the default composition so it becomes install-on-demand. A finding recorded for that step: the `go.mod` require cannot be fully removed while five integration tests import Stremio as a *fixture*, so the change is "drop from the composed set" plus a later migration of those tests, not a bare require deletion — and it should land with the screen, so a dev stack that loses the compiled-in Stremio can reinstall it.

**Built: the platform test suite is decoupled from extension modules, and the whole pipeline is proven live.** A platform test that imports an extension module is the coupling the tier split exists to prevent (ADR 0079, ADR 0081: extension modules run at runtime, they are not platform dependencies), and five tests did it. The four that tested *Platform* behaviour — the cross-provider enrichment bridge, playback resolution, the import invocation path, stream-source precedence — now use reusable fakes (`fakeImportModule`, `fakeStreamModule`) instead of the real Stremio and AIOStreams modules; core modules (TMDB, remote playback) stay real. The fakes are made to answer *only* for the identity, season and episode the Platform passes, so a bridge or precedence assertion cannot decay into a tautology. The fifth — the "a real module runs out of process" proof, which built the real binary from an import — is replaced by an integration test behind a `//go:build integration` tag that **installs Stremio from the official registry at runtime**, verifies its signature and digest, spawns it, and confirms it serves — importing zero extension modules. A scheduled CI job runs it; the default gate is hermetic and excludes it. **That live test earned its keep on the first run:** it caught a correctly-published module being refused at the handshake only because its self-reported build-graph version (`0.24.0+dirty`) is spelled differently from its release tag (`v0.24.0`). The manifest's version is the catalogue tag and the digest already binds the exact bytes, so the handshake no longer compares the version string — identity is the id and roles, authenticity is the digest. The whole producer→registry→install→spawn pipeline is now green end to end against the live registry. **The `go.mod` requires on `module-stremio-addons` and `module-aiostreams` still stand, because `main.go` composes them in;** they come out with the composition drop below.

**Built: the two-tier module overhaul is complete end to end (slice 2.5d part 2).** Two things close it. First, **extensions are no longer composed into the binary**: Stremio, AIOStreams and fanart.tv came out of `registerCapabilities` and out of `go.mod` entirely (`go mod tidy` removed the requires cleanly), so a fresh Platform carries only the three core modules and everything else is installed at runtime. `RequireComposedRoleClasses` still passes — only metadata/search is required and Cinemeta is the zero-config floor. The platform module now has **zero dependency on any extension module** anywhere: not `go.mod`, not the composition root, not the test suite. Second, the **SDUI browse-and-install screen**: a server-emitted settings screen lists what a user has installed (each with an Uninstall control) and what the trusted repository offers (each not-yet-installed with an Install control), emitting the `installExtension`/`uninstallExtension` Invoke actions from part 1. It is its own route because listing the available set reaches the repository over the network — done on open, not on every settings render — and it reads that set resiliently: an unreachable repository still shows the installed set and its controls rather than failing. `Installer.Catalogue` fetches and verifies the signed index for it; `Service.ListAvailableExtensions` is the authorized query. **The whole arc now runs for a user:** an extension is produced (cross-compiled, signed), published to a registry (a signed index on Pages the Platform trusts), browsed on a settings screen, installed (verified signature + digest, spawned, recorded), adopted across restarts from the on-disk cache, and uninstalled — none of it a platform dependency, all of it proven live.

**Corrected: all three extension modules are converted, not just Stremio.** The composition drop above got ahead of itself — it removed Stremio, AIOStreams and fanart.tv from the binary, but only Stremio had actually been made installable (a `cmd/main.go`, a `binaries` release job, a real release, a registry entry). AIOStreams and fanart.tv were left *stranded*: out of the binary and not yet installable either, which is the "working capability with no path" this roadmap's discipline exists to catch. Both now have the same machinery Stremio got — a one-line `host.Serve` `main.go`, `sdk/host v0.3.0`, and the release `binaries` job — and both cut their first out-of-process release (`v0.3.0`): AIOStreams (`stream`, `subtitles`, `settings_ui`) and fanart.tv (`artwork`, `settings_ui`). The official registry now catalogues all three, its signed index verifies against the Platform's embedded key, and the live runtime-install integration test installs and spawns **every** catalogued module — so the newly-converted binaries are proven to serve out of process, not just assumed to. The stranding is closed.

**Almost none of this waits on the Supervisor, which is a correction to what this thread recorded.** The runtime boundary — harness, extension host, `Caller` handle, lifecycle, egress proxy — is built in the Platform, against a hand-placed binary, in the order [ADR 0064](adr/0064-extension-module-boundary.md) set out. And under [ADR 0079](adr/0079-the-platform-manages-extension-modules.md) the *install and verification* of extension modules is the Platform's too, not the Supervisor's — so the signed-artefact download, the signature and digest check, the repository index and the user-consent surface are all buildable now rather than blocked on a component that does not exist. The Supervisor's only module responsibility is the **binary** it downloads, verifies and activates (ADR 0063), which is the core modules. What genuinely waits on the Supervisor has shrunk to that one artefact.

### Done: modules as typed providers, and two content planes — **done end to end (slices 1–4); the Shell renders live Platform screens**

Building the Stremio module against real addons showed the module contract is too narrow. A `Capability` today is a single write verb, `Import(query)` ([ADR 0019](adr/0019-module-capability-and-invocation.md)), which forces the caller to already hold a raw provider id (`movie/tt1254207`). But a Stremio addon is a *set of resources* — `meta`, `catalog`, `catalog/…/search`, `stream` — and the product needs all of them: a user should search Mosaic and have it use the module with **no raw ids**, an admin should browse the module's **collections** and pick which to publish as library, metadata should **enrich** existing nodes, and streams should attach to published items. And browsing a source must **not** flood the object graph — a source exposes far more than anyone curated.

Two decisions answer this, written ahead of the build (unusually) and now **Accepted** as slices 1–3 have landed:

- **[ADR 0027](adr/0027-modules-as-typed-capability-providers.md) — modules as typed capability providers.** The module contract becomes declared **provider roles**, one per resource kind: `MetadataProvider`, `SearchProvider`, `CatalogProvider`, `StreamProvider`. A module implements only the roles it fills and declares them in a new `Manifest.Provides []Role` — the manifest-shape growth [ADR 0019](adr/0019-module-capability-and-invocation.md) predicted, and what the `media_types` registry ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) has waited on. The Platform holds a **provider registry keyed by role**; that registry is the seam other modules will resolve through. Per the design call: the contract *supports* module-to-module use, but only the Platform resolves through it in this slice (build platform-first). `Import` survives, narrowed to "materialize *this* virtual result."
- **[ADR 0028](adr/0028-virtual-and-materialized-content.md) — virtual browse and materialized library.** Content exists in two planes. The **virtual plane** is what a provider returns on read — `SearchResult`/`CatalogItem` DTOs, never persisted; browsing is a read. The **materialized plane** is the object-graph library, written only by an explicit **curated act** (an admin publishing a collection, a user picking a search result), which reuses the existing import path. Streams **snapshot onto the curated set** (bounded by curation, so safe). Search is a **union** of providers + local library, deduped by source binding, each row marked *in library* vs *virtual*. This answers "don't overwhelm the DB" structurally: uncurated content never reaches a store.

**Build sequence — pick the lowest unbuilt slice; each passes `go build`/`go vet`/`go test -race` before the next:**

1. **SDK `v0.4.0` — done.** The four provider interfaces, the `SearchResult`/`CatalogItem` virtual DTOs, `Manifest.Provides`, and the import verb narrowed to a `ContentRef`. Published at `sdk` `v0.4.0`, no `replace`.
2. **Stremio providers — done.** All four roles implemented in `module-stremio-addons` (`v0.2.0`): `meta`/`stream` moved behind `MetadataProvider`/`StreamProvider`, and the `catalog` + `catalog/…/search` paths added. (Two blocking addon-reachability bugs were fixed first — manifest-URL normalisation and a User-Agent, without which Cloudflare-fronted addons 403'd.)
3. **Platform — done.** A `CapabilityRegistry` that resolves providers by role (and verifies declared roles at boot); `SearchAvailableContent` fans out to every `SearchProvider` and unions with the library, marking each result in-library by source binding; `ListModuleCatalogs`/`ListCatalogItems` browse a module's collections; `ImportContent` narrowed to materialise a `ContentRef`. Green against real PostgreSQL. (Each was first exposed as a GraphQL field; the screens above and the session transport's `Invoke` lane are what reach them now.)
4. **SDUI screens — done ([ADR 0029](adr/0029-sdui-emit-side.md)).** A Platform-owned screen registry builds `shell`, `home`, `search`, `collections`, `catalog`, `detail`, `settings` and the expert-mode diagnostics screens with the [`contracts`](https://github.com/mosaic-media/contracts) Go producer binding, and the session transport pushes them into the client's regions. Both a virtual and an in-library card open a **detail** screen: an in-library item shows its node's detail; a **virtual** item previews its metadata (through the module's `MetadataProvider`, via a `PreviewContent` query) and carries its sole library affordance — an **Add to library** button whose `Invoke: importContent` materialises it. Wiring the Shell surfaced the action ABI [ADR 0029](adr/0029-sdui-emit-side.md) predicted; it was first resolved over GraphQL and is now the session transport's own `Invoke` lane ([ADR 0061](adr/0061-one-client-transport.md)), which routes straight to the application services. Modules contribute data, not screens ([ADR 0008](adr/0008-sdk-as-public-contract-language.md)).

   **Verified end to end in a browser** against real PostgreSQL and the live Cinemeta + Torrentio addons: search returns virtual results with posters, a virtual result opens a metadata preview, *Add to library* materialises a Work + feature item + imdb source binding + a Torrentio magnet `RemoteLocation` part, and the item then reads *In library* on the next search. The `importContent`-returns-a-scalar half of the ABI was found this way — every Add-to-library click failed until it was fixed.

   **Artwork** is proxied through the Platform's own origin ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)) so a CORS-less CDN can no longer break the image or the artlight wash, and the viewer's IP never reaches the CDN. Slice 1 (the proxy) is built and verified live; the durable filesystem cache of a materialised item's chosen art, and picking the best of several artwork candidates, are slice 2.

   **Known gaps, deferred:** the `SearchBar`'s `$value` client-side input binding and `Query`-into-region partial refresh (the search chrome drives full-screen re-fetch for now); the artwork durable cache + candidate selection ([ADR 0030](adr/0030-artwork-proxy-and-cache.md) slice 2); a Shell loading-state race if a search is issued *during* the boot sign-in (harmless on a settled page); a real detail screen's playback/parts; and the Mosaic Design Language skin over the Shell's neutral tokens.

**Two seams left open on purpose** (named, not answered — an agent hitting them should stop and decide, not invent):

- **Provider precedence** — which provider wins when several fill one role. Moot with one module; belongs with cross-module use.
- **Cross-module authority and the system principal** — whose authority a module-to-module call carries, and the no-user identity background enrichment needs ([ADR 0017](adr/0017-how-a-capability-acts.md)'s reserved gap, shared with module-declared jobs).

### Done: a live, fully server-driven client — **both sides on the ADR 0041 transport; verified live in a browser**

Wiring the Shell exposed two ways the client-server model still cheated on the SDUI thesis; both were decided in ADRs ahead of the build.

- **[ADR 0031](adr/0031-server-owned-app-shell.md) — the Shell is a pure renderer; the app shell is server-emitted. Accepted; built.** The Shell no longer hardcodes any layout. `mosaic-sdui-react` gained two structural primitives — `AppShell` (a named-slot frame: brand, a nav slot, a top-bar slot, a content region) and `NavItem` (a nav button that emits its action and highlights on the current screen) — reusing the existing chrome CSS (the `sdui-react` runtime package). The Platform emits a `shell` screen (nav rail + top-bar search); the Shell fetches it once and fills its content region with the current screen, so **navigation swaps the content while the shell persists**. The Shell now owns only the connection, a client-only `Standby` state, and the renderer. **Verified live**, including the payoff: stopping the Platform drops the client to the standby page — no chrome, never a fake app.
- **[ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) — the client session over a two-lane Connect/gRPC transport. Accepted; Platform side built. Supersedes [ADR 0032](adr/0032-live-session-websocket.md).** The transport is judged by a four-client bar (web, Flutter, Compose, native iOS), not "works in a browser": one `.proto` generates typed stubs per language, and binary framing is cheaper on mobile. `internal/transport/session` serves a `SessionService` — lane 1 unary intents (`Navigate`/`Invoke`/`SubmitInput`/`Attach`), lane 2 one server-streaming `Subscribe` per session for push — over h2c so both lanes multiplex on one HTTP/2 connection. A per-session **outbound mailbox** (a single sender goroutine drains to `Send`; handlers only enqueue) replaces ADR 0032's single ordered read loop; a monotonic per-session `seq` with a bounded replay buffer gives stream resume, folding in [ADR 0033](adr/0033-supervisor-driven-live-handover.md)'s handover; input coalescing moved from the read loop into session state. `Invoke` routes straight to the application services (`ImportContent`/`ConfigureModule`). The record's **GraphQL split** — GraphQL kept alongside as the external/tooling surface — is the one part of it that did not survive: [ADR 0061](adr/0061-one-client-transport.md) abandoned the split by deleting GraphQL outright. The bespoke WebSocket (`internal/transport/live`, `github.com/coder/websocket`) is removed. Race-tested (mailbox ordering, resume/replay, fresh-connect rebuild, supersede-on-reconnect, the reaper). **The client side is ported and the chain works end to end.** `contracts` publishes the generated TypeScript (`@mosaic-media/sdui/session` and `/sdui-pb`), and the Shell's `packages/shell/src/lib/live.ts` is a Connect client (`@connectrpc/connect` + `connect-web`) — unary intents plus the `Subscribe` stream, the full `RegionUpdate` op-set, and resume from a last-seq cursor. **Verified live in a browser** against real PostgreSQL and the live addons: the home screen renders over the session, and Settings round-trips a unary intent into the Stremio module's `SettingsUI` and back as a pushed `RegionUpdate`. **Since closed:** the HTTP round-trip integration test exists (`httptest` + the generated clients, against real PostgreSQL), the runtime's GraphQL invoke/query fallback is gone with the rest of GraphQL ([ADR 0061](adr/0061-one-client-transport.md)), and UINode-as-protobuf (encoding option (b)) landed under [ADR 0044](adr/0044-contracts-protobuf-workspace.md).

- **Slice 3 — resilience + history. Built; verified live.** Three of the four follow-ups below landed. **Socket reconnection + resume:** the Shell reconnects with exponential backoff + jitter (a rolling upgrade drops many clients at once — jitter spreads the retries), re-sends `hello` and **re-declares its current route** on reconnect so the Platform re-renders the exact screen that was showing; a drop reads *Reconnecting…* and self-heals. Resume works because the auth session is DB-backed (survives a Platform restart) while the live-session state is disposable — the client re-declares `{session, route}` and the Platform inherits nothing. On the Platform side, graceful shutdown closes each socket with a **"going away" (1001)** status so the client treats it as a reconnect, not an error (verified by a `-race` test). **Browser back/forward + deep links:** the Shell wires the History API transport-agnostically — `pushState` on a real navigate, `popstate` → the navigate intent for that entry, `replaceState` (no push) for search-as-you-type so keystrokes don't spam history; screens have URLs (`/detail?ref=…`, `/search?text=…`) that are shareable and re-declared on connect. **Server-side input debounce:** the Platform coalesces rapid `input` intents (a short window) before sourcing, the ADR 0032 backstop that protects the upstream addons from a request per keystroke — behind the client's own ~220 ms debounce. All verified live against real PostgreSQL + the Cinemeta/Torrentio addons: search-as-you-type over the socket, back/forward through detail↔search, and a full kill-and-restart of the Platform that dropped the client to *Reconnecting…* and then restored `/search?text=akira` with its results untouched. **Superseded by [ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md):** the Platform-side mechanics here — the going-away socket close, the server-side coalescing on the read loop — are replaced by the Connect transport's stream resume (a reconnecting `Subscribe` presents its `seq` cursor and the server replays or rebuilds) and session-state coalescing. The client-side pieces (backoff reconnect, browser back/forward, deep links) still stand as behaviours but move onto Connect-Web when the runtime is ported.

- **The rich detail page — built; verified live ([ADR 0034](adr/0034-rich-metadata-preview.md)).** The detail screen was thin (a poster, title, overview, one action) because the *contract* was thin, not the data: Cinemeta returns a clearlogo, cast, rating, runtime and — for a series — a per-episode synopsis/still, and the module decoded almost none of it into a seven-field `ContentMetadata`. `ContentMetadata` grew (SDK `v0.5.0`) a `Logo`, `Cast []Person`, `Rating`, `Runtime`, and an `Episodes []EpisodePreview` **read projection** (a series' episode list for the screen — deliberately *not* the materialised tree, which `Import` still builds); the Stremio module (`v0.3.0`) now decodes the fields it dropped (incl. cast from the `links` array and per-`Video` overview/thumbnail/released). The emit-side composes a backdrop+clearlogo **HeroBanner** with the poster docked in its aside, meta pills (year · type · runtime · ★rating), a genre row, a **cast rail** of `PersonChip`s, and — for a series — a **SeasonSelector** over the selected season's `EpisodeRow`s with per-episode synopses and stills. The SDUI vocabulary already had every piece but one; the single addition was a `logo` binding on `HeroBanner`. `PreviewContent` stopped short-circuiting for an in-library ref, so **one ref-based builder serves both planes** — a virtual item and a library item render identically, differing only in the primary action (*Add to library* vs an *In library* marker). A library detail is therefore re-derived live from the provider rather than read from stored fields (as current as the source; the trade is a dependence on a reachable metadata addon — a durable metadata cache is the follow-up). **Verified live** against real PostgreSQL + Cinemeta: *Avatar* (movie) shows the Pandora hero with the AVATAR clearlogo, cast and genres; adding it re-renders the same detail with the *In library* badge; *Avatar: The Last Airbender* (series) shows the season selector and each season's episodes with stills and titles, the season switch re-navigating by ref+season and keeping the selector in sync.

  **Not built — no data in the source, recorded not invented ([ADR 0034](adr/0034-rich-metadata-preview.md)):** clearart/banner artwork, **movie collections** ("the other Avatar films") and **similar/related shows** ("Legend of Korra", the live-action adaptation). Cinemeta provides none of these — they are TMDB/Fanart.tv concepts — and collections/similar for library items additionally need the still-open relation-read gap (`ListFrom`/`ListTo`). They wait on a TMDB/Fanart-class provider and that read surface.

  **Deferred, on purpose:** **finer region updates** — an `input` still re-renders the whole content region rather than just the results grid; the transport now carries the region op-set explicitly ([ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md)'s `RegionUpdate` — `REPLACE`/`APPEND`/`PREPEND`/`REMOVE`/`PATCH`), but the whole-content `REPLACE` is correct today (React reconciliation keeps the search field focused and its value), so narrowing it is an optimization needing runtime region-host plumbing in the `sdui-react` package, left for later. And the **full Supervisor-driven handover** — a seamless make-before-break upgrade between Generations — folds into ADR 0041's stream resume (a rolling upgrade drops the `Subscribe` streams and reconnecting clients replay from their cursor); the deeper make-before-break dance recorded in **[ADR 0033](adr/0033-supervisor-driven-live-handover.md)** stays **deferred** because it depends on the Supervisor, which is largely unbuilt ([ADR 0004](adr/0004-supervisor-as-host-manager.md)–[ADR 0007](adr/0007-static-go-module-composition.md)).

**Carried gaps / cleanup:** the npm gap is closed — the three web repositories were consolidated into the [`web`](https://github.com/mosaic-media/web) workspace ([ADR 0042](adr/0042-frontend-workspace.md)), `@mosaic-media/sdui-react` is published from it by CI, and CI checks the published version against the git tag so a release cannot silently be a local build. The Shell's now-unused `mock/` and `gallery/` are still dead code to remove.

### Done: one client transport — **[ADR 0061](adr/0061-one-client-transport.md); GraphQL deleted**

[ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) left GraphQL standing as the "external/tooling" surface and named *consolidate* as the option it was deferring. Six months of building against that split settled it: the split was not a boundary between two audiences, it was **one live operation and a large unused surface**. The Shell called exactly one GraphQL operation — `signIn`, because `SessionService` had no way to mint a session — and nothing at all called users, permissions, config versions, the content mutations, module catalogs or the jobs/health stubs. Meanwhile `importContent`, `configureModule` and `playPart` each existed twice, once as a resolver and once as a session `dispatch` case, which is the duplication the split was supposed to avoid.

What landed:

- **`mosaic.auth.v1.AuthService`** in the [`contracts`](https://github.com/mosaic-media/contracts) contracts workspace — `SignIn`/`SignOut`, a service of its own because it is the one call made *without* a session. `internal/transport/auth` implements it over `AuthenticateLocalUser`/`RevokeSession`.
- **`internal/transport/rpc`** — the Platform's seven error categories mapped onto Connect status codes, plus the telemetry interceptor moved out of `session` and parameterised by component. The category mapping is the thing GraphQL never actually did: its handler returned HTTP 200 for everything and never populated the `extensions.category` the Shell was reading, so a wrong password and an unreachable Platform were indistinguishable to a client for as long as that path existed.
- **`internal/transport/graphql` deleted**, `graphql-go` dropped, `/graphql` unmounted, and the "make it runnable" end-to-end test rewritten: it now signs in over `AuthService` and then subscribes and navigates over `SessionService` against real PostgreSQL, asserting the pushed content region — the same stack, proven through the only transport there now is.
- **The web workspace lost its second client**: the `gql` client and the runtime's invoke/query fallback are gone, `devSignIn` runs over `AuthService`, and both services share one Connect transport so the traceparent interceptor ([ADR 0054](adr/0054-the-correlation-id-is-the-trace-id.md)) cannot be missing from half the calls. The SDUI `query` action kind — a raw GraphQL string and a region to refresh — is removed from the contract; nothing ever emitted one.

**What this costs, stated plainly.** There is no ad-hoc query surface any more, and reacquiring one is a new transport rather than a flag. Creating roles, granting them, drafting and activating config versions and setting user status are now **unreachable from a client** — they have commands, policy actions and tests, but no surface. That was already true (a mutation with no UI is not a feature) and is now simply visible.

Every one of those is enumerated in **[Unreachable capability](unreachable-capability.md)**, written alongside this change so the debt cannot quietly become "complete". Auditing the deleted schema to build it turned up something the transport change did not cause: `app.CreateLocalUser` is a full, tested command — boundary order, policy denial, unauthenticated rejection, real-PostgreSQL integration — whose **only callers are its own tests**, and which no transport has ever exposed. Mosaic has no way to create a second user, and has never had one. That is the register's thesis in a single row: a green build and a passing suite are not evidence a capability is reachable.

### Done: a curated stream provider beside the addon host — **built; the release detail it parses still cannot reach the Part**

`module-aiostreams` ([ADR 0076](adr/0076-a-curated-stream-provider-beside-the-addon-host.md)) is Mosaic's second stream source and its first **dedicated** one: a module per upstream rather than a host for whatever a user pastes in. It fills `stream`, `subtitles` and `settings_ui` and deliberately nothing else — no `ContentRef`, no `Import` (it refuses), so it is reachable only through [ADR 0073](adr/0073-stream-resolution-is-decoupled-from-metadata-provenance.md)'s enrichment fan-out. Titles come from a metadata module; this fills in what plays.

The reason it exists is a property `module-stremio-addons` cannot have: an addon list is unbounded and self-asserted, its deny-list is documented as non-exhaustive, and there is no ACL that makes "pick from ninety community addons" a safe default. AIOStreams is itself an aggregator, so pointing at one instance keeps the breadth and reduces the trust decision to one URL a user can read.

**One setting, and the default does not work on its own — on purpose.** With nothing configured the module points at the public instance ElfHosted runs, whose bare manifest declares `configurationRequired` with an empty `resources` array. So the module renders three distinct states (unreachable / reachable-but-unconfigured / serving), because the middle one is behaviourally identical to the first — no streams — and one click from the third. Defaulting to it anyway is what gives a fresh install a named instance and a working Configure link instead of an empty box.

**Precedence is module-id order**, the rule the enrichment fan-out already used: `aiostreams` sorts before `stremio`, so the curated aggregator is asked first and the addon host is the fallback. Alphabetical accident, intended outcome, recorded so it is not mistaken for policy — and asserted by an integration test rather than left as a comment.

**Two things this slice forced, and one it did not close:**

- **A settings index — built.** The settings host named one module by constant (`"stremio"`), so a screen was reachable only if you were the first module to contribute one. This module's screen is the whole path from installed to resolving, so it would have shipped unreachable. `ListSettingsModules` enumerates `settings_ui` providers and the host renders an index; that also retroactively gave `module-tmdb`'s credential form a way in, which it had never had. Expert mode moved to the index at the same time — it is an install-level surface and had been rendering under a heading about Stremio addons.
- **Hermetic module tests where the sibling module has none.** `module-stremio-addons`' container reaches real addons on purpose; this one must not, because the only realistic instance is somebody's and its URL is a credential. The fake is reached by putting its URL in the module's settings, which needs no seam in the production type — the instance *is* a setting.
- **A stream provider still cannot express what it knows — not closed.** The module parses container, codec, resolution and swarm health at its boundary; `StreamLink` carries neither container nor codec, and `attachResolvedStreams` writes only the edition label, natural order, location and size. So a Part materialised through enrichment loses every field [ADR 0048](adr/0048-stream-selection-against-a-client-profile.md)'s playability decision reads, and the probe ([ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md)) becomes the only source of them. It affects `module-stremio-addons` identically whenever *it* answers the fan-out rather than attaching its own Parts. Closing it is an additive SDK bump plus a pass-through; it was recorded rather than worked around, and it is the most self-contained SDK gap currently open.

### The first optional module — **done**

An **official optional module** is built exactly as a third-party module would be — its own Go module, importing only the SDK and nothing of the Platform's, compiled into the binary and invoked by the Platform. "Official" describes only its authorship (the Mosaic team), not its shape; the discipline is the point, because building it the third-party way is what proves the third-party way exists. It shipped as the **Stremio addon-source module** (movies and TV), not the anime module first sketched here — because the reference capability is *already* an anime importer, so a second anime consumer would only re-walk the same shape, whereas a different media type stresses the surface from a fresh angle and is the first thing to exercise [ADR 0014](adr/0014-storage-authority-and-transaction-scope.md)'s `RemoteLocation` Part path.

The reference capability already proved the *authoring* half — a package can be written against the SDK alone and drive `ContentService`. This slice built the *composition and invocation* half, which did not exist:

- **A capability/registration surface in the SDK** (a `v0.2.0` addition; the surface has grown additively through `v0.13.0` since, each bump driven by something a module needed). ADR 0008 always reserved "capability interfaces" and "module registration APIs" for the SDK, but ADR 0016 populated only the content services. The shape: a `Capability` interface a module implements — `Manifest()` plus `Import(ctx, ContentService, ImportRequest)` — and a minimal `Manifest`.
- **A capability registry and an `ImportContent` command in the Platform.** The composition root registers each module's capability; a generic `importContent(capabilityId, query)` command authenticates and authorises the caller, then invokes the named capability, forwarding that caller so the module acts as its invoking user ([ADR 0017](adr/0017-how-a-capability-acts.md)) and passing the Platform itself as the `ContentService`.
- **A served entry point over that command**, so a user triggers an import rather than a test doing it. It was a GraphQL mutation then and a session `Invoke` now.
- **Static composition** ([ADR 0007](adr/0007-static-go-module-composition.md)): the composition root imports the module and registers it, standing in for the Supervisor's eventual build-time module *selection*. **Not blocked on the Supervisor** — the mechanism of how a module plugs in is what this defines; the Supervisor automating *which* modules is later.

**Approach: a walking skeleton.** Build the thinnest real vertical end to end — SDK surface → a separate module → static composition → one invocation → real content in PostgreSQL — deciding shapes in code, then capturing what solidifies as ADRs (the module capability/invocation contract, and the composition model), written retrospectively once the code settles rather than as an RFC first.

**Invocation model:** the Platform invokes a *registered capability*. The module depends only on the SDK; the Platform owns the invocation surface and routes to it. The alternative — a module contributing its own GraphQL — was rejected because a third party can't import the Platform's transport without breaking the SDK-only boundary. Captured in [ADR 0019](adr/0019-module-capability-and-invocation.md) and [ADR 0020](adr/0020-optional-module-composition.md), written after the code settled.

**Resource-aware, so streams do not gate metadata.** The module uses whatever resources each configured addon declares (Stremio's `meta`, `stream`, …). Metadata creates the Work and its season/episode tree with an external-id binding; streams attach a `RemoteLocation` Part. The two are independent — a meta-only addon yields metadata with *no* Parts, so a user can enrich local media through Stremio addons without adopting remote streaming.

**Deliberately still deferred by this slice:** the full manifest shape (starts minimal, grows), the `media_types` registry (the Stremio module uses known media types), module-granular permissions (it acts with the invoking user's authority), and play-time stream *resolution*/transcoding (the future Remote Media module — the Stremio module only snapshots a stream location). Those remain future work below.

**Modules are the forcing function for the SDK.** Building the Stremio module is how the SDK's gaps get found. Two are identified so far:

- **User-managed module settings — done ([ADR 0021](adr/0021-module-settings.md)).** A user adds a Stremio addon by manifest URL at runtime rather than by an env var at composition time. A Platform-owned `ModuleSettingsStore` (one jsonb document per module id), generic `configureModule`/`moduleSettings` commands, and SDK `v0.3.0` handing a module its settings through `ImportRequest`. Retired the `MOSAIC_STREMIO_ADDONS` bridge. **A module now authors its own settings screen** through `SettingsUIProvider` ([ADR 0038](adr/0038-module-contributed-settings-ui.md)), which is the exception that proves ADR 0008's rule rather than a breach of it: the module declares a form, not a screen.
- **Module telemetry — done ([ADR 0059](adr/0059-modules-observe-through-the-sdk.md)).** A dependency-free `v1.Telemetry`, reached ambiently.
- **A module reports the version that was actually linked — done.** `v1.ModuleVersion` reads the build graph rather than a constant someone remembered to bump.
- **Module-declared cron/jobs — identified, not built.** A module needs to register recurring work the Platform runs (cleanup, periodic refresh). This converges three deferred pieces at once: the jobs runner (tables exist, no service), a scheduler/recurrence layer (none), and the **system principal** ([ADR 0017](adr/0017-how-a-capability-acts.md)'s named gap — a no-user job has no session to forward). **Four separate threads now want that trio** — this one, the resolution cache's URL refresh, the torrent engine's eviction, and telemetry retention deletion — and it should be built once.

**Harden the SDK toward stable (it is pre-1.0 on purpose, now `v0.13.0`), in parallel or alongside:**

- **The relation-read gap.** `ContentService` can write edges (`RelateContent`) but has no `ListFrom`/`ListTo` to read them, so a capability can't query the graph it builds. Small and additive — a `v0.x` bump. The most self-contained next thing.
- **A second capability** — a different media type (music, comics), built against the SDK, to stress the surface from a fresh angle and surface what a real second consumer needs before the surface stabilises. Likely forces the relation-read gap and, if it introduces a novel media type, the `media_types` registry.

**The rest of the module system** (the slice above starts it; these follow):

- **The full capability manifest shape.** The module slice starts it minimal (id, version, name) and grows it as needs appear — permissions a module declares, media types it sources, whatever the Supervisor's onboarding surfaces.
- **The `media_types` registry** ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) — catches a media type that was never real, which normalisation cannot. Waits on the manifest shape and on a module that introduces a new type (the Anime module does not).
- **Module-granular permissions and a system principal** ([ADR 0017](adr/0017-how-a-capability-acts.md)) — authority a module holds distinct from its invoking user, and a no-user identity for background work. (User permissions management is built, and the module slice acts with the invoking user's authority.)
- **Module distribution and the process boundary** — now decided rather than open: ADRs 0062–0065, the fourth thread above.

**The rest, unplanned in detail, each scoped when it starts:**

- **Export formats** — NFO for other systems, `.mos` for Mosaic-to-Mosaic portability, generated on demand from authoritative state ([ADR 0014](adr/0014-storage-authority-and-transaction-scope.md)).
- **Job queue** — the `jobs` tables exist with no service; `SELECT ... FOR UPDATE SKIP LOCKED` is the intended pattern, for import, provider sync and enrichment.
- **`LISTEN`/`NOTIFY`** — an accelerator over the outbox worker's poll, not a replacement; notifications drop when no listener is connected, so the poll stays the floor.
- **A second client** — the four-client bar [ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) chose the transport against (web, Flutter, Compose, native iOS) has only ever had one client behind it. Everything that would be proved by a second one is currently assumed, and the capability declaration on `Attach` is the first thing that would stop being optional.
- **Mosaic Design Language** — the values that replace the Shell's neutral tokens: acrylic with weight, artwork as the light source. The material work has started ahead of the record (the acrylic "Optical Parallax" sheet and the cinematic home carousel are built); what does not exist is the stated language behind them.

## Working rules

- **One slice at a time**, in order, each passing its gate before the next begins.
- **Report blockers, do not force past them.** The reference capability slice was stopped twice and reported instead of bodged. That was correct both times, and it is why the fix was a design decision rather than a workaround buried in code.
- **Code is authoritative where code exists.** Documentation describes what is built; it does not specify what is not.
- **When implementation contradicts a specification, the specification is wrong.** Fix it there, in the same session, rather than carrying a correction in a repository-local note.
