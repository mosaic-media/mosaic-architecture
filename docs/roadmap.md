# Roadmap

Derived from the real state of `platform`, not from a plan written ahead of it.

---

## Where the build actually is

The Platform boots against real PostgreSQL, serves its client API as two Connect services — `AuthService` to mint a session and the two-lane `SessionService` to spend it ([ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md), [ADR 0061](adr/0061-one-client-transport.md)) — plus two plain-HTTP origins on the same port, `/artwork` ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)) and `/playback/{ticket}` ([ADR 0045](adr/0045-playback-consumer-and-media-origin.md)). It runs an outbox worker with retry and dead-lettering, resolves secrets, reports component health, shuts down gracefully with a final outbox drain, holds a content-agnostic object graph, exposes a content command and query API over that graph, and records its own behaviour as structured logs and spans it can serve back ([ADR 0053](adr/0053-telemetry-is-ambient-in-context.md)–[ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md)). That API's surface has been extracted into a standalone SDK module — [`github.com/mosaic-media/sdk`](https://github.com/mosaic-media/sdk), published at `v0.13.0`, holding the content models, the ten `ContentService` methods, the `Capability` interface and its `Manifest`, the seven provider roles, the dependency-free `Telemetry` interface and an opaque `Caller` — which the Platform, the reference capability and both optional modules depend on as an external module. Every slice passes `go build`, `go vet` and `go test -race` against a real database.

**A slice recorded as done here means the Platform capability landed. It does not always mean a user can reach it.** Permissions management, configuration versioning and user administration are all recorded below as done, and are all correct — and none of them has a client surface. [Unreachable capability](unreachable-capability.md) is the register of that gap, kept separately because nothing in the build or the test suite will ever report it. Read the two together before concluding Mosaic can do something.

The repositories are public and licensed ([ADR 0022](adr/0022-licensing.md)): the Platform under AGPL-3.0 with a module-linking exception, the SDK under Apache-2.0, optional modules under their authors' choice (the first is MIT), and this documentation under CC-BY-4.0.

Two further slices — uniform store resolution and its PostgreSQL follow-up — were built and then reverted under [ADR 0012](adr/0012-capabilities-do-not-own-stores.md), which found they solved a case the architecture had already ruled out.

The content model, its published surface, a capability that uses only that surface, and the surface's extraction into a standalone SDK module have all landed — **the thesis test passes and the critical path is complete.** Since then two optional modules — the **Stremio** addon source and **remote playback**, the first *consumer* module — have been composed into the binary and invoked through a capability registry, so the composition-and-invocation half of the extension story works too ([ADR 0019](adr/0019-module-capability-and-invocation.md), [ADR 0020](adr/0020-optional-module-composition.md)). The Shell — a Server-Driven-UI client in the [`web`](https://github.com/mosaic-media/web) workspace ([ADR 0023](adr/0023-server-driven-ui-and-the-shell.md), [ADR 0024](adr/0024-primitives-and-definitions.md), [ADR 0042](adr/0042-frontend-workspace.md)) — is built too, and the Platform emits the screens it renders and pushes them over the live session. A user can search, add a film and watch it. What follows (media formats, module distribution, a second content capability) builds on a foundation that is proven rather than assumed.

The build spans **seven repositories**: `architecture`, `platform`, `sdk`, `sdui`, `web`, and the two modules. The three former web repositories were consolidated into `web` under [ADR 0042](adr/0042-frontend-workspace.md) and archived.

## The critical path

Everything below is one thread. Nothing else should start until it lands, because it is the test of Mosaic's central thesis: **that a developer who is not you can extend Mosaic through the SDK.**

### 1 — The content-agnostic object model — **done**

The node tree and relation graph, designed in [ADR 0013](adr/0013-object-graph.md), with authority and media linking settled in [ADR 0014](adr/0014-storage-authority-and-transaction-scope.md).

`nodes`, `parts`, `relations` and `source_bindings` shipped with domain types, store contracts, PostgreSQL implementations, the four stores added to `Tx`, and adapter-agnostic contract tests against real PostgreSQL. Identifiers are UUIDv7 in native `uuid` columns; the existing infrastructure tables keep their `text`/UUIDv4 ids and were not migrated. ADR 0013's four deliberate non-uniformities each have a contract test, since each is cheap to normalise away by accident.

This was the real blocker, and it had been mistaken for something else. The reference capability was recorded as blocked on an empty `contracts/platform/v1` and a closed `Tx`; both were symptoms of building the extension mechanism before the thing it extends. A capability now has somewhere to put an anime.

**Deliberately not in the slice, and still unbuilt:** export formats, the filesystem projection, streaming, the job queue, `LISTEN/NOTIFY`, and IPTV listings. None blocks the reference capability.

### 2 — Content commands and queries — **done**

The stores existed and were proven, but nothing above them did. No application service touched the graph, so the only way in was `Tx`, which the composition root holds — and a capability reaching for `Tx` directly would bypass the policy engine entirely.

This slice put the command order over the content model. Nine application services now exist, each validating, authenticating, authorising and — for writes — committing state and an outbox event in one transaction:

- **Reads:** `SearchContent` (title substring, media type, kind), `FindContentByExternalID` (provider id against the GIN index), `GetContentNode` (one node, optionally its direct children). Before this, `FindBySource` answered only *"have I bound this exact provider reference?"*; "do I already have this anime?" by title or external id was unanswerable, and the `nodes_external_ids_gin` and `nodes (media_type, title)` indexes had no query reaching them.
- **Writes:** `AddContentWork`, `AddContentChild`, `AttachContentPart` build the containment tree; `RelateContent`, `BindContentSource`, `ResolveContentBinding` write the association graph and resolve identity.

Two shapes were decided in the building. A child inherits its work id and media type from its parent, so a season cannot declare a different media type than its series. A binding is created confirmed or pending-review, never rejected — rejection is a resolution of an existing binding, not a state to create one in.

It was scheduled ahead of the reference capability because it is needed under every resolution of the boundary problem below, so none of it could be wasted. It did **not** resolve either blocker: every content command still authorises on a user session, and the contracts still reference `domain` types under `internal/`.

### 3 — Reference capability path — **done**

Attempted twice and correctly reported as blocked rather than forced, then blocked a third time on two things neither attempt had reached — both decided (ADR 0016, ADR 0017) and built. The capability now exists and passes: **the thesis holds.** A package that imports only `contracts/platform/v1`, owns no schema and touches no Platform internals sources an anime over HTTP, searches by provider id to avoid duplicating, creates the work with a source binding, builds its season/episode/part tree, and — honouring ADR 0013 — creates the source manga as its own Work joined by an adaptation edge rather than folding it into one tree. It runs against real PostgreSQL through the postgres harness: a fake provider serves canned metadata over `httptest`, and the test asserts the work is findable by external id, the tree is where it belongs, the edge and both bindings landed, every command emitted its event, and a second import is idempotent.

`capabilities/reference/` lives outside `internal/` and a boundary test parses its imports to keep it to `v1` and the standard library — the stop point made executable. It owns its provider integration outright (an HTTP metadata source on `net/http`), since the Platform offers no HTTP contract and does not need to (ADR 0007).

Two findings from building it. The published surface was **sufficient** — no private import was needed, which is the proof the contracts are ready. And one gap: `ContentService` has no *read* for relations (`ListFrom`/`ListTo`), so the test reads the adaptation edge through the store; the capability's own flow does not need it, so it is a candidate addition to `v1`, not a blocker.

The remainder of this section records how the block was cleared.

#### Done: the published surface ([ADR 0016](adr/0016-published-contract-surface.md))

The block was that every contract signature references a `domain` type under `internal/`, which Go forbids an external module from importing. The fix corrected the roadmap's own framing: a capability does **not** call the store contracts (`NodeStore`, `Tx`, `StorageAdapter`) — those are Platform↔engine plumbing. It calls the application-service API.

`contracts/platform/v1` now holds that surface: the content models (moved out of `internal/platform/domain`, not duplicated), the nine content command, query and result types, the `ContentService` interface `app.Service` implements, and an opaque `Caller`. The store contracts and the identity and configuration models stayed internal. The enforcement is a standing test — `test/sdkprobe` is a separate Go module that imports only `v1` and exercises the whole service; `test/sdkboundary` builds it, and because an external module cannot import `internal/`, a public signature that leaked an internal type fails that build. Verified by watching it fail on a deliberately added internal import.

#### Decided: how a capability acts ([ADR 0017](adr/0017-how-a-capability-acts.md))

A capability does not originate authority. The Platform invokes it within a context carrying a principal, and it forwards that context to every service it calls. For the reference capability the principal is the invoking user, so nothing in the policy engine changes and every created node traces to the person who caused it. This is realised as the `Caller`: a session reference the capability forwards, only as authoritative as the session behind it. A system principal for background work, and module-granular authority, are named future decisions rather than machinery built now.

#### How the block was cleared

Its purpose changed under [ADR 0012](adr/0012-capabilities-do-not-own-stores.md). It was to prove a capability could own a store and join a transaction. Since capabilities own no schema, it proves instead what a capability actually does, shaped like the anime module the platform exists to support: source metadata from an external provider, search existing content, create nodes and relations, and cause events. All four are done.

Sourcing metadata needs no Platform HTTP contract, which is worth stating because it looks like a gap and is not one. Capabilities compile into the binary with trust established before the build ([ADR 0007](adr/0007-static-go-module-composition.md)), so a capability imports `net/http` itself — as the reference capability's metadata source does. A Platform-provided client is worth having eventually for rate limiting and secret handling; it is not a prerequisite.

#### Not carried here after all: the `media_types` registry

The original plan ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) had this slice carry the registry that catches a media type which was never real (a missing separator or a misspelling, which normalisation cannot recover). It did not, and ADR 0015 is amended to say why: the reference capability is an anime importer that uses only known media types, so it never introduces a novel one, and the registry depends on the capability manifest shape, which is still undecided. The registry now waits on that manifest mechanism and on a capability that genuinely introduces a new type. Until then normalisation is the whole of the enforcement.

**Exit criteria — met.** The surface of [ADR 0016](adr/0016-published-contract-surface.md) is published into `contracts/platform/v1`, and one capability does all four using only those packages, acting as its invoking user ([ADR 0017](adr/0017-how-a-capability-acts.md)), owning no schema and touching no Platform code.

### 4 — SDK extraction — **done**

Whether the contracts proven across the completed slices can leave the Platform repository as a standalone SDK a third party can build against. They can, and they have: the surface is now its own module, [`github.com/mosaic-media/sdk`](https://github.com/mosaic-media/sdk), published at `v0.1.0`.

`contracts/platform/v1` moved out of the Platform repository into the SDK module unchanged, and the Platform now depends on it as an ordinary tagged dependency — no `replace` directive, so a fresh clone resolves it through the module proxy. Several separate modules build against the published SDK: the Platform, the reference capability, `test/sdkprobe`, and the Stremio module in its own repository. The surface holds no private Platform internals, because Go itself now forbids it — an external module cannot import the Platform's `internal/`, so a leak would fail to compile rather than needing a test to catch it.

**Exit criteria — met.** The surface left the repository as a standalone module, and the Platform and the reference capability build against it as external consumers.

The SDK is pre-1.0 deliberately (now at `v0.13.0`, thirteen additive bumps driven by what modules turned out to need) and may still change. The relation-read gap noted in step 3, and any surface a second capability turns out to need, are the kind of change a `v0.x` bump absorbs before the surface is declared stable.

### The stop point — cleared

> **The Platform is ready for SDK work when the reference capability uses only published contract packages and no private Platform internals.**
>
> **If the reference capability requires a private import, the contracts are not ready to publish.**

The reference capability landed against `contracts/platform/v1` alone, enforced by a boundary test that parses its imports. It needed no private import, so the line held rather than being waved through — which is what makes the surface ready to extract.

**Step 3 was the thesis test, and it passed; step 4 extracted the surface it proved.** A capability can be built entirely against the published contract surface, and that surface is now a standalone module a third party depends on rather than a package inside this repository. The extension model is real, end to end.

### Since the thesis: the platform runs

Beyond the critical path, the Platform is now a usable process rather than only a tested library:

- **A client API is served over HTTP** — the composition root constructs `app.Service`, and the API listens on `:8081` (the Supervisor handoff surface stays on `:8080`). It began as a hand-written GraphQL handler; that was replaced by the Connect transport and then deleted outright ([ADR 0061](adr/0061-one-client-transport.md), recorded below).
- **Real password auth** — an Argon2id hasher (`internal/adapters/crypto`), so sign-in actually verifies.
- **Permissions can be assigned through the Platform** — `PermissionStore` gained `CreateRole`/`GrantRole`, with commands behind them. `PermissionStore` was read-only before; authority could only be seeded by raw SQL. Those commands have no client surface, which is the [unreachable capability](unreachable-capability.md) register's first row.
- **The first admin bootstraps itself** — `bootstrap.EnsureAdmin`, env-gated and idempotent, so a human can start the binary and use it ([ADR 0018](adr/0018-first-admin-bootstrap.md), a deliberate bridge until Supervisor onboarding owns it). Proven end to end over HTTP against real PostgreSQL: sign in with a password, import a work and a season, query them back.

### Acceptance baseline

The standing gate every slice passes before the next begins:

- `go build ./...`, `go vet ./...` and `go test -race ./...` pass
- adapter contract tests pass against real PostgreSQL
- migration tests run from an empty database
- import boundary checks pass (`test/sdkboundary` builds `test/sdkprobe` as an external module; `capabilities/reference` is parsed for its imports)
- the boundary conformance suite passes — it reflects over `*app.Service` and fails when a caller-bearing method is missing from its table or does not answer `Unauthenticated` then `PermissionDenied` ([ADR 0066](adr/0066-authorization-is-carried-in-the-type.md))
- the end-to-end test signs in over `AuthService`, then subscribes and navigates over `SessionService`, asserting the pushed content region
- health probes answer against a running process

## What is next

The critical path is complete and the platform runs. Four threads are open, and two of them are half built.

**The active thread is remote playback** (immediately below) — **built through selection; the durable state and the payoff are not.** A film added from a Stremio addon plays in a browser today, chosen out of the whole candidate set and remuxed per stream where the browser cannot decode what the source had. What it does not do is remember where you got to.

Behind it, **telemetry and observability** — **built through the expert-mode UI**; metrics, the audit store and OTLP export remain. It was the limiting factor on development speed across seven repositories, and it is no longer: a click now carries one trace id to the committed row.

Behind that, **more than one user** (ADRs 0066–0069, partly built) — Mosaic has never had a second account, which is the difference between a household media server and a personal one.

A fourth thread was opened after those: **two module tiers and how a module is distributed** (ADRs 0062–0065, decided, nothing built), which supersedes the per-install build pipeline the Supervisor records assumed. The rest (exports, jobs, the remaining SDK gaps) runs around all four.

### The active thread: remote playback, the first consumer capability — **built through slice 3 except its two client-facing gaps; slices 4–7 unbuilt**

The library was **inert**: `Import` snapshotted a `RemoteLocation` Part whose `Ref` is a direct URL or a magnet ([ADR 0028](adr/0028-virtual-and-materialized-content.md)), nothing resolved it, nothing served bytes, and the `Play` action resolved to a toast. That is fixed — `module-remote-playback` (`v0.2.0`) resolves a Ref, the Platform mints a ticket and relays the bytes from its own origin, and a `Player` node renders in the Shell. **What remains is everything durable about a playback session**: nothing is remembered between one play and the next, so there is no resume, no continue-watching rail and no watched mark.

**Stremio is the reference, and what it shows is the split, not the protocol.** Mosaic already has the addon protocol, complete ([ADR 0037](adr/0037-completing-the-stremio-source-surface.md)). Stremio is *three* parts: addons return stream descriptors; a local **server** turns a descriptor into a seekable byte-range HTTP origin (torrent engine, remux, subtitle conversion, cache); a player consumes that origin, reporting a position that syncs independently of whichever addon served the bytes. Mosaic had the first and neither of the others, and reading the module contract as "addons" is what kept the gap invisible. **It now has two and a half:** the origin exists and remuxes, but it serves only what a debrid service already resolved to a URL — there is no torrent engine — and the player reports no position.

Four decisions were written ahead of the build, as [ADR 0027](adr/0027-modules-as-typed-capability-providers.md) and [ADR 0028](adr/0028-virtual-and-materialized-content.md) had been. **Three of them are built, each in part; [ADR 0046](adr/0046-playback-state-is-platform-owned.md) is untouched**, which is why nothing is remembered. The unbuilt parts are named in the sequence below.

- **[ADR 0045](adr/0045-playback-consumer-and-media-origin.md) — the playback consumer contract and the Platform-hosted media origin. — built for `Direct`; `Served` is not implemented by anything.** `RolePlayback`/`PlaybackProvider` is the first **consumer** role in a vocabulary that had been source-only. A resolution is discriminated — `Direct` (a URL, from an addon's plain `url` stream or a debrid service) or `Served` (an `io.ReadSeekCloser` the module produces, for a torrent engine or a subtitle transcode). The module never speaks HTTP: the Platform mints a signed, short-lived, session-bound ticket, hosts `GET /playback/{ticket}` and `http.ServeContent`s the reader, so ranges and seeking come free and the viewer's IP and any debrid token stay server-side ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)'s argument, applied where it matters more). Streams and subtitles from other modules are resolved **by the Platform** and passed in as data, keeping [ADR 0017](adr/0017-how-a-capability-acts.md)'s cross-module authority seam closed.
- **[ADR 0046](adr/0046-playback-state-is-platform-owned.md) — playback state is Platform-owned. — not built.** Position, duration and finished, keyed by **(user, node)** — not by part, because a user resumes an episode, not a particular release of it. A fifth Platform store beside the four content stores; written through `ContentService` as the invoking user; read as an in-progress query that the continue-watching rail renders server-side. The playback module cannot own it ([ADR 0012](adr/0012-capabilities-do-not-own-stores.md)), and should not: a local-file player and an export module read the same surface.
- **[ADR 0047](adr/0047-player-as-client-primitive.md) — the player is a client primitive over a server-issued ticket. — built for one direction.** The SDUI thesis gains its first stated limit, and a narrow one: **the server owns everything about a playback session except the decoding pipeline and the transport controls.** `playPart` resolves to a `Player` node carrying the ticket URL and a resume offset, and the client renders it. **The other direction does not exist**: the client declares no capabilities on `Attach` and reports no progress back, so the node carries no subtitle tracks and no next-up reference and the resume offset is always zero. Those are the same gap seen from two ends — slice 3's hard-coded browser preference and slice 5's missing state. Same move as `AppShell` ([ADR 0031](adr/0031-server-owned-app-shell.md)) — a structural primitive owning a mechanism the server names but does not micro-manage.
- **[ADR 0048](adr/0048-stream-selection-against-a-client-profile.md) — stream selection against a client profile. — built, against a server-assumed profile rather than a declared one.** The decision that makes playback *work*, and the one place Mosaic should **not** copy Jellyfin or [remux](https://github.com/lostb1t/remux). Their decision engines read a device profile and fall back to transcoding, which is correct when a library is one file per item. A Stremio aggregator returns *forty* candidates per item — different releases, containers, codecs, qualities — so the library is a menu, and transcoding an incompatible MKV does expensive work to reach a result already sitting in the list. **The Platform ranks candidates against the declared profile and picks the best playable one**; the user can override through the source picker; no playable candidate is an honest state with counts and reasons. The ranking is built and reports what it chose out of how many; the picker and the honest empty state are not. Whole-file transcoding stays deferred — what the build added instead was the finer axis [ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md) names, encoding one stream and copying the rest.

**The browser is a deliberately limited-mode client**, accepted knowingly. MSE takes only fMP4 and WebM, so **Matroska cannot pass through it whatever codec is inside** — a 1080p h264 MKV is as unplayable as an HEVC one — and Chrome decodes no AC3/EAC3. Selection cannot manufacture a release no scraper offers, so much of what it finds will be h264/AAC at 720p/1080p while 2160p HEVC stays out of reach in a browser. What closed most of that gap was not the renderer but the **per-stream decision** (slice 2): an MKV is rewritten into fMP4 with its video copied, and only the audio it cannot decode is re-encoded.

**The web player is a bare `<video>` element, not Shaka** — a revision of what this thread first planned, recorded as [ADR 0070](adr/0070-the-web-player-is-the-browser.md), which supersedes that bullet of ADR 0047 and the profile-source clause of ADR 0048. Shaka adds no codec support of its own — the browser decodes either way — and both paths the Platform serves today are progressive MP4, so it would forward to the same element behind a dependency. The trigger to revisit is a served format, not a date: it goes in when the origin emits HLS. Until then the browser's own `canPlayType` is what a profile would be measured from. **A desktop client with libmpv is the real answer to a fat profile** — it is what Stremio and Seanime both ship, and it is a later thread.

**Build sequence — pick the lowest unbuilt slice; each passes `go build`/`go vet`/`go test -race` before the next:**

1. **The skeleton — one file, playing, end to end. — done.** The thinnest real vertical, the method the first optional module used: `RolePlayback`/`PlaybackProvider` with **only** `Resolve` returning **only** `Direct`; `ResolvePlayback` reading the Part and minting a ticket; `internal/transport/playback` relaying with range pass-through; `module-remote-playback` resolving an `http(s)` Ref; a `Player` node ([ADR 0047](adr/0047-player-as-client-primitive.md)) rendered by the web runtime's `Player` primitive. No state, no selection, no `Served`, no lifecycle. It existed to hold the two contracts with the most risk — the player and the ticket — before anything was built on them, and both held: the ticket is minted in the transport and the upstream URL never leaves the server, `playPart` resolves to a bare node that rides its own region so the screen underneath survives the player closing.
2. **The ACL, the probe and the per-stream decision — done** ([ADR 0050](adr/0050-probing-and-the-per-stream-playback-decision.md), [ADR 0051](adr/0051-modules-as-anti-corruption-layers.md)). Slice 1 was run against a live AIOStreams instance and failed in a way that rewrote this slice. The resolved URL carried its container in a **query parameter** (`…MIRCrew.mkv&name=…&media_id=…`), so the extension heuristic saw none and would have relayed ten gigabytes of Matroska to a browser. Two fixes, and neither is more parsing.

   **The module translates at the boundary** — a dialect table keyed on addon manifest id, an explicit list of sources actually tested, a generic fallback for the rest, and the SDK's typed fields (`Part.Container`, `VideoCodec`, `AudioCodec`) filled instead of left empty. [ADR 0043](adr/0043-repository-naming-convention.md) always called a module an anti-corruption layer; this is it doing the job, and a community member adding a dialect is a module bump touching nothing else.

   **ffprobe decides what a release actually is**, run against the resolved URL — the approach [remux](https://github.com/lostb1t/remux) uses (`playback/probe.rs`), and strictly better than reading a filename. It range-requests the header rather than downloading the file, so asking is cheap even for a thirty-gigabyte release. The parse ranks the list; the probe settles the winner. **The result is not persisted on the Part**, as the plan called for: it is sealed into the ticket so the origin does not re-probe on every range request a seeking player makes, and it is re-derived on the next play. Persisting it is worth doing with the resolution cache, since both are the same durable/perishable question.

   **And the decision is per stream, not per file.** Copy the video, encode only the audio — which is why the origin plays an h264+E-AC3 release in a browser that decodes neither E-AC3 nor Matroska. "Remux or transcode" was too coarse an axis and the code written for it said so; it was corrected here. Building it added one thing the record did not predict: **HDR is tone-mapped rather than copied**, because passing HDR10 metadata to an SDR browser decoder produces the purple-and-green picture that made the bug obvious.

   **What did not land with it: HLS.** The plan called for anything encoded to be emitted as HLS, which is what makes an encoded stream seekable and would retire ADR 0045's non-seekable caveat. The origin emits **fragmented MP4 off a pipe** instead — it has no index and no length, so it answers `Accept-Ranges: none` and a player treats it as unbounded. **A remuxed stream therefore cannot be seeked or resumed**, which is stated rather than worked around: it is the constraint slice 5's resume mechanism will hit, and segmenting is what clears it.

3. **Selection, capability classes and the resolution cache — selection, the declaration, the classes and the cache are built; the source picker and the no-candidate state are not** ([ADR 0048](adr/0048-stream-selection-against-a-client-profile.md), [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md)).

   **Built.** Container, video codec and audio codec are carried on the attach-part command and filled by the module's dialect table; **import writes the whole candidate set as Parts** instead of one, sampled across resolutions rather than off the head of the list, and a re-import tops the set up; `ResolvePlayback` filters and ranks against a `PlaybackPreference` and reports what it chose out of how many, so a bad pick is distinguishable from a short list. `FirstPlayablePart` walks a work to its entry point.

   **The client now declares what it can play, and the hard-coded browser preference is gone.** `AttachRequest` gained an optional `ClientProfile` (sdui `v0.10.0`), the web Shell fills it from the browser's own `canPlayType` rather than from a table — which is what makes a Safari that decodes E-AC3 and a Chrome that does not resolve to different releases without either being told about the other — and the Platform stores it on the live session, so one read serves both the ranking and the per-stream plan. Ranking them against different profiles would re-encode the very release selection chose to avoid re-encoding. **HDR joined `PlaybackPreference`** at the same time: a client that cannot render it penalises HDR candidates, because the fix is a tone-map and a tone-map is a full video re-encode — the purple-and-green failure of slice 2, moved one step earlier where it costs nothing.

   **The capability class and the resolution cache are built, and verified against real PostgreSQL.** The class is a sorted digest of the declared profile, computed once on `Attach`; sorting is the load-bearing part, because two clients listing the same codecs in a different order must land in one class or the cache fragments with no symptom other than never seeming warm. `playback_resolutions` is keyed by **(part, capability class)** and by nothing else — not by user, not by device — and sits off `Tx` deliberately: it has no outbox event to be atomic with, and ADR 0049 requires its write not to block the stream. The integration test proves the hit by *removing the module*: a second Service with no playback capability registered, the configuration that otherwise fails with "no playback module is installed", resolves the same part successfully — so the answer can only have come from the cache. A different class misses, and an unclassed client bypasses the cache rather than sharing one bucket.

   **Departures worth recording.** A cache hit reports an empty `ModuleID`, which is accurate — no module was asked — and has the side effect that a cached play survives an uninstalled or unreachable playback module. The cache write happens on the request goroutine rather than in the background, which still satisfies the record: nothing is streaming yet when it runs, the client has not been handed a ticket, and a write failure is logged and swallowed rather than failing the play. And a client that declares no display ceiling still gets one on an *encode* (1080p), because an uncapped panel is not an argument for an uncapped encoder.

   **Not built.** The **origin's invalidate-on-read retry**: a dead cached link currently fails the play instead of being re-resolved transparently, which is the half of ADR 0049 that makes the cache safe rather than merely fast — it needs the ticket to carry the part and the class so the origin can re-ask. There is no **source picker** and no **no-candidate state**, so a user cannot override the ranking and an item with nothing playable presents as a failure rather than as an answer with counts and reasons. The **refresh cron** remains blocked on the jobs runner, the scheduler and the system principal, as ADR 0049 said it would be.

   The shape that makes it fast is the **durable/perishable split**: candidates persist forever, resolved URLs are cached lazily per capability class and corrected when they fail. Invalidate-on-read, with the origin re-resolving transparently on a failed fetch, is the mechanism; a TTL is only a hint, because a debrid link also dies whenever its torrent leaves the provider's cache.
4. **Cache-first rendering — not built** ([ADR 0052](adr/0052-cache-first-rendering-and-source-health.md)). Found by restarting the Platform under a live client: the home screen rendered *"Nothing to show yet — try adding an addon in Settings"* on an install with a configured addon and a library full of content. The client was blameless — it reconnected unprompted and the server rebuilt and pushed — but the emit-side discards catalog errors (`if err == nil`), so when every cold call failed the page had no rows and fell through to the empty state. A transient upstream failure presented as a configuration mistake, which is worse than an error because it sends a user to fix something that is not broken.

   The fix: source-backed screens render from a durable snapshot when one exists, revalidate in the background, and push the live result as a `RegionUpdate` — the first real use of the op-set ADR 0041 defined and nothing has exercised. The snapshot holds *items, never the rendered tree*: artwork and playback URLs are signed with process-scoped keys, so a cached tree would come back after a restart full of URLs signed by a key that no longer exists, producing a page that looks right and is broken. A source that stays unreachable gets a **persistent** shell notification rather than a toast, because a transient message for a lasting condition is announced once and then invisible.

5. **Playback state — not built** ([ADR 0046](adr/0046-playback-state-is-platform-owned.md)). The store and migration, the commands and queries, coalescing on the session, resume on the detail screen. The `Player` node already carries a `ResumeAt` the server always sets to zero, and the client already applies it once the element can seek — the surface is in place and there is nothing behind it. Its "resume at position T" mechanism is also half of stall-triggered reselection, which lands here rather than needing its own slice. **It runs into slice 2's carried gap:** a remuxed stream cannot seek, so resume is exact only on a directly relayed one until segmenting lands.
6. **The payoff — not built.** Continue-watching on home, watched marks on `EpisodeRow`, next-episode continuity — the point at which search → add → play → resume works end to end. **Play materialises on commitment, not on click**: a virtual item resolves and plays straight from its `ContentRef`, and is written to the library only once the viewer has actually watched past a threshold. That keeps [ADR 0028](adr/0028-virtual-and-materialized-content.md)'s "materialising is a deliberate act" intact by making *sustained watching* the deliberate act, rather than filling a library with everything anyone sampled for thirty seconds.
7. **The affordance gate — not built** ([ADR 0036](adr/0036-capability-gated-affordances.md)). Moved to the end, and the ADR's own sequencing revised with it: it argued the gate is buildable before any consumer exists, which was right while playback was indefinite. Building it *first* now would mean writing the predicate against a role interface that does not exist and shipping an empty state about to be exited. "Consumer absent" is only testable once "consumer present" exists to contrast against.

**Verification fixture: AIOStreams + TorBox, and it earned its keep.** The owner runs it, and it was a better fixture than a synthetic one: it is what caught the container-in-a-query-parameter bug that rewrote slice 2, the HDR copy that produced a purple picture, and a candidate list whose head was all one resolution. AIOStreams resolves the debrid side itself and returns streams carrying a `url`, which [`Stream.Ref()`](https://github.com/mosaic-media/module-stremio-addons) prefers over a magnet — so **the playback module needs no debrid API integration at all**, and its config-path manifest URL is the shape `normaliseAddonURL` already handles.

**Deferred and named, not omitted:**

- **Segmented output (HLS)** — the seekability gap slice 2 left open, and the closest thing to a prerequisite the remaining slices have: without it, resume works only on a directly relayed stream.
- **Full transcoding.** Deferred rather than rejected — it is the right answer for local files and for what per-stream encoding cannot cover, and it costs session lifecycle and hardware acceleration on top of the ffmpeg dependency the origin now has. It is also the **only** route to true adaptive bitrate: seamless mid-stream switching needs one encode segmented into an aligned ladder, and a menu of unrelated releases cannot supply that at any level of effort ([ADR 0048](adr/0048-stream-selection-against-a-client-profile.md) records why). Bandwidth-aware selection and stall-triggered reselection are what Mosaic gets until then.
- **A desktop client with libmpv.** The fat profile that makes most of the above moot for the users who install it.
- **The torrent engine** (the other half of `stremio-server`): sequential piece selection, cache and eviction — the `Served` half of ADR 0045's resolution. Until it lands, a magnet Part no debrid service resolves is honestly unplayable rather than silently broken. It is what forces **module lifecycle** (`Start`/`Stop`), a **Platform-granted scratch directory**, and — for prefetch and eviction, which have no user — the **system principal** [ADR 0017](adr/0017-how-a-capability-acts.md) reserved.
- **The URL refresh job.** Keeping in-progress items (and the next episode of an in-progress series) warm in the resolution cache. Deliberately split from the cache's own read path, which is self-sufficient: the job needs a scheduler, the jobs runner (the `jobs` tables exist with no service) and — since background work has no session — the **system principal** [ADR 0017](adr/0017-how-a-capability-acts.md) reserved. Those are one slice, and invalidate-on-read must not wait on them.
- **Per-user provider credentials.** Module settings are per-module, not per-user ([ADR 0021](adr/0021-module-settings.md)), so one debrid account serves the whole install. Harmless while the Platform relays and no credential reaches a client, but it is a real question [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md) names and does not answer.
- **Multi-user visibility.** Playback state is the first content state that differs between two users of one install. Whether a household shares a continue-watching rail is a real question ADR 0046 opens and does not answer.

**One carried gap this thread makes load-bearing.** The **relation read** (`ListFrom`/`ListTo`) — open since the reference capability — is needed for next-episode and series rollup, so it moves from tidiness to a user-visible feature. (The naming gap is closed: [ADR 0043](adr/0043-repository-naming-convention.md) now names the consumer case explicitly — a consumer module names its *capability*, because it has no upstream, and `module-remote-playback` is the worked example.)

### The next thread: telemetry and observability — **built through the expert-mode UI; metrics, audit and export remain**

The build spans seven repositories and more than one process, and a bug crosses several of both. A user action passes through the Shell, the session transport, a command handler, a module in its own repository, a third-party HTTP call, PostgreSQL and the outbox worker — seven hops, four repositories — and **nothing joined them**. Diagnosing a failure meant reading timestamps in three places and guessing. That cost was the limiting factor on development speed, which is what made this a thread rather than a chore.

What existed was better than it looked, and is what these records built on rather than replaced. `diagnostics.Logger` already had a **fail-closed** redaction model — a field is redacted unless explicitly classed safe. `domain.Event` already carried `CorrelationID` and `CausationID`, with a comment saying they were *"Empty until request-scoped propagation exists."* The `event_outbox` already committed atomically with the state change that produced it. The seven error categories were a ready-made span status and metric dimension. Config already declared a reload class per field.

What was missing was that almost none of it was used: the good logger was called from three non-test places, the boot narration was `fmt.Printf` and the session transport's lines were `log.Printf`. That is now inverted — telemetry is ambient in `context.Context` and reached with one line at a call site, two `Printf`s survive in non-test code, spans cover the nine seams, and a click in the browser and the row it commits share one trace id.

Eight decisions, written ahead of the build:

- **[ADR 0053](adr/0053-telemetry-is-ambient-in-context.md) — telemetry is ambient in `context.Context`. — built.** No constructor takes a logger; every method already takes a `ctx`, and that is the carrier. One line at the call site, full correlation in the output — the friction that pushed people to `fmt.Printf` is gone, and so are all but two of the `Printf`s.
- **[ADR 0054](adr/0054-the-correlation-id-is-the-trace-id.md) — the correlation ID is the W3C trace ID. — built.** No second identifier. The Shell mints it where the user actually clicks, a Connect interceptor carries `traceparent` on every call, `Event.CorrelationID` finally has a source, and the outbox *links* rather than parents so a later publish does not appear as a request that hung for minutes. Implementation corrected one thing: the span tree roots at the caller's span rather than at a phantom the server invented.
- **[ADR 0055](adr/0055-instrument-at-the-seams.md) — instrument at the seams, not the call sites. — built.** Nine of them. The uniform command handler order is the decisive one: the rule that made the codebase uniform is what makes it observable, and a new handler is instrumented because it is a handler. Seam nine — `netguard.ModuleClient`, handed to every module instead of `nil` — closed the SSRF gap alongside the tracing.
- **[ADR 0056](adr/0056-redaction-classes-are-the-pii-boundary.md) — redaction classes are the PII boundary; no scrubbing pipeline. — built, except its enforcement.** Redaction at construction, so a sensitive value is never buffered or written at all, and `Identifier` (salted hash) so occurrences correlate without storing who. **The vet-style check is not built** — the rule is currently developer discipline, which is the arrangement [ADR 0066](adr/0066-authorization-is-carried-in-the-type.md) demonstrated does not hold. The `licenseheader` tool is the model to copy.
- **[ADR 0057](adr/0057-audit-is-a-store-not-a-log-stream.md) — audit is a store. — not built.** `audit_log` on `Tx`, so the record commits if and only if the change did; append-only by database grant rather than by policy; read through the existing ABAC engine. The deliberate reversal: an audit write failure fails the command. There is no `audit_log` migration; audit retention is configured and floored in code with nothing writing the table it governs.
- **[ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md) — dual-sink storage and an in-product expert mode. — built for logs and spans.** File always (it is what records PostgreSQL failing); PostgreSQL additionally (migrations `0014`, `0015`), to make the UI queryable. An administrator sees logs and a **trace waterfall** inside Mosaic — no container, no external stack — reached only with `telemetry.read`, and the affordance is hidden from anyone without it. Retention is per-signal and read from the Active configuration, with audit floored in code. **Metrics and audit are absent from it**, because neither signal is produced yet; retention is declared but nothing deletes, since deletion needs the jobs runner below.
- **[ADR 0059](adr/0059-modules-observe-through-the-sdk.md) — the Platform owns the observability plane; modules hook into it through the SDK. — built (SDK `v0.13.0`, Stremio module `v0.17.0`).** A dependency-free `v1.Telemetry`, reached ambiently as `TelemetryFrom(ctx)` so one pattern covers `Import` and all seven provider roles. The Platform stamps attribution a module cannot forge and enforces per-module quota; a module configures no exporter, sink or retention. **The SDK does not take an OpenTelemetry dependency** — it has none today, that is load-bearing, and exporting OTel would publish an implementation choice as a contract.
- **[ADR 0060](adr/0060-the-supervisor-observes-independently.md) — the Supervisor observes itself, file-only.** Neither of ADR 0058's sinks covers a Platform that *never starts* — a failed migration, a bad DSN, a crash loop — because both live inside it. The Supervisor records the lifecycle a process cannot report about itself, keeps no database and no exporter, serves a minimal status page when the Platform is down, and merges into expert mode when it is up. **The dependency never inverts:** the Platform stays observable standalone.

**The shape of the answer, and what it rejects.** The instinctive stack — Collector, Prometheus, Loki, Tempo, Grafana — is right for an operations team and wrong for a self-hosted media server, where five more containers means nobody bothers. The split that resolves it: **instrumentation is architecture and is always on; export is configuration and defaults to nothing.** What that meant in the build is narrower than "instrument against the OTel API": the *formats* are the standard's — W3C `traceparent` on every edge, OTel's span and resource semantics — while the implementation is the Platform's own and takes no OpenTelemetry dependency. That is what keeps an OTLP exporter additive rather than a rewrite, and it is the same argument [ADR 0059](adr/0059-modules-observe-through-the-sdk.md) made for the SDK. The full stack is to ship as a `docker-compose.obs.yml` *developer* profile, with the exporter; neither exists yet.

**Order, and where it has reached.** Facade and process identity → correlation across processes → dual-sink storage → spans at the seams, including the SDK surface modules reach → the expert-mode UI → **metrics and the audit store → optional OTLP export**. Storage landed third on purpose: "queryable" cannot be retrofitted onto a file sink. The SDK surface rode with the seams because the module seam is where it is seeded, attributed and quota-bounded. **The first five are built, and they were what fixed the multi-repository debugging problem**; the two in bold remain, and they are the compliance and interop story rather than the diagnostic one. Nothing is instrumented against the OpenTelemetry SDK — the wire formats and semantics are OTel's so an exporter is additive, but the implementation is the Platform's own.

**What this thread waits on, and what waits on it.** Retention deletion and partition management are recurring no-user work, so both [ADR 0057](adr/0057-audit-is-a-store-not-a-log-stream.md) and [ADR 0058](adr/0058-telemetry-storage-retention-and-expert-mode.md) want the **jobs runner, a scheduler and the system principal** — the same trio [ADR 0049](adr/0049-resolution-cache-and-capability-classes.md)'s refresh cron and the torrent engine's eviction already want. Four callers now; it should be built once. Until it exists, partitions are created eagerly and retention is manual — stated, not silent. Nothing else in this thread is blocked on it.

**The Supervisor is decided but not buildable.** [ADR 0060](adr/0060-the-supervisor-observes-independently.md) says how it will observe; [ADR 0004](adr/0004-supervisor-as-host-manager.md)–[ADR 0007](adr/0007-static-go-module-composition.md) remain largely unbuilt and there is no Supervisor on disk, so nothing else in that record can be built yet. **The one piece that could land ahead of it did:** the Platform reads `MOSAIC_BOOT_ID` and adopts an inbound boot id rather than always minting one, so a Supervisor has something to hand over to when it arrives, and a boot has a name in the logs meanwhile. [ADR 0060](adr/0060-the-supervisor-observes-independently.md) also leaves one question open on purpose: where the shared record format lives, given two binaries must agree on it — a packaging decision that belongs with the Supervisor's own scaffold, and one the module-tier thread below now also touches.

**No longer undecided: the module process boundary.** This thread deliberately left the module seam a **wrapper rather than an inline edit**, so that moving modules out of process would replace it rather than duplicate beside it, and declined to write that record. It has since been written — [ADR 0064](adr/0064-extension-module-boundary.md), in the thread below — and it supersedes [ADR 0007](adr/0007-static-go-module-composition.md) for the extension tier exactly as anticipated. It also turns seam 9 from a courtesy into a control: in process the Platform sees a module's outbound calls only when the module uses the client it was handed, whereas through the boundary's forward proxy it sees every host, attributed per module, whether the module cooperates or not. [ADR 0059](adr/0059-modules-observe-through-the-sdk.md)'s surface is unaffected, which is the payoff for declaring an interface instead of exporting OTel — the module calls the same methods over a different transport. **SIEM export** remains a further audit exporter, left until a deployment needs one rather than guessed at now.

### The thread behind it: more than one user — **ADRs 0066–0069; the authorization model is built, the second account is not**

Mosaic has never had a second user. `bootstrap.EnsureAdmin` seeds the first and no transport has ever exposed `CreateLocalUser` — that is in the [unreachable capability](unreachable-capability.md) register, and it is the gap that makes everything below matter. A media server on a household VPS is a multi-user product or it is a personal toy, and every unbuilt piece here sits between those two.

**What forced the thread was a performance bug.** Phase 4 span data showed one search producing thirty-nine queries and ten full authenticate-plus-authorize cycles, because a per-result helper had no way to say "already inside the boundary" and so re-entered a public entry point once per result. [ADR 0066](adr/0066-authorization-is-carried-in-the-type.md) is the fix and is **built**: an `authorized` value only the boundary can construct, every handler entering through one function, plus the conformance suite described in the acceptance baseline above. Pulling that thread exposed the rest.

**Then the model itself turned out to have a hole. — [ADR 0069](adr/0069-privilege-cannot-escalate.md), built.** `CreateRole` accepted any permission set and `GrantRole` bound any role to any user, neither checking what the *caller* held — so `role.create` was really "hold every permission", two steps away: mint a role carrying everything, grant it to yourself. Every careful distinction elsewhere was decorative while that stood. Delegation now intersects with what the granter holds, and the grantable set is computed rather than assumed. It also corrected two earlier records: ADR 0057 and ADR 0058 had reasoned that `audit.read` and `telemetry.read` should be withheld from the bootstrapped account, which does not survive an account that can grant itself anything.

**Four accounts on one box is the whole specification**, and it is more demanding than it looks. An administrator; two adults differing only in whether they want anime; a child limited to PG12. [ADR 0067](adr/0067-three-authorization-mechanisms.md) finds that only the first is an action question, and that the other two are *different from each other* — one adult's exclusion is a preference, the child's limit is an access control, and building either as the other is a defect in opposite directions. Three mechanisms: roles for actions (**built**), a content scope for visibility (**not built**), preferences for taste (a general per-user preference store with its own policy actions is **built** — it went in for the expert-mode toggle — but nothing reads a taste preference, and it must stay out of the engine when something does).

**The credential side is decided and unbuilt.** [ADR 0068](adr/0068-one-principal-many-credentials.md): password, passkeys, per-key-attenuated API keys and legacy Basic-auth adapters all resolve to one Principal, stamped in one place. No JWT — not for fashion but because a claims-carrying token makes a tightened age limit take effect only when the token expires. Attenuation intersects live rather than at issue time, for the same reason.

**Two things block the child account, and neither is an authorization problem.** Ratings arrive from third-party addons in unvalidated JSONB whose correctness [ADR 0013](adr/0013-object-graph.md) assigns to the writing capability, so a child's safety would rest on data the Platform neither owns nor checks — that needs a Platform-owned field and an unrated-means-invisible default, which will hide a substantial part of a real library. Separately, the discovery plane returns *virtual* results with no stored attributes at all, so scoping the library does not scope search. Both are stated in [ADR 0067](adr/0067-three-authorization-mechanisms.md); neither is solved there.

**Order.** Content scope and the scoped `NodeQuery` first — it is what makes a second account mean anything, and it forces the request-scoped principal everything else needs. Then the admin surface, since users and roles have services, policy actions and passing tests but no way to reach them. Passkeys and API keys after, in either order. The legacy adapter last, and deliberately so: a Jellyfin client cannot ask to be scoped, so it is safe only once no code path can produce an unscoped read.

### The fourth thread: two module tiers, and how a module reaches a user — **decided in ADRs 0062–0065, nothing built**

Every module today is a Go library statically linked into one binary ([ADR 0007](adr/0007-static-go-module-composition.md)), and [ADR 0002](adr/0002-module-storage-and-delivery-model.md) went further: essential and community modules "differ only in delivery… Architecture: Identical." That symmetry was a deliberate anti-second-class-ecosystem stance and it has been proven twice — both modules are their own repositories importing only the SDK. **It does not survive contact with what the two kinds of module actually need**, and four records say what replaces it. None of it is built; the thread is written down so the Supervisor is not scaffolded blind.

- **[ADR 0062](adr/0062-two-module-tiers.md) — two tiers.** A **core module** for one of exactly two reasons — *coupling* (it must share the address space, as `postgres` must to join a `UnitOfWork`) or *guarantee* (Mosaic is not Mosaic without it, as playback and metadata are) — compiled into the binary, first-party, curated. Everything else is an **extension module** in its own process. The asymmetry is real and is stated rather than smoothed over: a third party can never be a core module, because both clauses exclude it. The tier is a delivery decision, not a contract decision — both tiers implement the same SDK interfaces and a module does not know which tier it is in.
- **[ADR 0063](adr/0063-platform-binary-built-by-ci.md) — Mosaic's CI builds and signs the Platform binary; the Supervisor selects rather than compiles.** This deletes [ADR 0006](adr/0006-supervisor-orchestrates-isolated-builds.md)'s per-install build pipeline, and the reason is what a self-hosted user would have experienced: a Go toolchain in the image, network to the module proxy before Mosaic serves anything, an arm64 NAS compiling the dependency closure behind a progress bar, and a third-party compile error as the first thing a new user sees. Every install would have produced a binary nobody had ever built. ADR 0062 is what makes the deletion possible — third-party modules leave the binary entirely, so the only thing being composed is a small curated set that is the same for everyone.
- **[ADR 0064](adr/0064-extension-module-boundary.md) — the boundary an extension module is crossed at, without changing what its author writes.** The SDK's Go interfaces stay the contract; the wire is an implementation of them on both sides, and nothing above the `CapabilityRegistry` knows it exists. The SDK splits rather than grows a dependency: `sdk` stays dependency-free, and a **nested module** at `sdk/host` carries the serving harness, so a module author's code is unchanged but for a `main.go` of roughly `host.Serve(stremio.New())`. All extension egress goes through a Platform-operated forward proxy: hosts attributed per module, content unseen, and — where the OS allows — no direct network at all, so the proxy is the only path rather than the easy one. **The process boundary is still not a general sandbox**, and the record says so.
- **[ADR 0065](adr/0065-module-distribution-and-trust.md) — signed artefacts and user-added repositories.** Trust moves from source to artefact: under ADR 0007 a module was source pinned by the Go checksum database, and it is now a binary someone downloads and executes. Mosaic's own repository is trusted by default, a user may add third-party repositories with explicit informed consent, and the provenance of what was installed stays visible afterwards — Jellyfin's plugin repositories are the worked example of the middle path between a curated closed set and an open one where the first malicious module is indistinguishable from the official ones.

**What it supersedes, and what that costs.** [ADR 0007](adr/0007-static-go-module-composition.md) narrows to the core tier, [ADR 0006](adr/0006-supervisor-orchestrates-isolated-builds.md)'s build sequence moves to CI, and [ADR 0002](adr/0002-module-storage-and-delivery-model.md) §1 and §3 are amended. The existing modules are unaffected as *source* — both already import only the SDK — but `module-stremio-addons` becomes an extension module and therefore stops being a Go dependency of the Platform, which is a composition-root change rather than a module change. Nothing here is buildable before the Supervisor exists, which is the thread's real dependency and is still [ADR 0004](adr/0004-supervisor-as-host-manager.md)-shaped and unwritten in code.

### Done: modules as typed providers, and two content planes — **done end to end (slices 1–4); the Shell renders live Platform screens**

Building the Stremio module against real addons showed the module contract is too narrow. A `Capability` today is a single write verb, `Import(query)` ([ADR 0019](adr/0019-module-capability-and-invocation.md)), which forces the caller to already hold a raw provider id (`movie/tt1254207`). But a Stremio addon is a *set of resources* — `meta`, `catalog`, `catalog/…/search`, `stream` — and the product needs all of them: a user should search Mosaic and have it use the module with **no raw ids**, an admin should browse the module's **collections** and pick which to publish as library, metadata should **enrich** existing nodes, and streams should attach to published items. And browsing a source must **not** flood the object graph — a source exposes far more than anyone curated.

Two decisions answer this, written ahead of the build (unusually) and now **Accepted** as slices 1–3 have landed:

- **[ADR 0027](adr/0027-modules-as-typed-capability-providers.md) — modules as typed capability providers.** The module contract becomes declared **provider roles**, one per resource kind: `MetadataProvider`, `SearchProvider`, `CatalogProvider`, `StreamProvider`. A module implements only the roles it fills and declares them in a new `Manifest.Provides []Role` — the manifest-shape growth [ADR 0019](adr/0019-module-capability-and-invocation.md) predicted, and what the `media_types` registry ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) has waited on. The Platform holds a **provider registry keyed by role**; that registry is the seam other modules will resolve through. Per the design call: the contract *supports* module-to-module use, but only the Platform resolves through it in this slice (build platform-first). `Import` survives, narrowed to "materialize *this* virtual result."
- **[ADR 0028](adr/0028-virtual-and-materialized-content.md) — virtual browse and materialized library.** Content exists in two planes. The **virtual plane** is what a provider returns on read — `SearchResult`/`CatalogItem` DTOs, never persisted; browsing is a read. The **materialized plane** is the object-graph library, written only by an explicit **curated act** (an admin publishing a collection, a user picking a search result), which reuses the existing import path. Streams **snapshot onto the curated set** (bounded by curation, so safe). Search is a **union** of providers + local library, deduped by source binding, each row marked *in library* vs *virtual*. This answers "don't overwhelm the DB" structurally: uncurated content never reaches a store.

**Build sequence — pick the lowest unbuilt slice; each passes `go build`/`go vet`/`go test -race` before the next:**

1. **SDK `v0.4.0` — done.** The four provider interfaces, the `SearchResult`/`CatalogItem` virtual DTOs, `Manifest.Provides`, and the import verb narrowed to a `ContentRef`. Published at `sdk` `v0.4.0`, no `replace`.
2. **Stremio providers — done.** All four roles implemented in `module-stremio-addons` (`v0.2.0`): `meta`/`stream` moved behind `MetadataProvider`/`StreamProvider`, and the `catalog` + `catalog/…/search` paths added. (Two blocking addon-reachability bugs were fixed first — manifest-URL normalisation and a User-Agent, without which Cloudflare-fronted addons 403'd.)
3. **Platform — done.** A `CapabilityRegistry` that resolves providers by role (and verifies declared roles at boot); `SearchAvailableContent` fans out to every `SearchProvider` and unions with the library, marking each result in-library by source binding; `ListModuleCatalogs`/`ListCatalogItems` browse a module's collections; `ImportContent` narrowed to materialise a `ContentRef`. Green against real PostgreSQL. (Each was first exposed as a GraphQL field; the screens above and the session transport's `Invoke` lane are what reach them now.)
4. **SDUI screens — done ([ADR 0029](adr/0029-sdui-emit-side.md)).** A Platform-owned screen registry builds `shell`, `home`, `search`, `collections`, `catalog`, `detail`, `settings` and the expert-mode diagnostics screens with the [`sdui`](https://github.com/mosaic-media/sdui) Go producer binding, and the session transport pushes them into the client's regions. Both a virtual and an in-library card open a **detail** screen: an in-library item shows its node's detail; a **virtual** item previews its metadata (through the module's `MetadataProvider`, via a `PreviewContent` query) and carries its sole library affordance — an **Add to library** button whose `Invoke: importContent` materialises it. Wiring the Shell surfaced the action ABI [ADR 0029](adr/0029-sdui-emit-side.md) predicted; it was first resolved over GraphQL and is now the session transport's own `Invoke` lane ([ADR 0061](adr/0061-one-client-transport.md)), which routes straight to the application services. Modules contribute data, not screens ([ADR 0008](adr/0008-sdk-as-public-contract-language.md)).

   **Verified end to end in a browser** against real PostgreSQL and the live Cinemeta + Torrentio addons: search returns virtual results with posters, a virtual result opens a metadata preview, *Add to library* materialises a Work + feature item + imdb source binding + a Torrentio magnet `RemoteLocation` part, and the item then reads *In library* on the next search. The `importContent`-returns-a-scalar half of the ABI was found this way — every Add-to-library click failed until it was fixed.

   **Artwork** is proxied through the Platform's own origin ([ADR 0030](adr/0030-artwork-proxy-and-cache.md)) so a CORS-less CDN can no longer break the image or the artlight wash, and the viewer's IP never reaches the CDN. Slice 1 (the proxy) is built and verified live; the durable filesystem cache of a materialised item's chosen art, and picking the best of several artwork candidates, are slice 2.

   **Known gaps, deferred:** the `SearchBar`'s `$value` client-side input binding and `Query`-into-region partial refresh (the search chrome drives full-screen re-fetch for now); the artwork durable cache + candidate selection ([ADR 0030](adr/0030-artwork-proxy-and-cache.md) slice 2); a Shell loading-state race if a search is issued *during* the boot sign-in (harmless on a settled page); a real detail screen's playback/parts; and the Mosaic Design Language skin over the Shell's neutral tokens.

**Two seams left open on purpose** (named, not answered — an agent hitting them should stop and decide, not invent):

- **Provider precedence** — which provider wins when several fill one role. Moot with one module; belongs with cross-module use.
- **Cross-module authority and the system principal** — whose authority a module-to-module call carries, and the no-user identity background enrichment needs ([ADR 0017](adr/0017-how-a-capability-acts.md)'s reserved gap, shared with module-declared jobs).

### Done: a live, fully server-driven client — **both sides on the ADR 0041 transport; verified live in a browser**

Wiring the Shell exposed two ways the client-server model still cheated on the SDUI thesis; both were decided in ADRs ahead of the build.

- **[ADR 0031](adr/0031-server-owned-app-shell.md) — the Shell is a pure renderer; the app shell is server-emitted. Accepted; built.** The Shell no longer hardcodes any layout. `mosaic-sdui-react` gained two structural primitives — `AppShell` (a named-slot frame: brand, a nav slot, a top-bar slot, a content region) and `NavItem` (a nav button that emits its action and highlights on the current screen) — reusing the existing chrome CSS (the `sdui-react` runtime package). The Platform emits a `shell` screen (nav rail + top-bar search); the Shell fetches it once and fills its content region with the current screen, so **navigation swaps the content while the shell persists**. The Shell now owns only the connection, a client-only `Standby` state, and the renderer. **Verified live**, including the payoff: stopping the Platform drops the client to the standby page — no chrome, never a fake app.
- **[ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) — the client session over a two-lane Connect/gRPC transport. Accepted; Platform side built. Supersedes [ADR 0032](adr/0032-live-session-websocket.md).** The transport is judged by a four-client bar (web, Flutter, Compose, native iOS), not "works in a browser": one `.proto` generates typed stubs per language, and binary framing is cheaper on mobile. `internal/transport/session` serves a `SessionService` — lane 1 unary intents (`Navigate`/`Invoke`/`SubmitInput`/`Attach`), lane 2 one server-streaming `Subscribe` per session for push — over h2c so both lanes multiplex on one HTTP/2 connection. A per-session **outbound mailbox** (a single sender goroutine drains to `Send`; handlers only enqueue) replaces ADR 0032's single ordered read loop; a monotonic per-session `seq` with a bounded replay buffer gives stream resume, folding in [ADR 0033](adr/0033-supervisor-driven-live-handover.md)'s handover; input coalescing moved from the read loop into session state. `Invoke` routes straight to the application services (`ImportContent`/`ConfigureModule`). The record's **GraphQL split** — GraphQL kept alongside as the external/tooling surface — is the one part of it that did not survive: [ADR 0061](adr/0061-one-client-transport.md) abandoned the split by deleting GraphQL outright. The bespoke WebSocket (`internal/transport/live`, `github.com/coder/websocket`) is removed. Race-tested (mailbox ordering, resume/replay, fresh-connect rebuild, supersede-on-reconnect, the reaper). **The client side is ported and the chain works end to end.** `sdui` publishes the generated TypeScript (`@mosaic-media/sdui/session` and `/sdui-pb`), and the Shell's `packages/shell/src/lib/live.ts` is a Connect client (`@connectrpc/connect` + `connect-web`) — unary intents plus the `Subscribe` stream, the full `RegionUpdate` op-set, and resume from a last-seq cursor. **Verified live in a browser** against real PostgreSQL and the live addons: the home screen renders over the session, and Settings round-trips a unary intent into the Stremio module's `SettingsUI` and back as a pushed `RegionUpdate`. **Since closed:** the HTTP round-trip integration test exists (`httptest` + the generated clients, against real PostgreSQL), the runtime's GraphQL invoke/query fallback is gone with the rest of GraphQL ([ADR 0061](adr/0061-one-client-transport.md)), and UINode-as-protobuf (encoding option (b)) landed under [ADR 0044](adr/0044-contracts-protobuf-workspace.md).

- **Slice 3 — resilience + history. Built; verified live.** Three of the four follow-ups below landed. **Socket reconnection + resume:** the Shell reconnects with exponential backoff + jitter (a rolling upgrade drops many clients at once — jitter spreads the retries), re-sends `hello` and **re-declares its current route** on reconnect so the Platform re-renders the exact screen that was showing; a drop reads *Reconnecting…* and self-heals. Resume works because the auth session is DB-backed (survives a Platform restart) while the live-session state is disposable — the client re-declares `{session, route}` and the Platform inherits nothing. On the Platform side, graceful shutdown closes each socket with a **"going away" (1001)** status so the client treats it as a reconnect, not an error (verified by a `-race` test). **Browser back/forward + deep links:** the Shell wires the History API transport-agnostically — `pushState` on a real navigate, `popstate` → the navigate intent for that entry, `replaceState` (no push) for search-as-you-type so keystrokes don't spam history; screens have URLs (`/detail?ref=…`, `/search?text=…`) that are shareable and re-declared on connect. **Server-side input debounce:** the Platform coalesces rapid `input` intents (a short window) before sourcing, the ADR 0032 backstop that protects the upstream addons from a request per keystroke — behind the client's own ~220 ms debounce. All verified live against real PostgreSQL + the Cinemeta/Torrentio addons: search-as-you-type over the socket, back/forward through detail↔search, and a full kill-and-restart of the Platform that dropped the client to *Reconnecting…* and then restored `/search?text=akira` with its results untouched. **Superseded by [ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md):** the Platform-side mechanics here — the going-away socket close, the server-side coalescing on the read loop — are replaced by the Connect transport's stream resume (a reconnecting `Subscribe` presents its `seq` cursor and the server replays or rebuilds) and session-state coalescing. The client-side pieces (backoff reconnect, browser back/forward, deep links) still stand as behaviours but move onto Connect-Web when the runtime is ported.

- **The rich detail page — built; verified live ([ADR 0034](adr/0034-rich-metadata-preview.md)).** The detail screen was thin (a poster, title, overview, one action) because the *contract* was thin, not the data: Cinemeta returns a clearlogo, cast, rating, runtime and — for a series — a per-episode synopsis/still, and the module decoded almost none of it into a seven-field `ContentMetadata`. `ContentMetadata` grew (SDK `v0.5.0`) a `Logo`, `Cast []Person`, `Rating`, `Runtime`, and an `Episodes []EpisodePreview` **read projection** (a series' episode list for the screen — deliberately *not* the materialised tree, which `Import` still builds); the Stremio module (`v0.3.0`) now decodes the fields it dropped (incl. cast from the `links` array and per-`Video` overview/thumbnail/released). The emit-side composes a backdrop+clearlogo **HeroBanner** with the poster docked in its aside, meta pills (year · type · runtime · ★rating), a genre row, a **cast rail** of `PersonChip`s, and — for a series — a **SeasonSelector** over the selected season's `EpisodeRow`s with per-episode synopses and stills. The SDUI vocabulary already had every piece but one; the single addition was a `logo` binding on `HeroBanner`. `PreviewContent` stopped short-circuiting for an in-library ref, so **one ref-based builder serves both planes** — a virtual item and a library item render identically, differing only in the primary action (*Add to library* vs an *In library* marker). A library detail is therefore re-derived live from the provider rather than read from stored fields (as current as the source; the trade is a dependence on a reachable metadata addon — a durable metadata cache is the follow-up). **Verified live** against real PostgreSQL + Cinemeta: *Avatar* (movie) shows the Pandora hero with the AVATAR clearlogo, cast and genres; adding it re-renders the same detail with the *In library* badge; *Avatar: The Last Airbender* (series) shows the season selector and each season's episodes with stills and titles, the season switch re-navigating by ref+season and keeping the selector in sync.

  **Not built — no data in the source, recorded not invented ([ADR 0034](adr/0034-rich-metadata-preview.md)):** clearart/banner artwork, **movie collections** ("the other Avatar films") and **similar/related shows** ("Legend of Korra", the live-action adaptation). Cinemeta provides none of these — they are TMDB/Fanart.tv concepts — and collections/similar for library items additionally need the still-open relation-read gap (`ListFrom`/`ListTo`). They wait on a TMDB/Fanart-class provider and that read surface.

  **Deferred, on purpose:** **finer region updates** — an `input` still re-renders the whole content region rather than just the results grid; the transport now carries the region op-set explicitly ([ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md)'s `RegionUpdate` — `REPLACE`/`APPEND`/`PREPEND`/`REMOVE`/`PATCH`), but the whole-content `REPLACE` is correct today (React reconciliation keeps the search field focused and its value), so narrowing it is an optimization needing runtime region-host plumbing in the `sdui-react` package, left for later. And the **full Supervisor-driven handover** — a seamless make-before-break upgrade between Generations — folds into ADR 0041's stream resume (a rolling upgrade drops the `Subscribe` streams and reconnecting clients replay from their cursor); the deeper make-before-break dance recorded in **[ADR 0033](adr/0033-supervisor-driven-live-handover.md)** stays **deferred** because it depends on the Supervisor, which is largely unbuilt ([ADR 0004](adr/0004-supervisor-as-host-manager.md)–[ADR 0007](adr/0007-static-go-module-composition.md)).

**Carried gaps / cleanup:** the npm gap is closed — the three web repositories were consolidated into the [`web`](https://github.com/mosaic-media/web) workspace ([ADR 0042](adr/0042-frontend-workspace.md)), `@mosaic-media/sdui-react` is published from it by CI, and CI checks the published version against the git tag so a release cannot silently be a local build. The Shell's now-unused `mock/` and `gallery/` are still dead code to remove.

### Done: one client transport — **[ADR 0061](adr/0061-one-client-transport.md); GraphQL deleted**

[ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) left GraphQL standing as the "external/tooling" surface and named *consolidate* as the option it was deferring. Six months of building against that split settled it: the split was not a boundary between two audiences, it was **one live operation and a large unused surface**. The Shell called exactly one GraphQL operation — `signIn`, because `SessionService` had no way to mint a session — and nothing at all called users, permissions, config versions, the content mutations, module catalogs or the jobs/health stubs. Meanwhile `importContent`, `configureModule` and `playPart` each existed twice, once as a resolver and once as a session `dispatch` case, which is the duplication the split was supposed to avoid.

What landed:

- **`mosaic.auth.v1.AuthService`** in the [`sdui`](https://github.com/mosaic-media/sdui) contracts workspace — `SignIn`/`SignOut`, a service of its own because it is the one call made *without* a session. `internal/transport/auth` implements it over `AuthenticateLocalUser`/`RevokeSession`.
- **`internal/transport/rpc`** — the Platform's seven error categories mapped onto Connect status codes, plus the telemetry interceptor moved out of `session` and parameterised by component. The category mapping is the thing GraphQL never actually did: its handler returned HTTP 200 for everything and never populated the `extensions.category` the Shell was reading, so a wrong password and an unreachable Platform were indistinguishable to a client for as long as that path existed.
- **`internal/transport/graphql` deleted**, `graphql-go` dropped, `/graphql` unmounted, and the "make it runnable" end-to-end test rewritten: it now signs in over `AuthService` and then subscribes and navigates over `SessionService` against real PostgreSQL, asserting the pushed content region — the same stack, proven through the only transport there now is.
- **The web workspace lost its second client**: the `gql` client and the runtime's invoke/query fallback are gone, `devSignIn` runs over `AuthService`, and both services share one Connect transport so the traceparent interceptor ([ADR 0054](adr/0054-the-correlation-id-is-the-trace-id.md)) cannot be missing from half the calls. The SDUI `query` action kind — a raw GraphQL string and a region to refresh — is removed from the contract; nothing ever emitted one.

**What this costs, stated plainly.** There is no ad-hoc query surface any more, and reacquiring one is a new transport rather than a flag. Creating roles, granting them, drafting and activating config versions and setting user status are now **unreachable from a client** — they have commands, policy actions and tests, but no surface. That was already true (a mutation with no UI is not a feature) and is now simply visible.

Every one of those is enumerated in **[Unreachable capability](unreachable-capability.md)**, written alongside this change so the debt cannot quietly become "complete". Auditing the deleted schema to build it turned up something the transport change did not cause: `app.CreateLocalUser` is a full, tested command — boundary order, policy denial, unauthenticated rejection, real-PostgreSQL integration — whose **only callers are its own tests**, and which no transport has ever exposed. Mosaic has no way to create a second user, and has never had one. That is the register's thesis in a single row: a green build and a passing suite are not evidence a capability is reachable.

### The first optional module — **done**

An **official optional module** is built exactly as a third-party module would be — its own Go module, importing only the SDK and nothing of the Platform's, compiled into the binary and invoked by the Platform. "Official" describes only its authorship (the Mosaic team), not its shape; the discipline is the point, because building it the third-party way is what proves the third-party way exists. It shipped as the **Stremio addon-source module** (movies and TV), not the anime module first sketched here — because the reference capability is *already* an anime importer, so a second anime consumer would only re-walk the same shape, whereas a different media type stresses the surface from a fresh angle and is the first thing to exercise [ADR 0014](adr/0014-storage-authority-and-transaction-scope.md)'s `RemoteLocation` Part path.

The reference capability already proved the *authoring* half — a package can be written against the SDK alone and drive `ContentService`. This slice built the *composition and invocation* half, which did not exist:

- **A capability/registration surface in the SDK** (a `v0.2.0` addition; the surface has grown additively through `v0.13.0` since, each bump driven by something a module needed). ADR 0008 always reserved "capability interfaces" and "module registration APIs" for the SDK, but ADR 0016 populated only the content services. The shape: a `Capability` interface a module implements — `Manifest()` plus `Import(ctx, ContentService, ImportRequest)` — and a minimal `Manifest`.
- **A capability registry and an `ImportContent` command in the Platform.** The composition root registers each module's capability; a generic `importContent(capabilityId, query)` command authenticates and authorises the caller, then invokes the named capability, forwarding that caller so the module acts as its invoking user ([ADR 0017](adr/0017-how-a-capability-acts.md)) and passing the Platform itself as the `ContentService`.
- **A served entry point over that command**, so a user triggers an import rather than a test doing it. It was a GraphQL mutation then and a session `Invoke` now.
- **Static composition** ([ADR 0007](adr/0007-static-go-module-composition.md)): the composition root imports the module and registers it, standing in for the Supervisor's eventual build-time module *selection*. **Not blocked on the Supervisor** — the mechanism of how a module plugs in is what this defines; the Supervisor automating *which* modules is later.

**Approach: a walking skeleton.** Build the thinnest real vertical end to end — SDK surface → a separate module → static composition → one invocation → real content in PostgreSQL — deciding shapes in code, then capturing what solidifies as ADRs (the module capability/invocation contract, and the composition model), written retrospectively once the code settles rather than as an RFC first.

**Invocation model:** the Platform invokes a *registered capability*. The module depends only on the SDK; the Platform owns the invocation surface and routes to it. The alternative — a module contributing its own GraphQL — was rejected because a third party can't import the Platform's transport without breaking the SDK-only boundary. Captured in [ADR 0019](adr/0019-module-capability-and-invocation.md) and [ADR 0020](adr/0020-optional-module-composition.md), written after the code settled.

**Resource-aware, so streams do not gate metadata.** The module uses whatever resources each configured addon declares (Stremio's `meta`, `stream`, …). Metadata creates the Work and its season/episode tree with an external-id binding; streams attach a `RemoteLocation` Part. The two are independent — a meta-only addon yields metadata with *no* Parts, so a user can enrich local media through Stremio addons without adopting remote streaming.

**Deliberately still deferred by this slice:** the full manifest shape (starts minimal, grows), the `media_types` registry (the Stremio module uses known media types), module-granular permissions (it acts with the invoking user's authority), and play-time stream *resolution*/transcoding (the future Remote Media module — the Stremio module only snapshots a stream location). Those remain future work below.

**Modules are the forcing function for the SDK.** Building the Stremio module is how the SDK's gaps get found. Two are identified so far:

- **User-managed module settings — done ([ADR 0021](adr/0021-module-settings.md)).** A user adds a Stremio addon by manifest URL at runtime rather than by an env var at composition time. A Platform-owned `ModuleSettingsStore` (one jsonb document per module id), generic `configureModule`/`moduleSettings` commands, and SDK `v0.3.0` handing a module its settings through `ImportRequest`. Retired the `MOSAIC_STREMIO_ADDONS` bridge. **A module now authors its own settings screen** through `SettingsUIProvider` ([ADR 0038](adr/0038-module-contributed-settings-ui.md)), which is the exception that proves ADR 0008's rule rather than a breach of it: the module declares a form, not a screen.
- **Module telemetry — done ([ADR 0059](adr/0059-modules-observe-through-the-sdk.md)).** A dependency-free `v1.Telemetry`, reached ambiently.
- **A module reports the version that was actually linked — done.** `v1.ModuleVersion` reads the build graph rather than a constant someone remembered to bump.
- **Module-declared cron/jobs — identified, not built.** A module needs to register recurring work the Platform runs (cleanup, periodic refresh). This converges three deferred pieces at once: the jobs runner (tables exist, no service), a scheduler/recurrence layer (none), and the **system principal** ([ADR 0017](adr/0017-how-a-capability-acts.md)'s named gap — a no-user job has no session to forward). **Four separate threads now want that trio** — this one, the resolution cache's URL refresh, the torrent engine's eviction, and telemetry retention deletion — and it should be built once.

**Harden the SDK toward stable (it is pre-1.0 on purpose, now `v0.13.0`), in parallel or alongside:**

- **The relation-read gap.** `ContentService` can write edges (`RelateContent`) but has no `ListFrom`/`ListTo` to read them, so a capability can't query the graph it builds. Small and additive — a `v0.x` bump. The most self-contained next thing.
- **A second capability** — a different media type (music, comics), built against the SDK, to stress the surface from a fresh angle and surface what a real second consumer needs before the surface stabilises. Likely forces the relation-read gap and, if it introduces a novel media type, the `media_types` registry.

**The rest of the module system** (the slice above starts it; these follow):

- **The full capability manifest shape.** The module slice starts it minimal (id, version, name) and grows it as needs appear — permissions a module declares, media types it sources, whatever the Supervisor's onboarding surfaces.
- **The `media_types` registry** ([ADR 0015](adr/0015-open-and-closed-vocabularies.md)) — catches a media type that was never real, which normalisation cannot. Waits on the manifest shape and on a module that introduces a new type (the Anime module does not).
- **Module-granular permissions and a system principal** ([ADR 0017](adr/0017-how-a-capability-acts.md)) — authority a module holds distinct from its invoking user, and a no-user identity for background work. (User permissions management is built, and the module slice acts with the invoking user's authority.)
- **Module distribution and the process boundary** — now decided rather than open: ADRs 0062–0065, the fourth thread above.

**The rest, unplanned in detail, each scoped when it starts:**

- **Export formats** — NFO for other systems, `.mos` for Mosaic-to-Mosaic portability, generated on demand from authoritative state ([ADR 0014](adr/0014-storage-authority-and-transaction-scope.md)).
- **Job queue** — the `jobs` tables exist with no service; `SELECT ... FOR UPDATE SKIP LOCKED` is the intended pattern, for import, provider sync and enrichment.
- **`LISTEN`/`NOTIFY`** — an accelerator over the outbox worker's poll, not a replacement; notifications drop when no listener is connected, so the poll stays the floor.
- **A second client** — the four-client bar [ADR 0041](adr/0041-cross-client-transport-two-lane-rpc.md) chose the transport against (web, Flutter, Compose, native iOS) has only ever had one client behind it. Everything that would be proved by a second one is currently assumed, and the capability declaration on `Attach` is the first thing that would stop being optional.
- **Mosaic Design Language** — the values that replace the Shell's neutral tokens: acrylic with weight, artwork as the light source. The material work has started ahead of the record (the acrylic "Optical Parallax" sheet and the cinematic home carousel are built); what does not exist is the stated language behind them.

## Working rules

- **One slice at a time**, in order, each passing its gate before the next begins.
- **Report blockers, do not force past them.** The reference capability slice was stopped twice and reported instead of bodged. That was correct both times, and it is why the fix was a design decision rather than a workaround buried in code.
- **Code is authoritative where code exists.** Documentation describes what is built; it does not specify what is not.
- **When implementation contradicts a specification, the specification is wrong.** Fix it there, in the same session, rather than carrying a correction in a repository-local note.
