# Glossary

---

# Purpose

This glossary defines the terminology introduced by **MDS-002 — Colour System**.

Unlike previous glossaries, these definitions describe architectural colour concepts rather than implementation values.

Every future MDS specification should use these definitions consistently.

Changing colour terminology changes how contributors reason about the Design System.

For that reason, these definitions should remain stable.

---

# A

## Accessibility Colour

A resolved colour that satisfies accessibility requirements while preserving semantic meaning.

Accessibility Colours are generated through Theme Resolution rather than manually authored.

---

## Atmosphere

The adaptive environmental lighting generated from the user's current entertainment.

Atmosphere is derived from artwork.

It influences Materials.

It never replaces Brand or Semantic Colours.

---

## Atmosphere Token

A Runtime Token representing environmental colour characteristics.

Examples include:

- Atmosphere.Primary
- Atmosphere.Reflection
- Atmosphere.Highlight

Atmosphere Tokens are generated.

They are never authored manually.

---

# B

## Brand Colour

A stable colour representing the Mosaic identity.

Brand Colours communicate:

- recognition
- consistency
- product identity

They intentionally do not communicate runtime emotion.

---

# C

## Colour Resolution

The deterministic process that transforms Semantic Colour intent into resolved implementation values.

Colour Resolution considers:

- Theme
- Accessibility
- Runtime Atmosphere
- Device

before producing final colours.

---

## Colour Synthesis

The runtime process responsible for generating environmental atmosphere from entertainment artwork.

Colour Synthesis influences Materials.

It never changes semantic meaning.

---

# L

## Light Source

The conceptual origin of Runtime Atmosphere.

Within Mosaic, artwork is treated as a light source rather than a colour palette.

This distinction underpins the Refraction System.

---

# M

## Material Colour

The colour produced after Runtime Atmosphere has interacted with a Material.

Material Colours are generated.

They are not authored.

---

# P

## Primitive Colour

A measurable physical colour value.

Examples include:

- Cyan 500
- Slate 950
- White
- Black

Primitive Colours intentionally contain no semantic meaning.

---

# R

## Reflected Colour

A colour generated by the interaction between Runtime Atmosphere and Materials.

Reflected Colours should appear subtle and environmental.

They should never dominate semantic hierarchy.

---

## Runtime Atmosphere

The adaptive environmental colour model generated from the user's current World.

Runtime Atmosphere represents emotional lighting rather than interface colouring.

---

# S

## Semantic Colour

A colour role representing meaning rather than implementation.

Examples include:

- Surface.Primary
- Text.Secondary
- Action.Primary
- Status.Success

Applications should consume Semantic Colours rather than Primitive Colours.

---

## Surface Colour

A Semantic Colour describing structural hierarchy.

Surface Colours communicate:

- depth
- grouping
- organisation

They should not communicate interaction or status.

---

# T

## Theme

A deterministic interpretation of Semantic Colours.

Themes implement:

- Light
- Dark
- OLED
- Accessibility

without changing semantic meaning.

---

## Theme Resolution

The process by which Semantic Colours become theme-specific implementation values.

Theme Resolution occurs before Runtime Atmosphere.

# Terminology Rules

Future contributors should:

- describe meaning before colour values
- describe atmosphere before implementation
- distinguish Brand from Atmosphere
- distinguish Semantic Colour from Primitive Colour
- avoid implementation-specific terminology inside architectural documentation

Colour terminology should remain independent from rendering technology.
