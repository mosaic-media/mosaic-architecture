## How status is decided

A specification counts as rewritten when fewer than 85% of its prose paragraphs are a single sentence. Untouched specifications sit at or near 100%; MEG-004 landed at 64% after its rewrite, so the threshold separates the two populations without needing a manual flag.

The other columns size the remaining job: `RFC` counts uppercase RFC 2119 keywords that MDG-001 chapter 10 says should be sentence case, and `Bare` counts code fences with no language, which markdownlint reports as MD040.

A small non-zero `RFC` count can be legitimate: a Normative Language table that *defines* the RFC 2119 keywords keeps them uppercase, because lowercasing there would destroy its meaning. MEG-004 retains five for that reason.

## For agents

1. Take the top unfinished specification unless told otherwise. The table is ordered by size of remaining work.
2. Rewrite it, then run `python3 scripts/prose_audit.py <folder>` and confirm it passes.
3. Record anything you could not resolve without inventing facts in [open-questions.md](open-questions.md).
4. Regenerate this file with `python3 scripts/rewrite_status.py --write` and commit it alongside the rewrite.
5. Never use `git stash` — other agents may hold uncommitted work. Use `git show HEAD:<path>` to read the committed version of a file.
